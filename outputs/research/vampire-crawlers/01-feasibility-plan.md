# 可行性方案：用竞品证据洗出原创游戏策划

## 目标重定义

本项目不是测试 Web 性能，也不是做 Vampire Crawlers 的 1:1 克隆。真正目标是验证一套流程：

1. 输入：竞品体验、Steam 全量用户评测、媒体评测、YouTube 动态表现、评论区反馈、新故事背景。
2. 处理：识别核心体验、差评暴露的突破口、可迁移的机制骨架、不可复制的 IP/资产部分。
3. 输出：一套原创世界观下的玩法策划、系统设计、数据结构和可运行 Web 原型。
4. 后续：如果 Web 原型验证通过，再迁移到其他引擎。

## 可行性判断

可行。原因是 Vampire Crawlers 的核心体验不是依赖高保真 3D 或复杂物理，而是依赖以下结构：

- 地牢探索给玩家空间推进感。
- 回合制卡牌让“自动战斗爽感”变成可操作的顺序连击。
- 升级、宝箱、宝石、武器进化和永久解锁组成长线滚雪球。
- 音效、粒子、数字、经验宝石、UI 抖动共同制造“爽感”。

这些都可以在 Web 原型中表达，并且可以用数据驱动方式保持迁移性。

## 伪 3D 表现方案

### 方案 A：Three.js 轻 3D 原型（推荐）

用 Three.js 做第一人称格子地牢，角色、敌人、特效可以先用 billboard sprite 或低模 GLB。

优点：
- 最接近 Vampire Crawlers 的空间体验。
- 能验证“伪 3D + 卡牌 UI + 战斗反馈”的组合。
- 后续迁移 Unity/UE 时，世界坐标、摄像机、格子、房间数据都容易映射。

缺点：
- 比纯 2D 多一层资产和镜头约束。
- 需要早期定义单位、格子尺寸、朝向、碰撞代理和镜头规则。

### 方案 B：Phaser 伪 3D / raycast 风格

用 Phaser 或 Canvas 做 Wolfenstein 式伪 3D，场景是 2D 网格投影。

优点：
- 实现快，适合机制验证。
- 资产成本低，可以快速替换世界观。

缺点：
- 与未来 Unity/UE 的迁移关系较弱。
- 很难验证 3D 场景中的 UI 读性、镜头、空间压迫感。

### 方案 C：纯 2D 策划沙盘

完全不做 3D，只做卡牌、奖励、构筑和回合循环。

优点：
- 最快验证卡牌和成长曲线。

缺点：
- 无法回答“第一人称地牢 + 爽感反馈”是否成立。
- 对 Vampire Crawlers 的核心流程复刻不完整。

## 推荐路线

选择 **方案 A：Three.js 轻 3D 原型**。

但要把架构写成可迁移：

- Simulation 层：实体、房间、战斗、卡牌、奖励、解锁、存档，全是纯 TypeScript/JSON。
- Renderer 层：Three.js 只负责镜头、网格地牢、sprite/mesh、粒子、动画。
- UI 层：DOM/React 或轻 DOM 负责卡牌、HUD、奖励选择、商店。
- Data 层：所有卡牌、敌人、掉落、地图、世界观文本用 JSON/CSV，便于以后迁移到 Unity/UE DataTable。

## 关键可行性约束

1. 原型必须先证明“动态爽感”，不是只证明 UI 能显示。
2. 视频证据必须包含短片段、评论、字幕和帧序列，不能只截图。
3. 策划输出必须是原创映射，不允许保留 Vampire Survivors 的角色、武器名、怪物、地图、美术和音乐。
4. Web 原型不是最终引擎承诺，只是深度能力验证工具。
5. 如果最终迁移其他引擎，必须保证数据结构和核心规则先被独立出来。
