## 7. 游戏 UX 架构师

### 当前判断

H2 HUD 已经把首屏主语从 realtime / burst 拉回到 Hyper-Turn 的关键决策：`CHAIN`、下一张期望 MP、敌人意图、Payoff 预览、结束回合后果、卡牌 MP 成本。这个方向符合当前唯一有效合同：0-30 秒内必须读到 `0 -> 1 -> 2` 或 `0 -> 1 -> payoff`，并在结束回合前读到敌人意图。

但首屏仍像“多个正确面板同时展开”：顶部状态条、四格 combat director、deal panel、target panel、enemy peek、手牌都在争夺下一步决策的解释权。玩家能找到信息，但不一定会立刻知道“我现在应该先点哪一张牌，点完后是否结束回合”。

### 10 个镜头观察

1. 首屏最强信息是四格 director，其中“下一张期望 MP 0”和手牌里的绿色 MP0 卡形成了有效引导，这是当前 HUD 最有价值的决策链路。
2. `CHAIN MP0?` 同时出现在顶部 chip 和 director；`MP0?` 对开发者表示链尚未开始，但对玩家像未知状态或报错，第一眼不如“从 MP0 开始”明确。
3. 敌人意图重复出现为顶部 `-17 HP` 和 director 的“将受 17 伤害”，强化了风险，但“将受”主语不明确，玩家可能需要一秒判断是敌人还是自己受伤。
4. End Turn 已在 deal panel 中可见，且 aria/title 带有未解决伤害；但可见文本层面的后果在移动截图中不够突出，按钮本身没有直接写出“结束会吃 17”。
5. Payoff 预览展示 `Row Cleave x1` 和“前排群攻 5”，能告诉玩家有终结牌，但它没有明确说明“先接到 x3 后再打会更好”。
6. 桌面首屏中央战场基本保留，HUD 占左上与底部，符合“保住 playfield”的方向；但左上信息块在 1440 宽下仍偏多，像战斗仪表盘而不是单一作战指令。
7. 移动首屏上半区堆叠了状态、director、deal、target、enemy peek；战场可见，但玩家决策路径被切成多段，需要从上到下再跳到底部手牌。
8. `target-panel` 写明“未选择，单体牌随机前排”，这个说明降低了首次出牌疑虑；但目标选择入口藏在“前排显影”，对于想精准击杀的玩家不够直接。
9. Debug Trace 在桌面收起、移动隐藏，说明 H2 已经把调试内容从首屏降权，这一点符合合同里的体验导向。
10. 当前 QA 记录桌面跑过 2 回合，移动只做视觉 smoke；因此“首屏能读懂”有证据，“移动玩家能按预期完成一轮决策”证据不足。

### 最大风险

最大风险不是信息缺失，而是“下一步行动没有成为唯一主语”。玩家看到链路、意图、payoff、目标、结束回合，但系统没有把它们压缩成一个明确命令：先打哪张、为什么、打完下一张是什么、什么时候结束回合。Hyper-Turn 的爽感依赖高速决策，如果首屏需要玩家在多个面板之间拼逻辑，0-30 秒验收会从“可操作”退化为“可读但慢”。

### 下一轮最小改动

最小改动应优先做一条全局“推荐下一步”决策线，而不是继续增加面板。它可以复用现有派生状态，在 deal panel 或 director 顶部显示一条可见文本：

- 未起链：`先打 MP0：Debt Hook`
- 已起链：`接 MP1：Redline Cut，可到 x2`
- payoff 成熟：`现在打 Row Cleave：前排群攻 15`
- 无可打牌：`无可打牌，结束将承受 17`

同时把 `MP0?` 改成更像玩家语言的“未起链”或“从 MP0 开始”，减少问号带来的状态不确定感。

### 验收方式

- 390x844 与 1440x1000 首屏截图中，不看手牌细节也能在 3 秒内指出“推荐下一步”。
- DOM 文本中能找到当前推荐牌名、推荐 MP、当前结束回合后果。
- 桌面手动跑 `Debt Hook -> Redline Cut -> Row Cleave -> End Turn` 后，每一步推荐文本都随状态推进。
- 移动端截图必须复核 H2 声称的顶部降权是否真的生效；当前 mobile 截图仍显示 LV/FSM/牌堆，与 CSS/批次说明不一致，需要下一轮重新确认。
- 验收记录不能只写“pass”，必须保留至少一张起手、一张接链后一张 payoff 成熟后的截图或 DOM 摘要。

## 8. 移动端手牌操作专家

### 当前判断

390x844 的移动布局已经具备可操作基础：底部两列手牌、MP 成本大徽章、绿色起链高亮、红色非起手/断链风险、结束回合按钮可见。`button { touch-action: manipulation; }` 和 HUD 的 pointerdown/click 抑制也说明当前实现考虑了移动点击重复触发。

主要短板是“高速连续点牌”的稳定性还没有被证据覆盖。QA 明确写了移动 smoke 是视觉-only，未跑单次 tap 是否只触发一次、End Turn 结果是否匹配预览、chain/break/repair/payoff 是否可在移动端连续完成。对于 Hyper-Turn 来说，这些不是边缘项，而是移动体验的核心验收。

### 10 个镜头观察

1. 390x844 截图中四张手牌按 2x2 排布，每张约半屏宽，点牌面积明显大于桌面版，首轮点牌不吃力。
2. MP 徽章在移动端仍是最醒目的读牌元素，`MP 0 / 1 / 2` 的辨识度高于卡名，这符合按费用升序连打的目标。
3. `Debt Hook` 作为绿色起链牌非常突出，玩家第一 tap 的候选基本明确。
4. 其他三张牌显示“非起手 x1”并使用红色卡底；这能表达不是最优顺序，但也可能被误读成不能打或危险牌。
5. 移动端 `.card-effect` 被隐藏，导致“MP1 单体 6伤害 / MP1 单体 9伤害”等细节不可见；玩家只能靠卡名、verb、目标和 payoff 行判断。
6. 卡牌高度在 82px 左右，足够点，但文字密度很高；更长中文卡名或更长目标文案会比当前示例更容易省略。
7. End Turn 在上方 deal panel 内，按钮可见，但高度约 32px，低于常见移动触控舒适区；玩家从底部手牌连续操作后需要跨很远去点它。
8. “前排显影 5”按钮和展开后的 enemy-slot 高度偏小；如果下一阶段要求移动端精准选目标，当前目标按钮比手牌按钮难点得多。
9. 手牌每出一张后很可能重新排布，因为按钮来自当前 hand map，未看到固定槽位策略；快速连打时位置跳动可能让第二 tap 点错。
10. QA 记录中移动 `Single tap maps to one intended card action`、End Turn resolution、break/repair 路径都未运行；现有证据只能证明“看起来能点”，不能证明“真的适合连续打牌”。

### 最大风险

最大风险是移动端把 Hyper-Turn 的“高速出牌爽感”变成“看清了但不敢快点”。如果出牌后卡位重排、End Turn 离手牌太远、目标选择按钮太小，玩家会被迫慢下来确认每一步。这样即使桌面验收通过，移动端仍无法证明 3-5 个短回合里的成功链、断链、修补、payoff 可以被稳定执行。

### 下一轮最小改动

最小改动优先不改玩法，只改移动操作可靠性：

- 移动端把 End Turn 做成靠近手牌的固定次级按钮，或在无可打牌时临时贴近底部。
- 保留本回合手牌槽位，出牌后用空槽/淡出占位，避免剩余牌瞬间重排。
- 把移动端目标选择按钮和 enemy-slot 的触控高度拉到至少 40-44px。
- 对“非起手 x1”补一个更明确的短文案，例如“可打但断链 x1”，避免把可打牌误解成禁用。

### 验收方式

- 在 390x844 上实际 tap：`Debt Hook -> Redline Cut -> Row Cleave -> End Turn`，确认每次 tap 只产生一个 intent。
- 移动端至少跑一次故意乱序，确认“可打但断链”能触发成功出牌而不是被玩家误认为 disabled。
- 移动端至少跑一次目标选择：展开前排、点一个 enemy-slot、再打一张单体牌，确认目标锁定可见且不误点。
- 截图记录出牌前、第一张牌后、第二张牌后、End Turn 前四个状态，检查手牌是否因重排造成操作风险。
- QA 表里 `M-06 / M-08 / M-09` 不能继续是 `not run` 或 `partial`；否则移动端仍不能算通过。

## 9. 可访问性 / 文本可读性专家

### 当前判断

当前 HUD 的高对比色、粗字重、MP 数字徽章和 focus-visible 都有正向基础。中文核心文案比上一轮更贴近玩法，例如“敌人意图”“结束回合”“目标”“未选择，单体牌随机前排”。但仍存在三类可读性风险：混合英文/调试词进入玩家层、关键后果被 title/ellipsis/小字号承载、红绿状态虽然有文字辅助但文案仍可能误解。

### 10 个镜头观察

1. `HP / MP / CHAIN / PAYOFF / FSM / Restart / Debug Trace` 与中文混排；其中 `FSM` 和 `Debug Trace` 属于开发语言，不应在玩家首屏承担信息。
2. `MP0?` 的问号对可访问性不友好；它不是玩家问题，而是系统状态，建议改成“未起链”或“从 MP0 开始”。
3. “将受 17 伤害”缺少主语；对中文读者更清楚的表达是“结束会受 17”或“敌方将造成 17”。
4. `-17 HP` 很清楚，但单独放在“意图”下时还需要玩家理解它是结束回合后果；可以配合可见文案而不是只靠 title。
5. “非起手 x1”比“断链 x1”温和，但它没有说明仍可打；合同要求断链不是禁牌，文案应避免让玩家以为红牌不可用。
6. 多处使用 `white-space: nowrap` 与 `text-overflow: ellipsis`，当前示例能容纳，但长中文卡名、长敌人名、复合效果会被截断。
7. 移动截图里敌人详情 `Debt Wisp 2 / Redline Brute 5 ...` 被省略，可能隐藏部分伤害来源；总伤害仍可见，所以不是阻断，但会影响理解来源。
8. 移动端卡牌隐藏 `.card-effect` 后，数字伤害主要消失，只剩 payoff 或卡牌 verb；对比较两张 MP1 单体牌的伤害差异不够友好。
9. 红色和青色承担了大量状态：危险、断链、起链、目标、敌人类型。当前文本有一定补充，但选中目标、危险等级、断链风险仍偏依赖颜色强弱。
10. End Turn 的 `aria-label` 和 `title` 有后果说明，但移动端用户不会看到 tooltip；可见文本必须独立承担理解，不应把关键信息只放在 title。

### 最大风险

最大风险是“文本看得到，但关键语义不够直给”。当前 HUD 对熟悉卡牌术语和调试术语的人是可读的，对第一次进入的中文玩家仍有门槛：`CHAIN / PAYOFF / FSM / MP0? / 非起手` 需要翻译成行动后果。Redline 当前验收不是看 UI 是否信息完整，而是看玩家是否能快速做对决策。

### 下一轮最小改动

最小改动是做一次文案和可见后果清理，不需要大改布局：

- 隐藏或改名玩家不需要的 `FSM`、`Restart`、`Debug Trace` 首屏文本，特别是移动端。
- 把 `MP0?`、`Start MP0` 合并成“从 MP0 开始”。
- 把“将受 17 伤害”改成“结束会受 17”或“敌方将造成 17”。
- 把“非起手 x1”改成“可打，断链 x1”或“非推荐 x1”。
- 让 End Turn 按钮附近可见地显示后果，不依赖 `title`。

### 验收方式

- 让中文读者只看 390x844 首屏 3 秒，说出：当前 HP/MP、第一张推荐牌、结束回合会损失多少 HP。
- 使用浏览器文本提取检查首屏是否仍出现 `FSM` 作为玩家层可见文本；如果出现，需要确认它只在 debug 或桌面开发层。
- 以 320px、390px、430px 三档截图检查卡名、关键数字、End Turn 后果没有被省略到不可理解。
- 检查色盲/低饱和模拟下，起链、断链、危险仍可通过文字判断。
- 检查移动端没有把关键说明只放在 `title`、hover 或 aria 中；aria 可以补充，但不能替代可见文案。

## 优先级建议

1. 先补“推荐下一步”单一决策线，把 chain、intent、payoff 压缩成玩家下一 tap。
2. 移动端必须从视觉 smoke 升级到真实 tap 验收，覆盖单 tap、连续出牌、End Turn、目标选择。
3. 修正文案歧义：`MP0?`、`将受`、`非起手`、`FSM` 是当前最小成本的可读性问题。
4. 处理移动端手牌重排和 End Turn 距离问题，避免 Hyper-Turn 连打变慢。
5. 复核 390x844 顶部降权证据；H2 说明/CSS 与现有 mobile 截图对 LV/FSM/牌堆的呈现不一致。
