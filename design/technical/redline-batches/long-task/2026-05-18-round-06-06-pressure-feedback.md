角色：第 6 轮专家 06，压迫体验与读牌反馈设计师  
工作目录：`/Users/roc/Game-001`  
任务边界：本任务只新增本文档；不改源码、不提交 git。  
本轮主题：修补牌 / 抽牌 / 临时资源 / 重排的 1:1 卡牌机制复刻，从体验角度判断修补牌应如何让玩家读到“救了一手但仍承压”。

## 0. 体验裁决

修补牌不应该被表达成“免费解困”。它应该让玩家感觉：

```text
我争取到了一次继续找答案的机会。
但敌人意图没有因为这张牌本身下降。
如果后续没有打出击杀、打断、终结或真实降意图动作，结束回合仍然要吃伤害。
```

因此第 6 轮的读牌目标不是把修补做得更爽，而是把修补做成高压下的临时续命动作。玩家打出 `blood_tithe`、`pulse_draw`、`wild_mana_stitch`、`paper_shatter` 时，HUD 必须同时告诉他两件事：

```text
这张牌帮我找下一步。
这张牌本身不降低当前意图。
```

一句话：**修补牌救的是手牌路线，不是当前敌意图。**

## 1. 当前玩家可见反馈事实

### 1.1 卡牌和奖励事实

当前 `prototype-web/src/data/cards.ts` 已经把抽牌修补推进到奖励层：

- `blood_tithe`：`0 MP`、自身、`drawCards: 1`、`availability: reward`、`rewardBranches: ['repair-resource']`、`mobileEffect: 抽1`。
- `pulse_draw`：`1 MP`、自身、`drawCards: 1`、`availability: reward`、`rewardBranches: ['repair-resource', 'route-bridge']`、`mobileEffect: 抽1/2`。
- `paper_shatter` / `lantern_captain`：2 费自身抽牌牌，文案是接链后 `抽3` 找终结或路线，但当前 `reorder` 仍主要是“整备/找牌”语义，不是玩家可操作的重排界面。
- `wild_mana_stitch`：0 费修补，自身，抽 1，并当前 MP +1；这张牌最容易被读成“免费补链 + 免费补手 + 免费补资源”。
- `rewardCardPool` 当前顺序让 `blood_tithe` 进入默认早期奖励，`pulse_draw` 在后续作为修补 / 承接选择出现。

### 1.2 HUD 已经给出的压力锚点

当前 HUD 已经有多个地方显示敌意图：

- 状态条 `intent-chip` 显示 `意图 -N HP`。
- `combat-director` 的敌人意图格显示 `结束回合 -N HP`。
- 出牌阶段小面板会显示未解决意图与可出牌数量。
- End Turn 按钮会把结束回合后果写进可见按钮 / title / aria。
- 敌人槽位显示 `本回合 -N` 或后排 / 下轮信息。

这些都是正确的压迫锚点，但它们和卡牌按钮的修补反馈是分离的。玩家看手牌时，最容易被卡牌按钮里的短句影响。

### 1.3 当前卡牌按钮和日志事实

当前手牌按钮会显示：

- `card-meta`：例如 `开链/抽牌 · 自身`、`承接/抽牌 · 自身`、`修补/抽牌 · 自身`。
- `chain-preview`：例如 `起链 x1`、`接链 x2`、`断链 x1`。
- `card-intent-preview`：攻击牌显示 `意图 N->M`；自身无伤害抽牌牌显示 `抽N找解`；无伤害无抽牌牌可能显示 `返MP找解` 或 `不降意图`。
- `card-effect`：桌面显示 `MP X · 抽N`；移动端 `<=640px` 会隐藏 `.card-effect` 和 `.card-payoff`。

当前战斗日志对出牌事件显示：

```text
出牌 Pulse Draw · 倍率 x2 · 抽2
```

这已经解决“倍率后到底抽几张”的问题，但还没有解决“抽牌后意图是否仍在”的问题。

### 1.4 运行时事实

当前自身抽牌 / 修补牌不会直接造成伤害，也不会直接降低 `enemyIntentSummary.totalDamage`。

`card.self.resource` 对自身牌只会根据牌面和倍率生成：

```text
DrawCards.count = card.drawCards * effectMultiplier
GainEnergy.amount = card.energyGain
```

敌人意图仍然在 End Turn 后结算。也就是说，系统真相已经是“修补不降意图”，但玩家可见文本还没有把这条真相贴到修补牌按钮和日志上。

## 2. 修补牌造成的误解点

### 2.1 “抽N找解”会被读成“已经解压”

`抽2找解` 是正确的找牌反馈，但它缺少后果句。玩家在高压时看到“找解”，会自然推断这张牌是系统给的救场按钮。如果按钮同一行没有显示 `仍-17`，他不一定会把视线拉回状态条和 End Turn。

风险文案：

```text
抽2找解
返MP找解
修补 抽1
```

这些都只讲“我获得了资源”，没有讲“敌意图还没动”。

### 2.2 “修补”天然带有治愈和修复语义

中文“修补”容易让玩家以为已经把局面修好了。实际机制更像“补费用链缺口”或“让路线不断”，不是修复血量、取消攻击、降低敌人意图。

如果 `wild_mana_stitch` 同时显示 `修补 抽1`、`当前MP+1`，但没有显示 `不降意图`，它会在心理上变成免费按钮。

### 2.3 HUD 压力锚点和卡牌按钮不在同一个阅读焦点

状态条、Director、End Turn 都能显示压力，但玩家出牌时主要看手牌 rail。移动端又隐藏了 `.card-effect`，所以移动端最关键的是：

```text
chain-preview
card-intent-preview
missing/authorization 状态
```

如果 `card-intent-preview` 只写 `抽2找解`，移动端玩家很可能看不到“仍会吃 -N HP”的事实。

### 2.4 日志记录了抽牌收益，没有记录压力保留

日志当前能显示 `倍率 x2 · 抽2`。这对倍率读数是好事，但从压迫体验看，它会强化“我得到了好处”。如果下一条没有同步显示 `意图仍-17`，玩家会把修补理解为成功结算，而不是风险仍未解除。

### 2.5 重排 / 整备文案容易被读成控制牌库

`paper_shatter`、`lantern_captain` 带有 `reorder` / `整备` 语义，但当前玩家没有真正的重排操作界面。若文案写得像“重排牌库”或“整理下一抽”，玩家会误以为自己获得了确定控制权。当前应继续把它压成“找牌 / 找终结”，不要把它包装成免费控顶。

## 3. P0 文案 / 反馈修改建议

### 3.1 核心文案合同

所有无伤害自身修补 / 抽牌 / 返 MP 牌，卡牌按钮必须同时表达：

```text
获得什么：抽N / 修补链 / 返MP
仍然怎样：仍-N / 不降意图
```

推荐短句：

| 场景 | 桌面卡牌按钮 | 移动卡牌按钮 |
| --- | --- | --- |
| `blood_tithe` 起手 | `抽1找MP1 · 仍-17` | `抽1仍-17` |
| `pulse_draw` 正确接 0 费 | `抽2找MP2 · 仍-17` | `抽2仍-17` |
| `pulse_draw` 断链或非起手 | `断x1 抽1 · 仍-17` | `抽1仍-17` |
| `wild_mana_stitch` 命中缺口 | `修补链 · 抽1 · 仍-17` | `修补仍-17` |
| `wild_mana_stitch` 未命中缺口 | `抽1返MP · 不降意图` | `不降意图` |
| `paper_shatter` 正确接链 | `抽3找终结 · 仍-17` | `抽3仍-17` |
| 牌堆不足 | `抽到1 · 仍-17` | `抽到1` + `仍-17` |

其中 `仍-17` 的数字来自当前 `enemyIntentSummary.totalDamage` 或同口径敌意图总和。不要写成 `减压`、`解除`、`安全`、`已修复`。

### 3.2 `card-intent-preview` 的 P0 改法

当前无伤害抽牌牌的 `card-intent-preview` 只显示 `抽N找解`。建议把它改成默认带压力后缀：

```text
抽N找解 · 仍-N
```

移动端空间不足时压成：

```text
抽N仍-N
修补仍-N
不降意图
```

优先级如下：

1. 如果牌会造成击杀或清掉意图，显示 `意图 N->M`。
2. 如果牌不造成伤害但会抽牌，显示 `抽N仍-N`。
3. 如果牌不造成伤害但会返 MP / 修补，显示 `修补仍-N` 或 `返MP仍-N`。
4. 如果牌没有当前压力价值，显示 `不降意图`。

这会把“找解”从成功反馈改成风险反馈：你正在找答案，但答案还没兑现。

### 3.3 Combat Director 的 P0 改法

Director 不需要新增长说明，只需要在修补牌打出后的下一帧维持压力句：

```text
敌人意图：结束回合 -17 HP
行动提示：已找牌，未解意图
```

移动端更短：

```text
仍-17
未解意图
```

不建议在 Director 写完整解释：

```text
你刚才使用了一张修补牌，它只修补费用链，不会降低敌人的攻击意图
```

这类解释只能进入 tooltip / 详情，不应进入常驻 HUD。

### 3.4 战斗日志的 P0 改法

出牌日志需要把抽牌收益和压力保留放在同一条短句里。

推荐：

```text
Pulse Draw x2 · 抽2 · 意图仍-17
Blood Tithe x1 · 抽1 · 意图仍-17
Wild Stitch · 修补链 · 意图仍-17
Paper Route x3 · 抽3 · 意图仍-17
```

若日志宽度不足，优先砍掉英文牌名的冗余，而不是砍掉 `仍-17`：

```text
脉冲 x2 抽2 仍-17
献血 抽1 仍-17
缝合 修补 仍-17
纸路 x3 抽3 仍-17
```

如果后续攻击或 payoff 真的降低意图，日志再显示：

```text
命中 BRU -18 · 意图17->12
终结 x3 · 意图17->0
```

这样玩家能读出两段因果：修补没有降压，兑现才降压。

### 3.5 卡牌静态文案的 P0 改法

静态 `rulesText` / `mobileEffect` 不必全都加长，但详情和 tooltip 必须明确边界：

| 卡 | 静态短文案建议 | 详情 / tooltip 边界 |
| --- | --- | --- |
| `blood_tithe` | `抽1。开链找牌。` | `本张不造成伤害，不降低敌意图。` |
| `pulse_draw` | `抽1。接链抽2。` | `用于找2费段；抽到后仍需打出降意图牌。` |
| `wild_mana_stitch` | `修补缺口。抽1，当前MP+1。` | `修补费用链，不等于取消本回合敌意图。` |
| `paper_shatter` | `抽1。接链抽3找终结。` | `整备是找牌语义；当前不提供手动重排牌库。` |

移动端不要把这些长边界句塞进按钮主体。按钮主体只保留 `抽N仍-N` / `修补仍-N`。

### 3.6 视觉反馈的 P0 改法

修补牌的视觉语气应是“短暂续线”，不是“压力清空”：

- 修补 / 抽牌反馈使用黄 / 青的资源反馈，不使用绿色清除、胜利闪光或敌意图熄灭动画。
- 打出修补牌后，敌人意图 badge 不消失、不淡出、不变安全色。
- End Turn 上的 `-N` 不变化，除非后续真实击杀或 payoff 降低意图。
- 可以给手牌 / 链路一个短促亮起，表示“路线继续”；不要给敌人一个被打断的反馈。

## 4. 验收标准

### 4.1 体验验收

P0 通过标准：

- 玩家看到 `blood_tithe` / `pulse_draw` / `wild_mana_stitch` / `paper_shatter` 时，能在卡牌按钮主可见区读到“抽牌 / 修补”和“意图仍在”。
- 打出修补牌后，End Turn 后果不被隐藏；如果意图仍是 17，按钮 / Director / 状态条至少一个主视线区域仍显示 `-17`。
- 战斗日志不能只记录 `抽2`、`返MP`、`修补链`；必须同条或紧邻显示 `意图仍-N`。
- 只有造成击杀、全场伤害、打断或真实降意图效果时，HUD 才显示 `意图 N->M`。
- 任何“整备 / 重排”文案不得让玩家以为可以手动控顶或免费改牌序，除非运行时真的实现了该行为。

### 4.2 文案验收

P0 禁止文案：

```text
免费解困
解除压力
安全
已修复
重排牌库
取消意图
```

P0 推荐文案：

```text
抽2仍-17
修补仍-17
不降意图
抽3找终结 · 仍-17
已找牌，未解意图
```

桌面 tooltip 可以写完整边界：

```text
本张只抽牌 / 修补费用链，不降低当前敌人意图；仍需后续出牌兑现。
```

移动端按钮不可依赖 tooltip。

### 4.3 UI 不超框验收

P0 视口：

- `320x568`
- `360x640`
- `390x844`
- `430x932`
- `640x360`
- `768x1024`
- `1366x768`

P0 DOM / 视觉标准：

- `.card-row` 在 `<=640px` 继续是横向 rail；卡数增加不得增高手牌区。
- `.card-button` 移动端高度保持当前预算，不能为了长解释增高。
- `.card-intent-preview` 在移动端使用 `抽2仍-17`、`修补仍-17`、`不降意图` 这类短 token；不得出现完整句。
- `.card-button strong`、`.card-meta`、`.chain-preview`、`.card-intent-preview`、`.missing-cost`、`.authorization-cost` 都必须落在对应卡牌按钮内。
- `scrollWidth <= clientWidth` 应作为主验收；若低优先级牌名允许省略，不能省略 `仍-N`、`缺MP`、`授权付`、`断x1` 这类决策信息。
- `.combat-feed li` 桌面需要单行裁切或短句预算，不能因为 `意图仍-17` 加入后覆盖手牌、Director 或目标面板。
- 奖励面板仍保留 viewport max-height 和内部滚动，不因为修补说明加入而撑出视口。

建议短句预算：

| 位置 | 推荐上限 | 示例 |
| --- | ---: | --- |
| 移动 `.card-intent-preview` | 6-8 个中文等宽 | `抽2仍-17`、`修补仍-17` |
| 桌面 `.card-intent-preview` | 12-14 个中文等宽 | `抽2找MP2 · 仍-17` |
| 日志短句 | 18-24 个中文等宽 | `脉冲 x2 抽2 仍-17` |
| Director 主句 | 8-12 个中文等宽 | `结束-17HP`、`未解意图` |
| Tooltip / detail | 可长 | 完整解释机制边界 |

### 4.4 QA 场景验收

必须覆盖以下固定场景：

1. 初始发牌后当前意图为正，检查攻击牌显示 `意图 N->M`，抽牌牌不显示降意图。
2. `Debt Hook -> Pulse Draw` 正确接链，`Pulse Draw` 出牌前按钮显示 `抽2仍-N` 或等价短句。
3. 打出 `Pulse Draw` 后日志显示抽牌数，同时显示 `意图仍-N`。
4. 选择 `Blood Tithe` 作为奖励进入下一手，按钮显示 `抽1仍-N`，不是只显示 `抽1`。
5. `Wild Mana Stitch` 可修补缺口时，按钮显示 `修补仍-N`；敌人意图 badge 不消失。
6. `Paper Route` 正确接链时显示 `抽3`，但不出现“重排牌库”或“控顶”承诺。
7. 移动端 `390x844` 与 `360x640` 连续 tap 发牌、顺链、修补、End Turn，确认按钮内文字不重叠、不超框，单 tap 不重复触发。

## 5. 本轮结论

第 6 轮应该把修补牌定位成“承压找路线”的机制，而不是“免费解困”的补偿。当前 runtime 已经符合“修补不直接降意图”的底层事实，但 HUD 和日志还需要把这件事贴到玩家最常看的地方。

最小 P0 改动方向：

```text
卡牌按钮：抽N找解 -> 抽N仍-N
修补按钮：修补/抽1 -> 修补仍-N
战斗日志：xN · 抽N -> xN · 抽N · 意图仍-N
Director：继续显示结束 -N，必要时补“未解意图”
移动端：只用短 token，不把完整解释塞进卡面
```

做到这一步，修补牌会被玩家读成“救了一手但仍承压”：它给玩家一次把坏手救回来的机会，但不会替玩家承担敌意图的后果。
