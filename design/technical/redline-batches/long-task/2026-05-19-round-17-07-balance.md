## 0. 结论

本轮没有发现必须改 runtime 常量的数值越界。第16轮已有的 `maxEnergy` 封顶和第15/16轮的升级曲线收口当前能支撑第17轮 build plan 边界：

- 路线奖励宽度只允许窄幅变化：`repair-cache` 相比 `elite-pressure` 只多 1 个 reward pick 语义宽度。
- `maxEnergyThisRunPlusOne` 在预览层和 runtime 重复选择时都只推到 4，不会从 3 堆到 5。
- 升级奖励保持单次小收益：单个升级选择的 `damageBonusPreview <= 3`，`redline_cut +1` 后仍低于 3 MP payoff 身份。
- 单条路线不同时打包奖励宽度、Max MP 和升级槽，避免“选这条就全都要”的无脑最优。

## 1. 新增测试

新增：

- `prototype-web/src/tests/sim/redline-round-17-balance.test.ts`

覆盖 4 个边界：

1. 路线奖励宽度  
   `repair-cache` 的 `rewardPickBonus` 精确为 1，`elite-pressure` 为 0；通过 `buildRewardChoices()` 验证两者实际宽度只差 1，并且 repair 路线仍覆盖 `repair-resource / payoff / route-bridge` 三类，不把奖励池挤成单一答案。

2. `maxEnergy` 上限  
   `deriveRunModifierPlan()` 对重复 `maxEnergyThisRunPlusOne` 去重，预览为 `3 -> 4`；runtime 连续两次选择 `elite-pressure` 后 `player.maxEnergy` 仍为 4。

3. 升级收益边界  
   `buildCardUpgradeChoices(redline_cut)` 首次只给 `raise-level / add-gem-slot` 两类增量选择；单次 `damageBonusPreview` 不超过 3，`redline_cut +1` 后仍低于 `severance_burst`；满等级后不再继续提供 `raise-level`。

4. build plan 不叠满  
   遍历路线候选，要求每条路线最多只带一个关键经济杠杆：奖励宽度、Max MP、升级槽三者不能同时出现。

## 3. 风险

当前测试证明的是“系统不会在数据层直接叠成无脑最优”，还不能证明玩家真实选择率均衡。后续仍要观察：

- 如果玩家总选 `repair-cache`，说明奖励宽度和修补倾向仍比高压收益更稳定。
