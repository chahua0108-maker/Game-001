## 0. 制作人裁决

第18轮不是“再加一个系统”的轮次，而是把第16-17轮已经接起来的链条压成一个能被玩家记住的短 run：

```text
节点1暴露短板 -> 奖励/路线形成 build plan -> 节点2验证并施压 -> 节点3迫使调整 -> 节点4-5兑现或失败
```

如果玩家只看到奖励、路线、升级都能工作，但每个节点像独立房间，本轮不得记到 `95`。  
如果玩家能复述“我为什么从补桥转向清污染，为什么后来补终结，以及失败是因为哪一次压力没处理”，本轮才算达成核心体验目标。

本轮必须坚持两个边界：

- 只验证单次冒险内部的连续节点体验。
- 不引入永久局外成长、账号升级、长期解锁或 meta 经济。

QA 继续只作门禁，不计入核心体验分。测试数量、gateScore、截图数量都不能替代玩家视角的 run 体验判断。

## 1. 10个玩家视角镜头

以下镜头用于验收“玩家实际看到和感受到什么”，不是工程 checklist。

1. 玩家进入节点1，第一手牌能明显看出一个问题：断链、污染卡手、缺终结或资源不足，而不是一开局就顺滑解完。
2. 玩家打完节点1后，奖励候选不是随机好牌，而是至少出现两个不同 build plan 方向，例如“补桥”和“清污染”。
3. 玩家选择路线时，能看懂路线会提高下一节点压力、奖励倾向或资源承压能力，而不是只看到抽象路线名。
4. 玩家进入节点2时，上一节点选择必须有可见后果：桥更稳、污染少一点、资源更宽，或压力更高。
5. 玩家在节点2遇到污染/失败压力升级：污染不再是装饰，而是实际造成抽牌、手牌、节奏或血量压力。
6. 玩家做第二次奖励选择时，build plan 需要演变：继续强化原计划、补新暴露短板，或放弃原路线转向保命。
7. 玩家进入节点3时，HUD/奖励/路线至少有一个地方能让玩家读出“当前计划是什么，当前风险是什么”。
8. 玩家在节点3或节点4看见一次坏结果：没有清污染导致卡手、没有补桥导致断链、没有补资源导致高压路线吃亏。
9. 玩家在节点4或节点5能看到兑现：终结牌更容易出现、关键牌被强化、资源足够承压、污染被处理后 combo 变顺。
10. run 结束或失败后，玩家能用一句话复述本次冒险：“我先补了 X，因为 Y 压力转去 Z，最后靠 W 兑现 / 因为没处理 V 失败。”

## 2. 3-5节点 journey 最低验收脚本

最低脚本要求连续跑同一次冒险内部的 `3` 个节点。优秀脚本延伸到 `5` 个节点。验收记录必须写出每个节点的短板、选择、压力变化和 build plan 演变。

### 2.1 最低合格：3节点

```text
节点1：暴露明确短板
- 例：起手有污染或断链，玩家看见当前 plan 是“补桥 / 清污染 / 补资源”之一。
- 通过条件：战斗结束后的奖励至少给出两个可读方向，且不是纯随机强度排序。

节点1奖励 + 路线：形成第一版 build plan
- 例：玩家选“补桥”，路线选择高压但奖励倾向更偏终结。
- 通过条件：路线说明能解释下一节点的收益或风险，并进入节点2。

节点2：验证第一选择，同时升级压力
- 例：桥确实更稳，但污染或血量压力上升。
- 通过条件：玩家能看出上一选择带来的好处，也能看出新风险。

节点2奖励：build plan 必须演变
- 例：从“补桥”转为“清污染”，或继续“补终结”准备爆点。
- 通过条件：奖励理由能解释为什么此时该转向或加码。

节点3：验证演变
- 例：污染处理后手牌更顺，或未处理污染导致明显坏手。
- 通过条件：节点3不能只是又打一场；必须能看出前两次选择共同塑造了当前局面。
```

3节点合格句：

```text
玩家能说清：节点1我发现了什么问题，节点2我验证了哪次选择，节点3我为什么转向或继续原 plan。
```

### 2.2 优秀验收：5节点

```text
节点1：暴露短板，奖励给出至少两个 plan。
节点2：验证第一次选择，并加入污染或高压路线副作用。
节点3：迫使玩家在清污染、补资源、补终结之间做一次有代价的转向。
节点4：出现一次失败边缘或坏手反例，证明压力不是装饰。
节点5：兑现或失败；兑现要能看到 build plan 成果，失败要能追溯到未处理短板。
```

5节点优秀句：

```text
玩家能复述整段 run：先解决 A，随后因为 B 压力转向 C，最后通过 D 兑现；或因为没有处理 E 而失败。
```
