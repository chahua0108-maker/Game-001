# 2026-05-18 Round 02-09：Redline 关键词与规则文案表

角色：关键词与规则文案专家  
范围：只建立 Redline 自己的关键词表，不复制竞品原文，不修改源码。  
目标：UI 可短、机制可测、详情可解释、移动端不超框。

## 0. 文案原则

1. **关键词不是气氛词。** 每个关键词必须对应至少一个结构化字段、一个可观测事件或状态变化、一个 QA 可断言结果。
2. **战斗内只写决策结果。** 卡牌按钮、敌人槽、结束回合按钮优先显示“能不能用、会改变什么”，不要把完整规则塞进按钮。
3. **详情文案解释边界。** 详情面板负责说明生命周期、支付限制、是否跨回合、是否跨战斗。
4. **不混用成长词。** “授权”“修补”“当前 MP”都属于本回合或本战斗节奏，不得写成永久成长、Max MP、局外升级。
5. **不靠自然语言驱动规则。** UI 文案从结构化字段派生；测试断言事件和状态，不从中文描述反推机制。

## 1. UI 长度合同

| 使用位置 | 建议长度 | 写法 |
| --- | ---: | --- |
| 卡牌按钮主 token | 2-6 个汉字，必要时加数字 | `授权+3`、`补2费`、`意图-5` |
| 卡牌按钮第二行 | 8-12 个汉字内 | `终结可用`、`保留到下回合` |
| 敌人槽 token | 2-8 个汉字 | `蓄力-10`、`可打断` |
| 结束回合按钮 | 8 个汉字内加数字 | `结束 -7`、`无伤结束` |
| 详情面板 | 每句 28 个汉字内，最多 2-3 句 | 解释限制和生命周期 |
| 移动端奖励卡 | 主结论 1 行，详情 2 行 clamp | 不塞完整机制历史 |

统一压缩规则：

- `本回合终局授权 +3` 在按钮内写 `授权+3`。
- `敌人意图将从 17 降到 12` 写 `意图17>12` 或 `意图-5`。
- `只支付终结牌` 写 `限终结`。
- `保留到下回合` 写 `保留`，详情再说明时间。
- 不在移动端依赖 hover/title；核心后果必须直接可见。

## 3. 关键词细则

### 3.1 授权

战斗内短文案：

- `授权+3`
- `限终结`
- `授权不足`

详情文案：

> 完成费用链后，获得本回合临时授权。授权只支付终结牌；未使用的授权在回合结束时清空。

测试口径：

- 成功链路后产生 `AuthorizationGranted` 或等价事件。
- 授权值等于规则给定值，例如 `+3`。
- 支付限制为 `payoff-only`，普通攻击、修补牌、状态牌不能误用授权。
- `EndTurn` 或离开玩家回合后授权清零。

避免超框：

- 卡牌按钮不要写“完成 0->1->2 后获得本回合临时授权”。
- 移动端统一显示 `授权+3`；详情面板再写完整句。

### 3.2 修补

战斗内短文案：

- `补1费`
- `补2费`
- `已修补`
- `无缺口`

详情文案：

> 修补会按当前期望费用接上链路。修补不等于免费支付；实际费用仍按牌面或资源规则结算。

测试口径：

- 链路未开始时，修补牌不能凭空完成中段，除非卡牌明确允许。
- 链路期望费用为 1 时，修补后记录为补 1；期望费用为 2 时，记录为补 2。
- 修补后 `nextExpectedCost` 前进，倍率按链路规则更新。
- 本回合修补次数限制由字段控制，不能靠文案约束。

避免超框：

- 用 `补N费` 代替“按当前链路缺口视为期望费用”。
- 卡牌详情保留“支付仍按牌面”这句，避免玩家误读。

### 3.3 终结

战斗内短文案：

- `终结`
- `已武装`
- `未武装`
- `清意图`

详情文案：

> 终结牌负责链路后的大结算。若使用授权支付，终结进入武装状态，优先清场或削减敌人意图。

测试口径：

- 只有满足 `payoff` 条件的牌显示终结状态。
- 授权支付终结时产生武装标记，未授权支付不应伪装为武装。
- 终结伤害、击杀、避免意图伤害都应进入 trace。
- 2 费路线段和 3 费终结牌必须可区分，不能只靠 `burst` 文案判断。

避免超框：

- 按钮写 `终结` + 状态 token。
- 详情写条件，不在按钮内写“完成链后用授权支付触发全场清算”。

### 3.4 意图

战斗内短文案：

- `意图-5`
- `17>12`
- `结束-7`
- `无伤`

详情文案：

> 意图是敌人在本回合结束时会结算的后果。玩家出牌后，剩余意图应立即更新，并与实际结算一致。

测试口径：

- 发牌阶段保存当前回合意图快照。
- 卡牌预览、敌人槽、结束回合按钮读取同一份意图数据。
- 击杀、打断、护栏后，预览伤害与结束回合实际 HP 变化一致。
- 补位敌人默认不继承本回合 active intent，除非事件明确声明。

避免超框：

- 敌人槽显示 `本回合-5` 或 `下轮`，不写完整句。
- 结束按钮显示最终扣血，详细拆分进详情或日志。

### 3.5 保留

战斗内短文案：

- `保留`
- `留牌`
- `下回合可用`

详情文案：

> 保留牌在回合结束时不会弃掉，会带到下回合。保留不复制牌，也不自动改变费用。

测试口径：

- 回合结束弃手牌时，保留牌不进入弃牌堆。
- 下回合开始后，保留牌仍在手牌或等价保留区。
- 保留状态在使用、消耗或到期时清除。
- 多张同名牌若只有一张保留，未来实例化后必须能区分。

避免超框：

- 卡面只显示 `保留`。
- 长解释放在详情；移动端不要写“回合结束时不会被弃掉”。

### 3.6 消耗

战斗内短文案：

- `消耗`
- `移出`
- `一次性`

详情文案：

> 消耗牌在本场战斗内移出循环。它不会进入弃牌堆，也不会被洗回抽牌堆。

测试口径：

- 打出后进入 `exhaustPile` 或等价消耗区。
- 抽牌堆空时洗弃牌堆，不包含已消耗牌。
- 战斗结束后按卡牌来源决定是否回到牌组，不能由 UI 文案决定。
- 临时牌若同时消耗和战斗后清理，以生命周期字段为准。

避免超框：

- 用 `消耗`，不用“本场战斗内从抽弃循环中移除”。
- 详情区说明“不是永久删牌”或“是否永久由来源决定”。

### 3.7 状态

战斗内短文案：

- `状态+1`
- `血印2`
- `塞状态`

详情文案：

> 状态是战斗内的临时规则层或污染牌。状态必须写明来源、层数、持续时间和触发点。

测试口径：

- 施加状态后，目标实体或牌区出现对应状态实例。
- 层数叠加、持续时间衰减、回合触发按字段执行。
- 正面状态和负面状态有明确分类，净化不能误删正面状态。
- 状态牌被抽到、弃掉、消耗时有对应事件。

避免超框：

- chip 用 `名称+层数`，例如 `血印2`。
- 详情再写“回合末触发”或“抽到时触发”。

### 3.8 诅咒

战斗内短文案：

- `诅咒+1`
- `诅咒入堆`
- `罚单入堆`

详情文案：

> 诅咒是重负面牌或长期代价。它通常比状态更难移除，是否跨战斗保留必须由规则字段说明。

测试口径：

- 获得诅咒后，诅咒实例进入指定牌区。
- 抽到诅咒时触发负面效果或占用手牌，具体由字段决定。
- 净化、事件奖励或战斗结束清理能按规则移除。
- 诅咒不应被普通状态清理误删，除非净化目标包含 `curse`。

避免超框：

- 按钮写 `诅咒+1`。
- 不在敌人槽写长句“结束回合后向你的弃牌堆加入一张诅咒牌”。

### 3.9 净化

战斗内短文案：

- `净化1`
- `清状态`
- `清诅咒`
- `无负面`

详情文案：

> 净化移除指定负面对象。它可以清状态、状态牌或诅咒，但每张牌必须明确自己的目标范围。

测试口径：

- 有合法负面目标时，净化后目标减少指定数量。
- 无合法目标时，按钮禁用、灰显或显示 `无负面`。
- 清状态、清诅咒、清牌堆污染应分别断言，不能用一个模糊事件覆盖。
- 净化不默认回血、不默认抽牌，除非效果字段另写。

避免超框：

- `净化1` 是默认最短写法。
- 目标范围用第二 token：`清诅咒`、`清状态`。

### 3.10 打断

战斗内短文案：

- `打断`
- `断蓄力`
- `取消召唤`
- `降级`

详情文案：

> 打断处理敌人的特殊意图。成功后，原意图会被取消、降级或延迟；普通攻击是否保留由敌人规则决定。

测试口径：

- 只有带可打断窗口的意图显示 `可打断`。
- 打断成功后，敌人意图类型和结算结果发生结构化变化。
- 结束回合不能结算已取消的原特殊意图。
- 对不可打断意图使用打断牌时，应显示无效、降效或不允许选择。

避免超框：

- 敌人槽写 `可打断`。
- 卡牌按钮写 `打断`；详情写“取消/降级/延迟”的具体结果。

### 3.11 护栏

战斗内短文案：

- `护栏+6`
- `挡6`
- `挡后-2`

详情文案：

> 护栏是承受伤害前的缓冲层。它先吸收敌人意图伤害，再由剩余伤害扣 HP。

测试口径：

- 获得护栏后，玩家护栏值增加。
- 敌人意图结算时先扣护栏，再扣 HP。
- 护栏到期规则明确：回合末清空、逐回合衰减或持续到战斗结束。
- 多段伤害按顺序消耗护栏，不能每段都重复使用同一护栏值。

避免超框：

- 顶部状态写 `护栏6`。
- 结束按钮可写 `挡后-2`，详情解释原始意图和护栏抵扣。

### 3.12 整备

战斗内短文案：

- `整备2`
- `看2`
- `看2留1`
- `抽1`

详情文案：

> 整备用来整理接下来的手牌节奏。它可以预览、重排、抽取或弃置牌堆顶，但每张牌只能承诺已实现的动作。

测试口径：

- 整备数量与实际可见牌数一致。
- 玩家选择后，抽牌堆、弃牌堆、手牌变化可断言。
- 如果只实现抽牌，不得在 UI 显示重排承诺。
- 整备后的下一张抽牌必须与牌堆顺序变化一致。

避免超框：

- 按钮只写 `整备N` 或具体动作 `抽1`。
- 详情面板写“看、留、弃、抽”的完整顺序。

## 4. 推荐事件与字段口径

这些不是本轮源码改动要求，只是后续实现时让 QA 能稳定断言的命名建议。

| 关键词 | 建议字段 | 建议事件 |
| --- | --- | --- |
| 授权 | `tempAuthorization`, `authorizationRestriction` | `AuthorizationGranted`, `AuthorizationSpent`, `AuthorizationExpired` |
| 修补 | `chain.repairedThisTurn`, `repairMode`, `countsAsCost` | `ChainRepaired`, `GapRepaired` |
| 终结 | `cardType: payoff`, `payoffArmed` | `PayoffTriggered`, `PayoffResolved` |
| 意图 | `enemyIntents`, `enemyIntentSummary` | `EnemyIntentSnapshot`, `IntentReduced`, `IntentResolved` |
| 保留 | `lifecycle.onTurnEnd: retain` | `CardRetained`, `RetainedCardReleased` |
| 消耗 | `lifecycle.onPlay: exhaust` | `CardExhausted`, `CardMoved` |
| 状态 | `statuses[]`, `statusType` | `StatusApplied`, `StatusTicked`, `StatusRemoved` |
| 诅咒 | `cardType: curse`, `cursePersistence` | `CurseAdded`, `CurseTriggered`, `CurseRemoved` |
| 净化 | `cleanseTarget`, `cleanseCount` | `CleanseResolved`, `CleanseFailed` |
| 打断 | `interruptWindow`, `interruptResult` | `IntentInterrupted`, `IntentDowngraded` |
| 护栏 | `guard`, `guardDuration` | `GuardGained`, `GuardAbsorbed`, `GuardExpired` |
| 整备 | `prepareCount`, `prepareMode` | `DeckPrepared`, `CardsReordered` |

## 5. 禁用写法

| 禁用写法 | 问题 | 改写 |
| --- | --- | --- |
| `获得 3 点永久授权` | 误导为局外或永久成长 | `本回合授权+3` |
| `修补免费补任意费用` | 误导支付规则 | `补当前缺口，支付按牌面` |
| `终结牌总是清场` | 承诺过度 | `武装后高额清场` |
| `敌人将会进行攻击并造成若干伤害` | 按钮过长 | `结束-7` |
| `保留牌复制到下一回合` | 机制错误 | `保留到下回合` |
| `消耗后永久删除` | 未必符合来源规则 | `本战斗移出` |
| `状态就是诅咒` | 分类混乱 | `状态` 与 `诅咒` 分开 |
| `净化全部异常` | 目标不清 | `净化1` / `清诅咒` |
| `打断所有敌人行动` | 承诺过度 | `打断特殊意图` |
| `护栏让你无敌` | 承诺过度 | `护栏+N，先挡伤害` |
| `整备后抽到好牌` | 不可测 | `看N / 留N / 抽N` |

## 6. 最小验收清单

- 12 个关键词都有短文案、详情文案、测试口径和超框规避策略。
- 按钮文案不超过 12 个汉字；移动端核心结果不依赖 hover。
- 每个关键词至少有一个结构化断言点，不能只靠中文描述验收。
- 授权、修补、终结明确写清“本回合”“链路”“支付限制”，不暗示永久成长。
- 状态、诅咒、净化的目标范围分清，避免以后 QA 无法判断清理对象。
- 意图、打断、护栏都能落到结束回合 HP 变化或敌人意图变化上。
- 整备只承诺已实现动作；未实现重排时，UI 不得显示“重排成功”。
