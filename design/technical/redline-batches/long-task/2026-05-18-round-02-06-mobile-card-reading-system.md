目标：基于当前 HUD 已有“意图预览”和移动端横向手牌 rail，设计 `rulesText`、`mobileEffect`、`keyword`、`detail` 四层短文本系统，保证后续补齐完整卡牌机制后，卡牌仍不依赖省略号硬裁，也不发生文字超框。

## 1. 当前阅读面结论

当前 `prototype-web/src/ui/hud.ts` 的卡牌按钮已经有一套可用的战斗决策骨架：

- `card-cost`：费用，移动端最稳定的视觉锚点。
- `strong`：卡名。
- `card-meta`：角色 + 目标，例如终结 / 全场。
- `chain-preview`：起链、接链、断链。
- `card-intent-preview`：动态意图预览，例如 `意图 12->7`。
- `card-payoff` / `card-effect`：当前在小屏被隐藏。
- `missing-cost` / `authorization-cost`：支付状态。

当前移动端 CSS 已经把 `.card-row` 改成横向滚动 rail，单张卡固定为 `flex: 0 0 min(232px, 78vw)`、高 `98px`。这比两列网格更安全，但它也意味着每张卡只有约 5 行可读空间。后续补防御、状态、消耗、保留、负面牌、升级分支后，如果继续把自然语言 `description` 塞进按钮，会不可避免地被裁掉。

所以本轮建议的核心不是“再给卡牌加一段文字”，而是把卡牌文本拆成四层：

1. `mobileEffect`：移动手牌里永远可见的最短决策句。
2. `rulesText`：卡牌规则的短版权威文本，用于奖励、桌面、详情首段。
3. `keyword`：可复用机制词，不让每张牌重复解释同一规则。
4. `detail`：长说明，只进入详情 sheet / 图鉴 / debug，不进入手牌按钮。

## 2. 字段建议

### 2.1 推荐字段合同

```ts
type KeywordId =
  | '开链'
  | '接链'
  | '修补'
  | '终结'
  | '授权'
  | '意图'
  | '护栏'
  | '抽牌'
  | '返MP'
  | '消耗'
  | '保留'
  | '状态'
  | '过载'
  | '净化'
  | '重排';

interface CardReadingText {
  rulesText: string;
  mobileEffect: string;
  keyword: KeywordId[];
  detail: string;
}
```

字段职责：

| 字段 | 必填 | 显示位置 | 职责 | 禁止承担的职责 |
| --- | --- | --- | --- | --- |
| `mobileEffect` | 是 | 移动端手牌按钮、横屏手牌 rail | 一眼判断这张牌做什么，例如 `单体9`、`前排5`、`抽1`、`全场16` | 不解释授权、构筑目的、完整条件 |
| `rulesText` | 是 | 桌面卡面、奖励卡、详情首段、无障碍标签 | 权威短规则，最多两句 | 不写设计意图、不写“不是永久成长”这类反复教育 |
| `keyword` | 是，允许空数组 | 卡面短 chip、详情词条 | 把复用机制收敛为稳定 token | 不堆 5 个以上标签，不替代数值效果 |
| `detail` | 是 | 长按 / info / 图鉴 / debug 详情 | 完整解释、边界、教学补充 | 不进入 `.card-button` 主体，不作为移动端唯一规则来源 |

### 2.2 字段优先级

移动端手牌按钮里的优先级应固定：

1. 费用与可支付状态。
2. 卡名或短名。
3. `mobileEffect`。
4. 动态 `card-intent-preview`。
5. 链路 / 授权状态。
6. 关键词 chip，空间不足时只保留 1 个主关键词。

注意：`card-intent-preview` 是战斗态动态计算，不应该写进卡牌数据。`mobileEffect` 只描述基础效果，动态目标、倍率、授权支付、意图变化仍由 HUD 计算并独立显示。

## 3. 文案长度预算

### 3.1 移动手牌预算

按当前移动端单卡宽度约 232px、高 98px 估算，中文系统字体下推荐预算如下：

| 元素 | 建议上限 | 强制上限 | 超出处理 |
| --- | ---: | ---: | --- |
| 卡名 `name` | 8-12 个中文字等宽 | 14 个中文字等宽 | 增加 `shortName` 或裁掉副标题，不用自动缩放字体 |
| `mobileEffect` | 6-10 个中文字等宽 | 12 个中文字等宽 | 必须重写，不建议省略号 |
| `keyword` 单个 label | 2-4 字 | 5 字 | 进入词条，不在卡面展示长词 |
| 移动端可见关键词数量 | 1-2 个 | 3 个 | 第 3 个以后只进详情 |
| `chain-preview` | 6-8 字 | 10 字 | 用 `0>1`、`接x2`、`断x1` 这类短格式 |
| `card-intent-preview` | 8-12 字 | 14 字 | 用 `意图12>7`，不要写完整句 |
| 支付状态 | 4-8 字 | 10 字 | `缺MP1`、`授权付`，不要写 `需MP/终局授权` |

移动端手牌不应该显示 `rulesText` 全文。它只显示 `mobileEffect` 加动态预览。

### 3.2 `rulesText` 预算

`rulesText` 是规则短版，不是详情。建议：

- 中文 18-28 字为佳。
- 最多 2 句，最多 2 行。
- 每句只表达一个规则动作。
- 数字用阿拉伯数字。
- 条件词固定化：`若续链`、`若击杀`、`授权时`、`回合末`。
- 不重复解释关键词含义。

推荐格式：

| 类型 | 模板 |
| --- | --- |
| 单体攻击 | `造成 9。` |
| 前排攻击 | `对前排造成 5。` |
| 过牌 | `抽 1。若续链，抽牌受倍率影响。` |
| 修补 | `修补缺口。抽 1，当前MP+1。` |
| 终结 | `全场 16。可用终局授权支付。` |
| 防御 | `获得 6 护栏。` |
| 意图处理 | `造成 4，并使目标意图-2。` |
| 状态牌 | `无效果。打出后消耗。` |
| 风险资源 | `当前MP+2。加入 1 张过载。` |

### 3.3 `detail` 预算

`detail` 不进手牌按钮，因此可以承担完整解释，但仍应克制：

- 建议 60-120 中文字。
- 强制上限 180 中文字。
- 超过 180 字拆到关键词词典、图鉴条目或设计文档。
- 详情 sheet 必须可滚动，不能把卡牌按钮撑高。
- `detail` 可以解释“当前 MP 不会保留到后续回合”“授权只支付 3 费终结牌”等边界，但不要每张同机制卡都重复长段落。

## 4. 关键词设计

### 4.1 关键词不是标签垃圾桶

`keyword` 应只放玩家需要复用记忆的机制词。攻击 / 自身 / 第一排这类目标信息不建议都塞进关键词，因为当前 `card-meta` 和 `mobileEffect` 已经能表达。

建议优先关键词：

| 关键词 | 卡面短解释 | 详情词条解释方向 |
| --- | --- | --- |
| 开链 | 从 0 费开始连锁 | 作为费用升序链的入口。 |
| 接链 | 延续费用升序 | 与上一张费用相邻时提高倍率。 |
| 修补 | 补当前缺口 | 可按当前期望费用计入链路。 |
| 终结 | 高价值收束牌 | 通常消耗授权或高费用，解决多目标压力。 |
| 授权 | 本回合临时支付 | 只在本回合有效，按限制支付指定牌。 |
| 意图 | 敌人回合伤害 | 清掉有意图敌人可降低结束回合伤害。 |
| 护栏 | 防御资源 | 本回合抵消敌意图或直接伤害。 |
| 消耗 | 本战移出循环 | 打出后进入消耗堆，不再进弃牌循环。 |
| 保留 | 留到下回合 | 回合末不弃掉。 |
| 状态 | 临时污染牌 | 通常占手牌，可能可消耗或需净化。 |
| 过载 | 风险账单 | 由强资源牌制造，未来回合产生压力。 |
| 净化 | 移除污染 | 移除状态、负面或 debuff。 |
| 重排 | 调整抽牌顶 | 预视、置顶、过滤下一轮抽牌。 |

移动手牌默认只显示 `keyword[0]`。如果卡牌是新机制教学牌，可显示 `keyword[0..1]`，但不得超过 2 个 chip。完整关键词列表进入 detail。

### 4.2 关键词显示规范

- 关键词 chip 不应该独占一整行；应和 `card-meta` 或 `mobileEffect` 共享紧凑区域。
- 单个 chip 高度建议 16-18px，字重 800，避免 24px 以上的标签条。
- 关键词解释不使用 hover 作为唯一入口；移动端通过详情 sheet 或词典入口查看。
- 关键词顺序有语义：最影响操作判断的排第一，例如 `修补` 优先于 `抽牌`，`终结` 优先于 `授权`。

## 6. CSS / DOM 要求

### 6.1 DOM 分层

建议后续实现保持按钮内分层稳定：

```html
<button class="card-button" data-card-id="...">
  <span class="card-cost">...</span>
  <strong class="card-title">...</strong>
  <span class="card-meta">角色 / 目标</span>
  <span class="card-mobile-effect">单体9</span>
  <span class="card-intent-preview">意图12&gt;7</span>
  <span class="card-state-line">接x2 / 授权付 / 缺MP1</span>
  <span class="card-keywords">修补</span>
</button>
```

DOM 规则：

- `mobileEffect` 必须是独立节点，不要混进 `title` 或 `aria-label`。
- `rulesText` 不进入移动手牌按钮主体，可进入桌面卡、奖励卡或 detail sheet。
- `detail` 只进入详情容器，不进入 `.card-button`。
- `keyword` 使用稳定 ID 渲染，UI 不从自然语言 `description` 正则推断关键词。
- 动态意图预览继续独立计算；不要把 `mobileEffect` 写成 `意图12>7` 这种战斗态文案。

### 6.2 移动手牌尺寸

在当前横向 rail 方案下，建议继续固定卡牌高度，不让文案决定卡高：

- 竖屏手机：单卡高度 98-108px，最多 112px。
- 横屏手机：单卡高度 82-92px，隐藏次级关键词，只保留费用、短名、`mobileEffect`、支付 / 意图状态。
- 单卡宽度：保持 `min(232px, 78vw)` 或同级固定宽，不因文字变宽。
- `.card-row` 只允许横向滚动，不允许因为卡牌多而纵向增高。
- 出牌 / 抽牌导致卡牌数量变化时，rail 高度不变。

### 6.3 文本 CSS 约束

所有动态文本节点必须满足：

- `min-width: 0`。
- 单行短字段使用 `overflow: hidden; text-overflow: ellipsis; white-space: nowrap;`，但字段预算应保证正常情况下不触发省略号。
- `rulesText` 在非手牌位置可用 `line-clamp: 2`，不可无限撑高 reward card。
- `detail` 容器使用 `max-height: calc(100dvh - 32px); overflow: auto;`。
- 不使用 `font-size: vw` 动态缩字。
- 不用 hover / `title` 作为移动端核心说明。
- 支付状态、不可打原因、授权支付优先级高于关键词；空间不足时隐藏关键词，不隐藏支付状态。

### 6.4 详情入口

移动端不建议把“点卡查看详情”和“点卡出牌”混在同一个短按行为里。推荐二选一：

1. 长按卡牌 350-450ms 打开详情，短按仍出牌。
2. 卡牌右上角增加 32-36px info 热区，点击打开详情，主体短按出牌。

若采用 info 热区，实际触控热区应至少 44px，可视觉上保持小图标但扩大 hit area。详情 sheet 内容顺序：

1. 卡名、费用、目标。
2. `rulesText`。
3. 当前战斗计算结果：倍率、目标、意图变化、支付方式。
4. `keyword` 词条解释。
5. `detail`。

打开详情时应暂停或阻断卡牌出牌 pointer 事件，避免长按后误触出牌。

## 7. 测试建议

### 7.1 文案数据测试

为所有卡牌增加数据级检查：

- 每张可进入 hand / reward 的卡都有 `rulesText`、`mobileEffect`、`keyword`、`detail`。
- `mobileEffect.length` 不超过预算；中文、数字、英文按等宽近似统计。
- `rulesText` 不超过 2 句 / 2 行预算。
- `keyword` ID 必须在关键词注册表里。
- `keyword.length` 可大于移动端显示数量，但卡面只渲染前 1-2 个。
- `detail` 允许较长，但超过 180 中文字时测试提示拆分。

### 7.2 DOM 布局测试

建议用 Playwright / Chromium 增加 HUD 极限快照：

- 视口：`320x568`、`360x640`、`390x844`、`430x932`、`640x360`、`844x390`、`768x1024`、`980x720`。
- 手牌：4 张、6 张、8 张。
- 卡牌文本：最长中文名、最长英文名、最长 `mobileEffect`、3 个关键词。
- 状态：授权 active / inactive、缺 MP、断链、接链、终结可打、终结不可打。
- 敌人意图：0、1、3 个敌人有意图，验证 `card-intent-preview`。

自动断言：

- `.card-row` 高度固定，不随手牌数量增加。
- 每个 `.card-button` 的 `scrollHeight <= clientHeight`。
- `.card-title`、`.card-mobile-effect`、`.card-intent-preview`、支付状态节点的 rect 均落在按钮 rect 内。
- `.card-row` 不与 `.run-layer-panel`、`.enemy-peek`、`.deal-panel` 发生不可接受重叠。
- detail sheet 打开后不会让背景手牌可点。

### 7.3 交互测试

- 短按可出牌，长按 / info 可打开详情，二者不互相触发。
- 禁用卡点击不会出牌，但仍可打开详情。
- 详情 sheet 可滚动到 `detail` 底部。
- 横屏下打开详情仍可关闭，关闭按钮触控区域不小于 44px。
- 使用系统大字体或浏览器 125% zoom 时，手牌按钮不超框，详情可滚动。

### 7.4 视觉回归测试

建议保存以下截图：

- 390x844：4 张常规手牌。
- 390x844：8 张极限手牌，含长名、状态、终结、授权。
- 320x568：小屏短高压力。
- 640x360：横屏手机。
- 390x844：奖励三选一，每张 reward 展示 `rulesText` 两行。
- 390x844：卡牌详情 sheet。

失败输出应包含重叠矩形 JSON，而不只是截图。

## 8. 验收标准

P0 验收：

- 未来 28-30 张卡以内，不需要把完整 `description` 放进手牌按钮。
- 移动端手牌按钮只显示 `mobileEffect`，不卡死在 `rulesText`。
- 所有关键支付 / 授权 / 意图状态不被关键词挤掉。
- 320x568 与 640x360 下无卡牌按钮文字超框。
- 奖励卡最多显示 `rulesText` 两行，完整说明进详情。

P1 验收：

- 状态、消耗、保留、护栏、过载、净化、重排加入后，只新增关键词和结构化效果，不扩写每张卡长句。
- 详情 sheet 能解释完整机制，但不影响短按出牌。
- 中英文混排卡名、长怪物名、长关键词词条都不会撑高卡牌按钮。

一句话结论：移动端手牌应该读 `mobileEffect + 动态意图 + 支付状态`，不是读完整规则书。`rulesText` 负责短规则，`keyword` 负责复用机制，`detail` 负责长解释；四层拆开后，完整卡牌机制扩展才不会把横向手牌重新撑爆。
