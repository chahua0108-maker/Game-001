# 2026-05-19 核心体验相似度 100 分制控制表

## 0. 停止条件

用户目标：

```text
越像竞品越成功；改为 100 分制；核心体验达到 95 分才停；QA 相关不算成绩。
```

硬停止条件：

- 核心体验总分 `>= 95 / 100`。
- 最大轮次 `30`；到第 30 轮仍未达 95，则停止扩系统，改为只做缺口清单和人工验收。
- `npm run check` 通过。
- `npm run qa:similarity` 通过。
- `npm run qa:ui` 通过。
- `npm run qa:lifecycle` 通过。
- 浏览器验收后 `page/context/browser/server` 全部关闭，端口无残留。
- 无玩家可见文字超框、横向页面溢出、raw debug token 泄漏。
- 仍遵守版权边界：只对齐机制结构与体验节奏，不复制竞品卡名、原文、美术、素材或 UI 构图。

QA 是门禁，不计入核心体验分。QA 失败时本轮不能合格；QA 通过时也不加分。

未达到核心体验 95 分前继续追加轮次。

## 0.1 评分口径修正

第 14 轮曾记录过 `72 / 100` 的混合分。这个分数包含了自动化证据、cleanup 和 QA 完整度，不再作为官方成绩。

从本文件本版起，官方成绩只看玩家实际玩的核心体验：

- 玩家是否能形成卡牌 roguelike 的决策压力。
- 每张牌是否有可信生命周期和构筑含义。
- 战斗、奖励、路线、下一战是否形成连续 run。
- 局内成长是否真实改变后续打法。
- UI 只按“玩家是否读得懂、点得动”计入体验；QA 脚本本身不计分。

## 3. 当前缺口

第 16 轮不能停止。`80 / 100` 到 `95 / 100` 还差 `15` 分，主要缺口如下：

| 缺口 | 影响分 | 当前问题 |
| --- | ---: | --- |
| Build plan 可理解性 | 4-6 | 玩家能选路线，但还不能清楚看出“我这条路线是在补桥/补终结/清污染/堆某张牌”。 |
| 升级、路线、奖励合流 | 4-5 | 三者已兼容，但还没有形成一套显性的构筑计划和复测闭环。 |
| 污染/失败压力联动 | 3-4 | 污染、承压、路线和失败边界仍偏分散，缺少连续 3-5 节点的压力升级。 |
| 局外成长目标 | 3-5 | 仍是安全占位，没有可复测的最小档案/解锁目标。 |
| 内容密度与玩家复测 | 4-6 | 还需要更多可反复玩的 3-5 节点 run 镜头，而不是单条路线样片。 |

## 5. 每轮通用流程

1. 派出 10 个新专家，不复用上一轮身份。
2. 本轮只选一个主系统切片。
3. 代码必须配 sim 测试。
4. 若涉及玩家可见体验，必须配浏览器 QA。
5. 浏览器 QA 后必须关闭 page/context/browser/server。
6. 更新 100 分评分。
7. 本地 git 提交本轮结果。
8. 若 `<95`，继续下一轮。
