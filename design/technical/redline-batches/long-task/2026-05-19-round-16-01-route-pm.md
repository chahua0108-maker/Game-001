## 3. 第16轮必须交付的玩家流程

### 3.1 主流程状态机

第16轮完成后，玩家应该能在一次连续游玩中看到：

```text
PlayerTurn / EnemyTurn
  -> CombatResolved 或 NodeClear
  -> RewardPending
  -> RewardChosen
  -> RoutePending
  -> RouteChosen
  -> NextBattlePrepared
  -> PlayerTurn
```

关键要求：

1. 奖励选择后不能直接进入下一战；必须进入 `RoutePending` 或等价玩家可见阶段。
2. 路线选择必须使用第15轮的候选语义：稳定/修补路线 vs 高压/收益路线。
3. 选择路线后才推进到下一节点，并准备下一战。
4. 下一战开始时，HUD 能读到刚才选了哪条路、带来了什么变化。
5. 到达最终节点时不再弹路线选择，直接进入胜利/结算路径。

如果工程上暂时不新增正式 FSM enum，也必须有等价的 `reward pending -> route pending -> next battle` 阶段字段，且玩家点击路径不能混在一个按钮里。

### 3.2 路线选择 UI

最低 UI 形态不需要地图，但必须是可操作的路线选择，而不是纯展示。

路线卡至少包含：

| 字段 | 要求 |
| --- | --- |
| 路线名 | 例如“维修补给岔路”“高压债务岔路”；不要显示 raw id。 |
| 节点推进 | 显示 `节点 1 -> 2` 或等价短句。 |
| 下一战预览 | 显示奖励倾向、压力变化或资源变化之一。 |
| 选择按钮 | 玩家明确点击后才进入下一战。 |
| 选择后反馈 | 短 toast/日志/HUD token：`已选择高压债务岔路，下一战终结奖励倾向提高`。 |

移动端红线：

- 两张路线卡可上下排列，不要求并排。
- 390px 与 360px 宽度下不得横向溢出。
- 路线预览文案最多两行；长文案省略或压成短 token。
- CTA 不能被底部区域遮挡。

### 3.3 下一战应用 route context

路线不能只是 UI 文案。至少要有一项被下一战真实读取：

| 路线 | 最小可见影响 | 不允许的假实现 |
| --- | --- | --- |
| 维修补给岔路 | 下一战 reward branch 更偏 `repair-resource`，或奖励复核/候选数体现 `rewardPickBonus: 1` | 只在 HUD 写“更偏修补”，实际候选不变。 |
| 高压债务岔路 | 下一战压力、收益或终结倾向更高；若使用 `maxEnergyThisRunPlusOne`，只在下一战开局生效并标注本次冒险 | 当前回合立刻加能量，或写成永久 MP。 |

本轮可以选择更保守的实现：

- 先只把 route context 接到下一战 reward candidate bias。
- 暂不接 `player.maxEnergy`，避免 MP 边界混乱。
- 高压路线用敌意图/奖励倾向/终结奖励概率表达，不强行做数值 buff。

只要玩家能在下一战看到候选奖励或压力确实因路线变化，本轮即可加分。

## 4. 技术验收标准

### 4.1 Runtime 接线

第16轮的 runtime 接线应满足：

1. `completeCombatRouteNode()` 与 `selectShortRunRouteNode()` 由正式 runtime 入口调用，不能只留在测试 helper。
2. route state 必须进入 world/snapshot 或等价 UI 可读结构。
3. 奖励选择后生成 pending route candidates；路线选择前不能开始下一战。
4. route chosen 后清空 pending candidates，写入 route history 和 next battle context。
5. 下一战准备阶段读取 next battle context，并在读取后保留可展示的“最近路线”信息。
6. restart run 时 route state 清空，不污染下一次 run。
7. 失败结算时不能继续弹路线选择。
8. 最终节点完成时不能继续生成下一节点候选。

### 4.2 Snapshot / HUD 合同

HUD 不应从卡名或 reward id 猜测路线状态。snapshot 至少要能表达：

- 当前节点 / 最大节点。
- 是否有 pending route candidates。
- 候选路线的玩家名、预览、from/to node。
- 最近选择的路线。
- 下一战 route context 摘要。
- route context 是否已应用到当前战。

玩家可见文案红线：

- 不显示 `rewardRerollPlusOne`、`maxEnergyThisRunPlusOne`、`repair-resource` 这类 raw id。
- 不显示“永久”“账号”“档案强化”“局外成长”等词。
- `maxEnergyThisRunPlusOne` 如被使用，文案必须是“本次冒险 Max MP +1 / 下一战生效”，不能写“授权 +1”。
- 奖励来源要避免把 `level-up` 误写成节点完成，除非 runtime 已经拆清 source。

### 4.3 Reward / Node Source 红线

本轮必须修正或绕开第15轮的关键语义风险：

```text
level-up reward != node-clear reward
```

最低验收方式：

- 只有节点清算或明确的战后奖励会进入 route pending。
- 战斗中单纯 level-up 不应自动推进 `run.currentNode`。
- 如果当前 prototype 仍把击杀 XP 奖励当作短 run 节点奖励，需要在字段/文案中说明这是 demo milestone，而不是正式 node clear。

推荐技术方向：

- `RewardState.source` 增加或使用等价字段区分 `level-up` / `node-clear`。
- route pending 只从 `node-clear` 进入。
- reward history 记录 source，方便 QA 与 PM 复核。

## 5. 核心体验验收标准：72 -> 80

达到 `80 / 100` 必须满足以下全部体验验收：

1. 玩家完成一场战斗后，进入奖励选择。
2. 玩家选择奖励后，进入路线选择，而不是自动下一战。
3. 路线选择至少有两项，且两项的预览有明确差异：修补/稳定 vs 高压/收益。
4. 玩家点击其中一条路线后，进入下一战。
5. 下一战 HUD 明确显示上一条路线的结果或影响。
6. 下一战至少有一项真实状态受路线影响：奖励倾向、奖励候选/复核、敌方压力、资源上限或等价 run context。
7. 奖励和路线的影响能被玩家用一句话复述：`我拿了 X，选了 Y，所以下一战 Z 变了`。
8. 最终节点和失败边界不退化：终点胜利、HP 归零失败仍可读。
9. 不出现 raw debug enum、永久成长误导、局外档案误导。
10. QA 门禁全部通过，但不把 gateScore 写入核心体验分。

如果只满足 1-5，建议评分为 `76-78`，不记 `80`。  
如果满足 1-7，但移动端或 raw 文案不合格，体验可以到 `80` 的机制层，但本轮不能验收，通过前必须修 UI 门禁。
