## 1. 当前循环一句话判断

当前 Redline 已经从失败的 realtime 压力切回回合制卡牌框架，但玩家实际读到的仍像“按费用顺序清前排、结束吃总伤害”的脚本化战斗，还没有形成“看敌意图 -> 用有限手牌和 MP 做取舍 -> 明确改变回合后果 -> 奖励修正牌组”的卡牌核心循环。

## 2. 五段循环诊断

| 维度 | 当前状态 | 为什么压迫感仍不像卡牌游戏 |
| --- | --- | --- |
| 回合目标 | 发牌时已有 `enemyIntentSummary.totalDamage`，开局前排总意图约 17。HUD 也显示意图和 End Turn 后果。 | 目标还是总数字，不是“这手牌要把 17 压到 5 以下”的明确任务。玩家看不到每张牌将把意图从多少降到多少。 |
| 资源压力 | `maxEnergy = 3`，正好支持 `0 + 1 + 2`；完成链后获得 `tempAuthorizationMP +3`，只能支付 3 费全场 payoff。 | 底层规则方向正确，但起手没有 payoff，奖励又不自然出现，导致授权常像一个亮起来但暂时无处消费的状态。 |
| 出牌决策 | chain 只看费用升序；起手固定 `0,1,1,2`，能稳定教学。 | 主要决策仍是排序题。两张 1 费牌还没有形成“伤害、抽牌、修补、贪 payoff”的明显分歧。 |
| 敌人反馈 | 有前排/后排、补位、意图快照、`PayoffResolved.preventedIntentDamage` 等数据。 | 击杀后立即补位会让画面压力回满；敌人本体和前排按钮没有强标注“本回合攻击 / 下轮攻击”，玩家不容易相信自己真的减少了危险。 |
| 奖励闭环 | 击杀给 XP，升级后三选一，奖励进入当前 run 的 deck/drawPile。 | 默认 `xpThreshold = 45` 对 3-5 回合样片太慢；选牌后先发下一手再加牌，反馈延迟，奖励没有回应刚才的失败或路线问题。 |

## 3. 五个最关键问题

| 优先级 | 问题 | 影响 |
| --- | --- | --- |
| P0 | 缺少“牌面级后果预览”。 | 玩家知道结束回合会扣血，但不知道打某张牌会让扣血从 `17 -> 12`、`17 -> 5` 还是仍然 `17`。没有这个，压迫感只是总伤害提示，不是卡牌解题压力。 |
| P0 | 费用链仍像升序排序题。 | 当前正确路线主要是 `0 -> 1 -> 2`，chain 不看路线类型，同费段差异不够。玩家缺少“先杀高伤 Brute、还是抽牌找 payoff、还是保留 Wild 修缺口”的判断。 |
| P0 | 奖励闭环自然不可见。 | 默认阈值太高，3-5 回合很难自然看到奖励；奖励即使出现也延迟进入下一次抽牌，无法形成“我刚才缺修补，所以现在拿修补”的即时闭环。 |
| P1 | 敌人意图和补位反馈不够可信。 | runtime 已区分本回合攻击者和补位敌人，但视觉层仍容易让玩家觉得“我清了前排，怎么前排又满了”。这会削弱清场和 payoff 救场。 |
| P2 | 发牌循环缺少后果沉淀。 | 初始牌组 4 张、手牌 4 张，第一手等于整副牌；没有状态牌、污染、保留、弃牌选择，导致“上回合没处理好影响下一手”的卡牌循环压力还没出现。 |

## 4. 可执行改动建议

1. 在 HUD 加一条“本回合目标 / 推荐下一步”：例如 `目标：把结束回合伤害 17 压到 <=5`，再显示 `先打 MP0：Debt Hook`、`接 MP1：Redline Cut`、`现在 Row Cleave 可把意图降到 X`。
2. 给每张可打牌增加简短后果预览：`杀 BRU：意图 17 -> 12`、`横扫：意图 17 -> 5`、`断链：伤害不足，仍会受 12`。移动端必须常显，不能只放 tooltip。
3. 把默认单体目标从随机前排改为可预测规则：优先本回合有 intent 且伤害最高，其次可击杀目标。HUD 同步显示“默认目标是谁”。
4. 前排敌人按钮显示两类 badge：`本回合 -5` 和 `下轮`。击杀后补上来的敌人可以被攻击，但不得混入本回合未解决伤害。
5. 把 3-5 回合 demo 的首奖阈值降到 8-12 XP，并修正阈值单调性；奖励选择后应在下一手或下一次 draw 中必定可见。
6. 保持 `clearance_order` 作为 2 费展开 / mini-payoff；3 费 `severance_burst`、`red_ledger_burst` 必须有 unarmed / armed 差异，避免 x1 也像免费清场。
7. 写一个固定 5 回合验收脚本：成功链、断链、Wild/抽牌修补、armed payoff、奖励回应。不要继续用随机 run 证明核心乐趣。
8. End Turn 后合并反馈为一条结果：`避免 12，承受 5，新前排下轮攻击`。这比逐条 combat feed 更像卡牌回合结算。

## 5. 本轮主线程优先落地的 2 个最小改动

1. **先做牌面级意图变化预览。**  
   不需要新系统，基于现有 `enemyIntents` / `enemyIntentSummary`、卡牌 damage、当前目标和前排范围，先覆盖单体与前排牌即可。目标是让玩家在出牌前读到“这张牌能不能减少本回合伤害”。

2. **把奖励闭环压进 3-5 回合。**  
   把 demo 首奖阈值调到 8-12，并保证选中的修补 / 资源 / payoff 奖励在下一手或下一次 draw 中可见。不要先扩卡池，先证明奖励能回应刚才的牌序问题。

## 6. 需要验收的现象

- 首屏不用打开 debug，玩家能读到：当前 MP、结束回合伤害、下一张期望费用、本回合目标。
- 打出正确 `0 -> 1 -> 2` 后，HUD 显示的未解决意图明显下降，至少能从 `17` 降到 `0-5` 或同等区间。
- 乱序或低收益路线仍可出牌，但结束回合伤害明显高于正确链，玩家能看见代价。
- 击杀本回合攻击者后，意图数字立即下降；新补位敌人显示为 `下轮`，不会混入本回合扣血。
- Wild / draw 修补至少在固定脚本中出现一次，并显示它补的是哪一段费用。
- armed 3 费 payoff 必须记录并显示 `preventedIntentDamage > 0`；unarmed payoff 不能比正确链更划算。
- 3-5 回合内自然出现一次奖励，奖励三选一覆盖修补 / 资源或延长 / payoff，并且选中卡能很快进入手牌循环。
- 结束回合前显示的未解决意图总和，必须等于实际 `EnemyAttacked` 扣血总和。
- 390x844 移动视口不能依赖 hover/title/debug 才能理解目标、意图变化和 End Turn 后果。
