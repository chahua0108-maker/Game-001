## 0. 总裁决

第 12 轮不选单边推进。

制作人裁决：

```text
本轮做“两者各做最小切片”，但顺序必须是：
先落 3-5 回合竞品相似度复测脚本，
再落牌区生命周期 v1 的最小代码切片。
```

如果本轮人力或时间只够做一件事，优先级是：

```text
复测脚本 > 生命周期 v1 代码 > 新卡 / 新 UI / 新大系统
```

原因很直接：第 11 轮已经通过 Wild MP3 延链，把核心战斗爽点向竞品推进了一步；但第 11 轮汇总也明确指出，当前仍缺真实 3-5 回合玩家镜头。若第 12 轮直接钻进生命周期 v1，项目会继续用内部合同证明自己，却不能回答用户现在最关心的问题：玩家是否真的觉得它更像竞品。

同时，完全不碰生命周期也不够。第 11 轮制作人和多位专家已经指出，牌区物理规则是卡牌 roguelike 的底座。第 12 轮应允许生命周期 v1 开最小刀，但它必须服务复测脚本里的镜头，而不是自行扩成抽、弃、洗、保留、消耗、状态、诅咒、升级的完整系统。

## 1. 基于第 11 轮的制作判断

第 11 轮已经达成：

- Wild MP3 延链成为真实战斗镜头，`ChainExtended` 和 HUD 短 token 让 `0 -> 1 -> 2 -> Wild -> payoff` 可被玩家读到。
- 自动化已覆盖 sim、UI、三视口、cleanup，`npm run check` 与 `npm run qa:ui` 可以作为继续迭代的回归门槛。
- 第 11 轮相似度评分仍只有 `7 / 14`，主要短板不是 Wild，而是 3-5 回合体验、牌区生命周期、局内强化和 run/meta 感知。
- 第 11 轮复测导演已经给出 15 个体验镜头，其中 P0 不是新增机制，而是验证玩家能否读压力、接链、修补、选奖励、期待清场和理解失败。
- 第 11 轮制作人原本偏向生命周期 v1，但第 11 轮实际落地的是 Wild 延链，因此第 12 轮需要重新校准：不能再先写大底座，再事后找玩家镜头。

第 12 轮的制作目标应从“再多做一个系统”改成：

```text
用复测脚本证明当前核心循环哪里像竞品、哪里不像；
用生命周期 v1 最小切片补一个会被复测看到的结构短板。
```

## 2. 第 12 轮必须守住的裁决边界

允许：

- 新增或整理 3-5 回合复测脚本，覆盖成功、坏手、奖励、爆发、失败反例。
- 在脚本结果里输出可读 JSON 或 Markdown 摘要，标记 `像卡牌解局 / 像教程脚本 / 看不懂`。
- 生命周期 v1 只做最小可测切片：`CardMoved` 或等价事件、`exhaustPile`、一个保留规则、一个物理状态牌规则。
- 生命周期切片必须有 sim 测试，且不破坏 Wild、payoff 授权、reward、paper topdeck、HUD 短 token。
- UI 只补必要短 token，不做新面板。

禁止：

- 不做完整 `CardInstanceId` 迁移。
- 不做完整声明式效果解释器。
- 不做遗物、升级、删牌、商店、地图、局外成长。
- 不做完整诅咒/状态生态。
- 不新增大批内容牌。
- 不扩 Wild 到无限延链。
- 不把复测脚本变成只会跑成功路线的演示脚本。
- 不把生命周期 v1 做成看不到玩家镜头的纯内部重构。
- 不复制竞品卡名、规则原文、美术、UI 构图或素材。

## 3. 裁决镜头

| # | 裁决镜头 | 本轮裁决 | 必须验收的玩家/工程信号 |
| ---: | --- | --- | --- |
| 1 | 开局读压 | 先由复测脚本锁住，而不是先写生命周期。 | 玩家或脚本能在出牌前记录 HP、MP、End Turn 伤害、敌意图。 |
| 2 | 快速升序 | 复测脚本必须保留 `0 -> 1 -> 2` 成功线。 | `effectMultiplier` 或 HUD token 呈现连续升序，不只是按钮可点。 |
| 3 | Wild 延链 | 第 12 轮不继续扩 Wild，只回归第 11 轮成果。 | `wild_gap_key` 的 MP3 延链仍可见，且不误记为 `ChainRepaired`。 |
| 4 | 坏手修补 | 复测脚本必须包含一次坏手或缺口。 | 玩家能看到修补后压力仍存在，例如 `仍-X` 或等价结果。 |
| 5 | 抽牌未命中 | 必须有失败反例，不准只有清场成功线。 | 抽牌未找到关键牌时，脚本仍记录剩余意图和下一步风险。 |
| 6 | 奖励响应 | 奖励镜头优先于新卡扩张。 | 奖励选择能被归因到上一手问题：缺入口、缺桥、缺修补或缺终结。 |
| 7 | 爆发清场 | payoff 仍是核心爽点，不新增更大招式。 | 清场必须被记录为前面链路铺垫的结果，而不是系统突然送演出。 |
| 8 | 失败结算 | 失败是 P0 验收镜头，不是测试失败。 | End Turn 扣血与预览一致，玩家能理解为什么掉血。 |
| 9 | 消耗牌 | 生命周期 v1 只开最小消耗规则。 | 一张消耗牌打出后进入 `exhaustPile`，不进入 discard，不洗回。 |
| 10 | 保留牌 | 保留只做一个可测规则，不做完整保留生态。 | 回合末保留牌下回合先进入手牌，再补抽到手牌上限。 |
| 11 | 状态牌 | 状态牌只做一张物理污染牌。 | 状态牌能进入牌区、抽到占手牌位，并按 v1 规则弃置或消耗。 |
| 12 | 牌区移动事件 | 生命周期 v1 的核心是事件化，不是 UI 堆字。 | `CardMoved` 或等价事件至少包含 `cardId / from / to / reason`。 |
| 13 | 移动端读牌 | 新 token 必须短，不准把长规则塞进按钮。 | 360/390 视口无水平溢出、无关键文本超框。 |
| 14 | 回归门槛 | 本轮不能牺牲第 11 轮成果。 | Wild、paper topdeck、payoff 授权、reward add-card 全部回归通过。 |

## 4. 本轮必须做

P0 必须做：

1. 新增 3-5 回合复测脚本或等价可重复流程。
2. 复测脚本至少覆盖：成功升序、坏手修补、奖励响应、爆发清场、失败反例。
3. 复测输出必须记录：回合数、HP、MP、敌意图、手牌、费用链、授权状态、奖励选择、失败信号。
4. 复测结果必须有明确边界字段：`similarityScope=mechanic-slice-only`、`notAFullClone=true`。
5. 生命周期 v1 只做最小可测代码切片：消耗堆、保留、物理状态牌、牌区移动事件四选三也可以，但必须至少包含牌区移动事件和消耗堆。
6. 新增生命周期 sim 测试，证明普通弃牌、消耗、保留或状态不会破坏洗回与奖励路径。
7. 保留第 11 轮 Wild MP3 延链测试与 HUD 短 token 回归。
8. 保留 `npm run check` 和 `npm run qa:ui` 作为最终门槛。

P1 可以做：

- 把复测结果落到 `outputs/browser-qa/round-12/` 或等价目录。
- 给 `qa:ui` 增加 round-12 scenario 参数。
- 给生命周期 token 增加短文案，例如 `保留`、`消耗`、`状态`、`洗回`。

P2 暂缓：

- 截图基线。
- 半自动玩家口述采集。
- 更多奖励变体。
- 生命周期可视化面板。

## 5. 本轮不能做

本轮明确不能做：

- 不能把第 12 轮变成完整生命周期大迁移。
- 不能在没有复测脚本的情况下先做大代码切片。
- 不能新增一批新卡来掩盖底层生命周期不足。
- 不能新增遗物、宝石、升级、删牌、商店或地图。
- 不能把 `paper_shatter` 扩成通用 tutor。
- 不能让状态牌或保留牌需要全新大 UI 才能理解。
- 不能让 `exhaustPile` 参与洗回。
- 不能让保留牌绕过手牌上限而无限堆积。
- 不能让状态牌只存在于 debug，不进入真实牌区。
- 不能把自动化通过写成“已经完整复刻竞品”。
