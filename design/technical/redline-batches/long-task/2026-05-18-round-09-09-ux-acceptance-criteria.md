## 0. 一句话准入

UI 验收通过不是“没有报错、没有超框”就够了。玩家必须在 3-5 回合样片中持续读到这件事：

```text
我正在用卡牌链路处理敌人意图；修补和整备只是帮我找下一步，只有击杀、打断、授权终结或真实降意图才算解除压迫。
```

如果玩家在移动端只看到“抽牌”“整备”“奖励”，却读不到当前仍会承受多少伤害，判定为 UX 未通过。

## 1. 验收场景矩阵

本轮 UX 验收必须覆盖 5 个固定场景。每个场景都需要桌面和移动端证据；移动端以 `360x640` 为硬门槛，`390x844` 只作为主流确认。

| 场景 | 必须证明 | 不可接受 |
| --- | --- | --- |
| 初始高压回合 | 玩家看到 HP/MP、费用链、前排意图、End Turn 后果，并能理解本回合目标。 | 敌意图只在某个小角落出现，手牌区读不到当前压力。 |
| 修补/抽牌牌出手 | 打出 `blood_tithe`、`pulse_draw`、`wild_mana_stitch` 或同类牌后，UI 同时反馈抽牌/修补收益和 `仍-X`。 | 文案只显示 `抽N找解`、`修补成功`，让玩家误以为压力已解除。 |
| 授权终结窗口 | 完成费用链后，玩家看得出 `授权+3`、`授权付`、终结牌可打，以及终结后意图下降或清场。 | 授权文案长到被省略，或终结牌只是从 disabled 变 enabled 但没有解释支付来源。 |
| `paper_shatter` 整备 | `Paper Route` / `paper_shatter` 打出后，玩家可见 `整备`、`顶终结`/`下抽终结`/`整备无牌` 之一，并能看到它如何影响下一抽或下一步路线。 | 只有 debug trace 证明置顶，移动端没有玩家可见结果。 |
| 奖励选择后下一轮 | 奖励三选一能让玩家判断“补路线、拿终结、拿修补”三类取舍，选择后下一轮能看到奖励进入牌组或抽牌循环。 | 奖励卡长文案被截断，玩家不知道选项解决什么问题。 |

## 2. 核心压迫读数验收

### 2.1 必须同屏可读

玩家做出牌决策时，以下信息必须在当前视野中可见或无需 hover 即可触达：

| 信息 | 通过标准 |
| --- | --- |
| 当前 HP | 不被手牌 rail、奖励面板或安全区遮挡。 |
| 当前 MP / 可支付状态 | 手牌按钮上能读到 `MP`、`缺MP1`、`授权付` 等状态。 |
| 费用链状态 | 能读到 `0>1>2`、`起链`、`接x2`、`断x1` 或等价短 token。 |
| 敌意图总压 | 至少一处稳定显示 `-X HP` 或 `回合损X`。 |
| 单张牌对压力的影响 | 攻击牌显示 `N->M`；无伤害修补/抽牌牌显示 `抽N仍-X`、`修补仍-X` 或 `不降意图`。 |
| End Turn 后果 | 按钮或相邻提示必须显示结束回合将承受的后果。 |

### 2.2 压迫读数失败条件

以下任一条出现，UX 验收失败：

- 修补牌打出后，当前敌意图仍未下降，但卡牌、Director、日志均没有 `仍-X` 或 `不降意图`。
- End Turn 可点击时，玩家无法在按钮附近判断会损失多少 HP。
- 敌人前排有多个意图来源，但 UI 只给抽象总数，完全看不出哪些前排正在压迫。
- 高压局面下，奖励或整备反馈覆盖了压迫读数超过一次决策时间。
- 移动端玩家必须横向滚动到某张牌之外，才能知道本回合会受到多少伤害。

## 3. 卡牌短 Token 验收

### 3.1 玩家可见词表

固定 UI 表面只允许短 token 承担主信息。完整规则进入 title、detail、图鉴或 debug。

| 概念 | 通过 token | 失败文案 |
| --- | --- | --- |
| 授权获得 | `授权+3` | `终局授权 +3` 出现在手机状态位。 |
| 授权支付 | `授权付` | `终局授权支付`。 |
| 缺授权 | `缺授权` | `需MP/终局授权`。 |
| 整备角色 | `整备` | `整备/找牌`、`重排牌库`。 |
| 整备结果 | `顶终结`、`下抽终结`、`整备无牌` | `已将某张牌放到抽牌堆顶部`。 |
| 修补 | `修补`、`修补MP1` | `修补/抽牌/返MP`。 |
| 抽牌承压 | `抽3仍-17` | `抽牌找解` 单独出现。 |
| 终结 | `终结` | `payoff` 玩家可见。 |
| 目标 | `前排`、`全场`、`默认BRU` | `默认最高意图 Redline Brute`。 |

### 3.2 字段长度准入

以 `360x640` 为硬准入：

| UI 字段 | 建议上限 | 必须保留的信息 |
| --- | ---: | --- |
| 卡名 | 中文 8-12 字 / 英文 14 字符 | 能识别牌。空间不足时使用短名。 |
| 角色 token | 2-4 字 | `开链`、`承接`、`整备`、`修补`、`终结`。 |
| 动态压力预览 | 5-9 字 | `抽3仍-X`、`14->0`、`不降意图`。 |
| 支付状态 | 3-6 字 | `缺MP1`、`缺授权`、`授权付`。 |
| 奖励规则首句 | 24-32 中文字内 | 动作、对象、数值必须在截断前出现。 |
| Director 结果 | 4-8 字 | `下抽终结`、`回合损12`、`整备无牌`。 |

失败标准不是“出现省略号”本身，而是省略号吞掉了唯一的决策信息。若 `rulesText` 被截断但 `mobileEffect` 和动态预览已完整表达收益与压力，可以通过。

## 4. 授权与整备反馈验收

### 4.1 授权窗口

授权体验必须形成清楚的因果链：

```text
完成链路 -> 授权+3 -> 终结牌显示授权付 -> 打出终结 -> 压力下降/清场
```

通过标准：

- 授权获得发生时，状态条或日志出现 `授权+3`。
- 终结牌因授权可支付时，卡牌按钮显示 `授权付`，不能只移除 disabled。
- 授权不足时显示 `缺授权` 或 `缺MP1`，不能显示混合 slash 长句。
- 打出授权终结后，玩家能看到意图 `N->M`、全场伤害、清场或奖励触发的结果。
- 本回合授权失效或消耗时，有可解释反馈；玩家不能误以为这是永久资源。

失败条件：

- 授权信息只在战斗日志里，移动端手牌按钮没有支付来源。
- `授权付` 与 `缺MP` 同时出现导致状态冲突。
- 终结牌可打但没有任何文本说明为什么突然可支付。

### 4.2 整备反馈

`paper_shatter` / `Paper Route` 的整备验收以玩家可见因果为准：

```text
打出 Paper -> 整备结果 -> 当前或下一次抽牌验证
```

通过标准：

- 卡牌角色显示 `整备`。
- 打出后至少出现一个结果 token：`顶终结`、`下抽终结`、`入手终结`、`整备无牌`、`终结在手`。
- 若命中终结，后续抽牌或下一手路线能验证结果；若未命中，必须显示 `整备无牌`。
- 移动端结果不能只依赖 `.combat-feed`，因为该区域可能隐藏；Director、状态提示或手牌可见反馈必须承担结果。
- 整备反馈不得承诺不存在的操作，例如 `选择`、`查看牌库`、`重排牌库`，除非真实 UI 已经支持。

失败条件：

- 运行时确实置顶了，但玩家只看到 `抽3`，不知道整备是否生效。
- 置顶的是具体牌名长句，导致移动端溢出或截断。
- 未命中时没有反馈，玩家以为点击失效。

## 5. 奖励面板验收

奖励面板的 UX 目标是让玩家判断“这张奖励解决我刚才哪个问题”，不是展示完整规则书。

### 5.1 奖励三选一必须读出分支

每次奖励至少要能读出以下三类中的明确定位：

| 分支 | 可见表达 | 玩家理解 |
| --- | --- | --- |
| 修补 | `修补`、`抽1`、`MP+1`、`仍-X` 相关 detail | 帮我处理坏手或费用缺口。 |
| 路线/整备 | `整备`、`抽1/3`、`顶终结` | 帮我更稳定找到下一步或终结。 |
| 终结/payoff | `终结`、`全场16`、`授权付` | 完整链路后的收束奖励。 |

如果三张奖励全都看起来像“抽牌找解”，验收失败，即使数值不同。

### 5.2 移动端面板准入

在 `360x640`、`390x844`、`430x932` 均需满足：

- 页面无横向滚动。
- `.reward-panel` 左右不越界，建议至少 12px 安全边距。
- 三张奖励牌都可通过面板内部滚动触达，最后一张底部完整可点。
- 每张奖励牌触控高度不低于 44px。
- 标题、角色、摘要、规则行不互相覆盖。
- `small` 或 `em` 被截断时，核心收益已在另一处可见。
- 奖励选择单击只触发一次，不穿透到底层手牌或 End Turn。

### 5.3 奖励文案失败条件

- `Paper Route` 奖励卡没有显示 `整备`、`抽1/3` 或 `顶终结` 任一核心收益。
- `rulesText` 第一行只写背景解释，动作和数值在第二行后被截断。
- 长英文、斜杠串或连续符号撑破卡片。
- 选中后面板关闭了，但玩家看不到奖励进入牌组、手牌或后续抽牌循环的证据。

## 6. 移动端手牌 Rail 验收

移动端手牌 rail 是主要决策面，必须比桌面更严格。

### 6.1 布局准入

在 `360x640` 和 `390x844`：

- `.card-row` 只允许横向滚动，不允许因为卡牌数量变化把页面撑高。
- 单张卡高度稳定，出牌、抽牌、授权状态变化时 rail 高度不跳动。
- End Turn 按钮不被 rail 或系统安全区遮挡。
- 横向滚动时不能带出页面级横向滚动。
- 每张卡的点击区域不低于 44px 高，且相邻卡之间不会误触。
- 出牌后手牌重排不能让下一次 tap 落到错误卡上；连续点击必须按玩家意图执行。

### 6.2 手牌信息准入

每张可决策卡至少保留：

```text
费用 / 短名 / 角色或效果 / 动态压力或支付状态
```

对于关键牌：

| 牌型 | 移动端必须读到 |
| --- | --- |
| 0 费开链 | `0`、`开链` 或等价起手提示。 |
| 1 费承接 | `1`、`接x2` 或等价续链提示。 |
| 2 费展开 | `前排`、`N->M` 或授权触发线索。 |
| 修补牌 | `修补` / `抽N` 与 `仍-X`。 |
| 整备牌 | `整备` 与 `抽1/3`；结果在打出后可见。 |
| 终结牌 | `终结`、`全场`、`授权付` 或 `缺授权`。 |

失败条件：

- 移动端只显示卡名和费用，核心效果需要 hover 或桌面日志。
- `.card-effect` 被隐藏后，`抽N`、`整备`、`授权付` 等唯一信息也随之消失。
- `抽3仍-17` 被裁成只剩 `抽3`，导致压力信息丢失。
- 手指横扫 rail 时，页面整体左右晃动或出现横向滚动条。

## 7. 验收证据格式

每次 UX 验收建议保留一份短报告，至少包含：

```json
{
  "viewport": "360x640",
  "pressureReadableInHandFocus": true,
  "repairShowsRemainingIntent": true,
  "authorizationCauseReadable": true,
  "preparationResultVisible": true,
  "rewardCoreBenefitReadable": true,
  "handRailStable": true,
  "horizontalOverflowDetected": false,
  "tapMisfireCount": 0,
  "blockingUxFailures": []
}
```

截图最少覆盖：

- 初始战斗高压状态。
- 修补/抽牌牌 hover 或移动端可见状态。
- 授权获得与授权支付终结。
- `Paper Route` 奖励卡。
- `Paper Route` 打出后的整备结果。
- `360x640` End Turn 可见状态。

## 8. 最终通过判定

### 必须全部通过

- 玩家在手牌决策焦点中能读到当前压迫，而不是只在战场或日志读到。
- 所有固定 UI 表面使用短 token；完整规则不挤占移动端主决策面。
- 授权和整备都有清楚的 cause/effect 反馈。
- 奖励面板能表达分支取舍，且移动端可完整触达。
- 手牌 rail 稳定、可横扫、可单击，不遮挡 End Turn，不制造页面横向溢出。

### 可以后续优化但不阻断

- 桌面 tooltip 文案不够优雅。
- combat feed 只有桌面可见，只要移动端另有结果反馈。
- 奖励牌 detail 还不完整，只要首屏短文案足够决策。
- 动画节奏偏硬，只要状态变化因果清楚。

### 必须阻断

- 玩家无法判断结束回合会损失多少 HP。
- 修补/抽牌牌让玩家误读为已经解除敌意图。
- `paper_shatter` 的整备结果只在 debug 或隐藏日志中存在。
- `360x640` 出现页面横向滚动、关键按钮遮挡、奖励不可触达或单击误触。
- 授权终结可支付但没有玩家可见支付来源。
