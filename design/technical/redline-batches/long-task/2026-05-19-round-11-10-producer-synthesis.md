## 1. “越像竞品越成功”的制作解释

这里的“像竞品”不能理解成复制竞品卡名、规则原文、美术、UI 形状、数值模板或关键词命名。

正确解释是：

```text
复刻成熟卡牌 roguelike 的结构可信度：
玩家相信这是一套真实卡牌战斗系统，而不是一个用卡牌 UI 演出来的脚本 demo。
```

所以第 11 轮的成功标准不是“多做几张像 Slay the Spire / Monster Train / Wildfrost 的牌”，而是让玩家和开发者看到这些底层事实：

- 牌真的在牌区之间移动。
- 打出、抽牌、弃牌、洗回、消耗、保留都有可追踪事件。
- 状态牌是物理卡，会进入牌堆、占手牌、被清理或被消耗。
- 回合末不是把所有手牌一刀切丢掉，而能表达保留、消耗和状态差异。
- 后续升级、诅咒、遗物、触发器、删牌、商店、路线节点有地方挂。

这才是“越像竞品越成功”的底座版本。

## 2. 当前 10 轮结果的制作复盘

### 镜头 01：P0 demo 已经成立

当前已成立的样片是：读敌人意图、用 0 -> 1 -> 2 费用链铺垫、获得本回合 payoff-only 授权、打 3 费终结、拿奖励进入下一手，并用 `paper_shatter` 验证极窄整备顶终结。

制作裁决：这个 demo 不应被推倒；第 11 轮必须在不破坏它的前提下补底座。

### 镜头 02：第 10 轮“停手”不是永久停手

第 10 轮停手是为了防止收敛轮继续膨胀，不是说项目不再写代码。现在用户给出新目标，第 11 轮可以重新开代码，但必须换成竞品结构相似度导向。

制作裁决：允许继续代码，但不允许继续样片堆料。

### 镜头 03：当前最像竞品的是资源压力

`energy / maxEnergy`、每回合发牌恢复 MP、payoff-only 临时授权、敌人意图和结束伤害，已经具备成熟卡牌战斗的压力读法。

制作裁决：资源层先不要大改；不要做永久 Max MP、不要做局外属性成长、不要把授权改成泛用能量。

### 镜头 04：当前最不像竞品的是牌区生命周期

当前虽然有 `deck / hand / drawPile / discardPile`，但消耗、保留、状态牌、物理污染、弃牌原因、洗回事件还不完整。成熟竞品的卡牌系统首先要让“卡在哪里、为什么移动、下回合还在不在”可信。

制作裁决：第 11 轮最小切片应选牌区生命周期，不选新敌人、新奖励、新大招。

### 镜头 05：`paper_shatter` 已经完成它该完成的事

`paper_shatter` 的价值是证明“整备置顶”这个同类机制结构可以在 Redline 语义下成立：只搜 `drawPile`，只找 payoff，命中后置顶，再由抽牌拿进手牌。

制作裁决：不要把第 11 轮变成 `paper_shatter` 二期。它已经足够像竞品里的受限 topdeck 结构，下一步该补更基础的生命周期。

### 镜头 06：`lantern_captain` 诱惑很大，但现在不选

`lantern_captain route-only 置顶` 是合理 P1，但如果现在做，项目会继续沿着“整备花样”扩，而不是补竞品底座。它会增加可玩镜头，却不解决“消耗 / 保留 / 状态牌还不存在”的结构短板。

制作裁决：第 11 轮不做 `lantern_captain` 真实找牌。

### 镜头 07：Wild 修补已进入可验收状态

第 6 轮把 `wild_mana_stitch` 改成真实修补成功才返当前 MP，这已经把 Wild 从全能福利拉回“有条件路线保险”。

制作裁决：第 11 轮不要继续调 Wild 数值；只保留回归测试，防止牌区生命周期改动破坏它。

### 镜头 08：奖励进入下一手是正确竞品结构

奖励牌进入当前 run 并影响下一手，比“选了奖励但下一局才看到”更像成熟卡牌 roguelike 的短反馈。`rewardBranches` 显式化也是正确合同。

制作裁决：第 11 轮不要扩奖励池；只保证新增生命周期不会破坏 reward add-card 到牌区的路径。

### 镜头 09：UI 短 token 是硬护栏

当前 `授权+3`、`缺MP1`、`授权付`、`抽N仍-X`、`整备：顶终结` 已经保护了移动端读牌。成熟竞品不是靠长说明压进按钮，而是靠短状态、详情层和反馈节奏。

制作裁决：第 11 轮如果新增状态 / 保留 / 消耗反馈，也必须走短 token，不把长规则塞回卡牌按钮。

### 镜头 10：QA 链路足够做回归门槛

第 9-10 轮已经有 `npm run check` 和 `npm run qa:ui`，能覆盖机制测试、build、三视口 UI、顶终结证据和 cleanup。

制作裁决：第 11 轮不做 QA 平台，但必须新增生命周期单测，并保留现有最终验收命令。

### 镜头 11：竞品相似度不是内容量，而是系统可信度

多 10 张牌不会自动像竞品；有消耗、保留、状态牌、触发时机、牌区移动证据，才会像竞品。当前项目最大风险是把“像”理解成卡牌文案和效果花样。

制作裁决：第 11 轮禁止新增大批内容牌。

### 镜头 12：下一轮必须从底层往上，不从表现往下

如果先做 UI 面板、路线地图、商店、遗物、局外成长，会产生大量展示资产和状态分支，但底层牌区仍不可信。成熟竞品的第一性不是地图，而是手里的卡、牌堆里的卡、弃牌堆里的卡、被消耗的卡都是真的。

制作裁决：第 11 轮代码切片必须先让牌区生命周期可测，再考虑更大的 roguelike 层。

## 4. 应该选哪个最小切片

第 11 轮唯一推荐最小切片：

```text
牌区生命周期 v1：
统一牌区移动事件 + 消耗堆 + 保留到下一手 + 1 张物理状态牌。
```

### 4.1 切片目标

让当前 runtime 能表达成熟卡牌游戏最基础的三类牌：

| 类型 | 竞品结构 | Redline v1 表达 |
| --- | --- | --- |
| 普通牌 | 打出后进弃牌堆，回合末未打出也进弃牌堆 | 保持现状，但补结构化移动事件和原因 |
| 消耗牌 | 打出后进入消耗堆，不再洗回 | 新增 `exhaustPile` 或等价区 |
| 保留牌 | 回合末不弃，下回合先进入手牌，占手牌位置 | 新增 retained 逻辑 |
| 状态牌 | 被加入牌堆，抽到占手牌，按生命周期弃置或消耗 | 新增 1 张测试状态牌，不做复杂 debuff |

### 4.2 最小实现边界

允许做：

- 新增或整理牌区移动 helper。
- 发出 `CardMoved` 或等价事件，至少包含 `cardId / from / to / reason`。
- 给打出牌支持 `onPlay: discard | exhaust` 的最小生命周期。
- 增加 `exhaustPile`。
- 支持 1 张保留牌：回合末留到下一手，发牌时先带入，再补抽到手牌上限。
- 支持 1 张物理状态牌：能加入 draw/discard，抽到占手，打出或回合末按合同清理。
- 新增 sim 单测覆盖普通弃牌、消耗、保留、状态牌、奖励加卡不退化。
- 保留现有 `paper_shatter`、Wild、授权、reward、UI QA 回归。

禁止做：

- 不做完整 `CardInstanceId` 迁移。
- 不做升级系统。
- 不做遗物。
- 不做诅咒生态。
- 不做净化入口。
- 不做商店、地图、路线节点。
- 不做完整触发器系统。
- 不做完整声明式效果解释器。
- 不新增大量真实内容牌。

### 4.3 为什么不是 `lantern_captain route-only`

`lantern_captain route-only` 是一个好切片，但它仍然属于整备家族。当前项目已经有 `paper_shatter` 证明了受限 topdeck，继续做第二张会让“整备”更完整，却不会让系统更像成熟竞品的底层。

制作排序：

```text
先牌区生命周期 v1。
再 lantern_captain route-only。
再考虑 discard search / retain / status 的组合玩法。
```

### 4.4 为什么不是完整 reorder / scry

完整 reorder / scry 需要预视 UI、排序、置底、取消、移动端输入状态和大量边界测试。它很像竞品，但不是当前最低风险的相似度提升。

当前最低风险提升是：

```text
让牌真的能被消耗、保留、污染和移动，而不是先做一个更复杂的预视面板。
```

## 5. 不要碰哪些大系统

第 11 轮明确冻结以下系统：

| 系统 | 裁决 | 原因 |
| --- | --- | --- |
| 完整 `CardInstanceId` | 不碰 | 会引发全 runtime、UI、测试迁移；放到生命周期 v1 稳定之后 |
| 完整效果解释器 | 不碰 | 可以先留文档；第 11 轮只需生命周期，不需要迁移所有旧牌 |
| 完整触发器 / 遗物 | 不碰 | 没有生命周期事件前，触发器会挂在不稳的底座上 |

| discard search / 全牌库 tutor | 不碰 | 强度和 UI 成本过高，且会绕过抽牌压力 |
| 手动 scry / reorder UI | 不碰 | 移动端复杂度高，当前不是最小刀 |
| 局外成长 / 永久 Max MP | 不碰 | 会污染当前 run/meta 边界 |
| 商店 / 地图 / 路线节点 | 不碰 | 这些是 run 包装，不是当前最缺底座 |
| 升级 / 删牌 / 净化 | 不碰 | 需要实例或更完整奖励类型，放后续 |
| 美术 / 动画 / 音效 / 公开试玩包装 | 不碰 | 第 11 轮目标是机制结构相似度，不是表现层 |
| QA 平台 / 截图基线 / CI | 不碰 | 现有 `qa:ui` 足够做本轮门槛 |

## 6. 第 11 轮验收标准

### 6.1 机制验收

牌区生命周期 v1 必须满足：

- 普通牌打出后进入 discard，事件记录 `played -> discard`。
- 回合末未打出的普通牌进入 discard，事件记录 `end-turn -> discard`。
- 消耗牌打出后进入 exhaust，不进入 discard，不参与后续洗回。
- 保留牌回合末留存，下回合先进入手牌，再补抽到手牌上限。
- 状态牌可以被加入牌区，抽到后占手牌位。
- 状态牌的清理规则明确：v1 只允许一种简单规则，例如打出后消耗，或回合末弃置。
- 抽牌堆空时回填逻辑不把 exhaust 牌洗回。
- 奖励加卡进入当前 run 的路径不退化。
- `paper_shatter` 顶终结不退化。
- Wild 修补成功返 MP 不退化。
- payoff-only 授权不退化。

### 6.2 UI 验收

第 11 轮不要求做完整新 UI，但不能破坏现有 UI：

- 三视口仍无 console error。
- 三视口仍无水平溢出。
- 三视口关键文本仍无超框。
- 状态 / 保留 / 消耗如有可见反馈，只用短 token。
- 不把长生命周期说明塞进手牌主按钮。

### 6.3 文档验收

若第 11 轮代码落地，必须在同目录新增短验收记录，说明：
