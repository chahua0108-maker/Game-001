## 1. 移动端核心交互路径

### 1.1 竖屏主路径

当前 demo 的移动竖屏核心路径应该是：

1. 进入战斗，系统从 `Deal` 自动进入 `PlayerTurn`，发 4 张起手牌。
2. 玩家先读顶部资源：HP、MP、终局授权、CHAIN、敌人意图。
3. 玩家读 `combat-director`：本回合链路、下一张期望费用、敌人意图、终结预览。
4. 玩家在底部手牌区按费用顺序点牌：优先 MP0，再 MP1，再 MP2，最后看 payoff / 终局授权。
5. 若要指定单体目标，先点右侧 `前排显影`，再点前排敌人槽，最后点单体牌。
6. 无牌可打或想承受后果时，点 `deal-panel` 里的结束回合。
7. 升级时出现奖励面板，三选一后回到下一轮。

这条路径的核心问题是视线跳跃太长：顶部读链路和意图，底部点牌，中上部点结束回合，右上角展开目标。竖屏上玩家不是围绕一个拇指区连续操作，而是在 3 个区域之间来回确认。

### 1.2 横屏主路径

横屏手机高度通常只有 360-430px。当前 CSS 主要按宽度断点处理，`max-width: 980px` 的布局会在横屏手机上仍保留：

- 多格 `status-strip`；
- 2x2 的 `combat-director`；
- `deal-panel`；
- `target-panel`；
- 底部多张手牌。

这会让横屏比竖屏更危险：纵向高度不够，顶部信息和底部手牌会一起挤压战场，甚至发生视觉重叠。横屏需要单独策略：顶部只保留一行关键资源，底部手牌横向滚动或单行压缩，结束回合放到右手拇指区。

### 1.3 当前局外层 / 奖励入口路径

当前 `run-layer-panel` 显示本次清算、节点、最近奖励、局外档案占位。它是被动信息，不是实际入口。奖励层通过 `Reward` flow 自动出现三选一面板，局外层目前没有可点击入口。

移动端设计上这是正确的方向：局外层不应该在战斗中抢主操作区。但当前 `run-layer-panel` 在小屏以底部固定位置显示，容易和手牌区争空间，建议移动端默认只保留一个很薄的“节点/奖励”状态，不展示“局外档案”占位。

## 2. 最可能挡住玩家判断的 UI 区域

### 2.1 顶部信息堆栈

当前移动端顶部会堆叠 `status-strip`、`combat-director`、`deal-panel`、`target-panel`、`enemy-peek`。这些信息各自有价值，但一起展开时会形成“上半屏控制台”。

风险：

- 玩家先看到很多正确指标，但下一步行动不够唯一。
- 敌人区和战场主体被上方 DOM 压缩，尤其横屏。
- `combat-director` 已经包含链路、意图、终结预览，顶部 `CHAIN / 意图` chip 再重复一次，信息密度偏高。

判断：移动端第一优先级不是继续增加说明，而是压缩成一条“下一步行动线”。

### 2.2 手牌区与局外层重叠

`HAND_SIZE` 当前是 4，起手通常是两列两行，尚可接受。但移动端存在动态手牌风险：

- 抽牌或奖励后可能临时超过 4 张；
- `card-row` 在小屏是两列网格，会继续向上增长；
- `run-layer-panel` 在小屏用 `bottom: 178px` 固定，动态手牌超过两行时会与手牌区发生视觉重叠。

判断：手牌区必须有最大高度和滚动 / 抽屉策略，不能让卡牌数量决定 HUD 向上侵占多少战场。

### 2.3 结束回合按钮

结束回合在 `deal-panel` 内，移动小屏高度约 32px，且位置在上半屏。对 Hyper-Turn 连续点牌来说，玩家从底部手牌点完后，需要把手指移到中上部结束回合，节奏断开。

当前按钮文案已经能显示 `结束回合 -X`，这是好方向；问题是触控位置和尺寸不适合手机主路径。

判断：结束回合应该成为靠近手牌的固定次级动作，至少在移动端满足 44px 高度。

### 2.4 目标选择区

目标选择入口是 `前排显影` toggle，移动端高度约 30px；展开后 `enemy-slot` 在 640px 以下最小高度约 28px。这个尺寸低于手机触控舒适区。

风险：

- 精准点某个前排敌人时误触概率高。
- `target-panel` 只是提示，不可点击；玩家可能误以为目标已自动处理，不知道要展开前排。
- 展开目标条后会进一步占据战场上半区。

判断：如果单体牌允许随机目标，可以默认弱化目标选择；如果精准击杀是核心，目标槽必须做成 44px 以上的横向目标条。

### 2.5 卡牌文本

小屏下 `.card-effect` 被隐藏，卡牌只剩卡名、角色/目标、链路预览和少量 payoff 文案。MP 成本很清楚，但伤害、抽牌、效果比较会变弱。

风险：

- 两张同费用单体牌的差异不明显。
- `非起手 x1` / `断链 x1` 可能被误读成不能打。
- 长中文卡名和长效果会被 `nowrap + ellipsis` 截断；移动端没有 hover，title 不能承担核心说明。

判断：移动端卡牌可以短，但不能隐藏“费用、是否推荐、伤害/抽牌、目标”四个决策字段。

### 2.6 奖励面板

奖励面板在移动端改为单列三张卡，卡牌最小高度约 86px。这比战斗目标按钮更适合触控。但面板没有明确的 `max-height + overflow-y` 策略，若奖励描述变长、系统字体变大或横屏高度不足，容易压住底部区域或溢出。

判断：奖励层应作为模态底部 sheet 或居中 modal 均可，但必须可滚动，并且不要让背景战斗 HUD 同时争夺注意力。

## 3. 触控目标尺寸 / 卡牌文本 / 滚动策略

### 3.1 触控目标尺寸判断

建议手机主操作目标不低于 44px 高；次级但可点击目标不低于 40px。

当前判断：

| 区域 | 当前代码趋势 | 判断 |
| --- | --- | --- |
| 手牌按钮 | 82-88px 高 | 合格，是当前最稳定的触控面 |
| 奖励卡 | 移动单列，约 86px 起 | 基本合格 |
| Game Over 重开 | 44px | 合格 |
| 结束回合 | 小屏约 32px | 不合格，应升到 44px |
| 前排显影 toggle | 约 30px | 不合格 |
| 前排敌人槽 | 小屏约 28px | 不合格，精准目标选择不可依赖 |
| 顶部状态 chip | 32px 左右 | 非主点击目标，可接受，但信息密度高 |

### 3.2 卡牌文本策略

移动端卡牌应固定 4 个层级：

1. 最大：MP 成本和推荐状态。
2. 第二：卡名或短别名。
3. 第三：目标和角色，例如 `开链 / 单体`、`承接 / 单体`、`展开 / 前排`、`终结 / 全场`。
4. 第四：一个数字效果，例如 `9伤害`、`前排5`、`抽1`、`终结16`。

不建议把核心效果只放在 title、aria-label 或 hover tooltip。移动端看不到 hover，aria 也不能替代可见决策信息。

### 3.3 手牌滚动策略

当前两列网格适合 4 张牌，不适合动态 5-6 张以上。建议：

- 竖屏：底部手牌最多显示两行；超过 4 张后横向分页或横向滚动，不继续向上增长。
- 横屏：手牌默认单行横向滚动，卡高压缩，但 MP 和推荐状态不能丢。
- 出牌后保留本回合槽位占位，至少短暂保留 200-300ms，避免快速连点时剩余卡牌重排导致误触。
- 若做“长按查看详情”，只能作为补充；主路径仍应单 tap 可决策。

### 3.4 敌人区滚动策略

移动端不应展示完整 5x3 槽位表。当前只显示前排显影是正确收敛，但触控尺寸过小。

建议：

- 默认只显示“前排威胁摘要”：总伤害、最低血目标、可击杀目标数量。
- 点开后显示前排 5 个目标，横向滚动或 5 等分均可，但每个目标触控高度至少 44px。
- 后排队列不作为移动端常驻 HUD，只在补位、奖励、暂停详情中展示。

### 3.5 奖励 / 局外层滚动策略

奖励层：

- 竖屏单列三选一；
- `max-height: calc(100dvh - safe-area - 32px)`；
- 内部 `overflow-y: auto`；
- 选择按钮整卡可点，保留 44px 以上主触控区。

局外层入口：

- 战斗中只保留状态，不放常驻大入口；
- 真正入口放在暂停 / 结算 / 奖励后，而不是首屏战斗 HUD；
- 移动端隐藏“局外档案：未开放”这类占位，避免玩家以为它是可操作目标。

## 4. 三档方案

### 4.1 最小修

目标：不改玩法，只修移动端最明显的误触和遮挡。

建议改动：

1. 小屏隐藏或极度压缩 `run-layer-panel`，避免和手牌区重叠；只保留一行节点状态。
2. 小屏把 `combat-director` 从 2x2 四格压成一条“下一步行动”提示，例如 `先打 MP0：Debt Hook`。
3. 小屏结束回合按钮移到手牌附近，或复制一个底部 44px 次级按钮；按钮可见文案保留 `结束 -X HP`。
4. `enemy-peek-toggle` 与 `enemy-slot.targetable` 移动端高度提升到 44px。
5. 卡牌小屏恢复一个短效果字段，不再完全隐藏 `.card-effect`；至少显示 `9伤害 / 抽1 / 前排5`。
6. `非起手 x1` 改成 `可打，断链 x1` 或 `非推荐 x1`，避免误解为禁用。

预期收益：当天即可降低移动端误触、重叠和读牌成本。

### 4.2 半天修

目标：建立真正的移动 HUD 信息架构。

建议改动：

1. 增加横屏专用布局：顶部一行资源，底部单行手牌，右侧固定结束回合 / 目标按钮。
2. 建立移动端“动作栏”：推荐下一张、结束回合后果、目标锁定在一个区域内完成。
3. 手牌采用固定槽位与短暂出牌占位，避免连续 tap 时卡牌重排。
4. 敌人目标选择改为底部或中下部抽屉，只展示前排 5 个可点目标。
5. 奖励面板改成可滚动 sheet，隐藏战斗层的非必要 HUD。
6. 建立 390x844、430x932、844x390 三档静态布局验收，不只看桌面。

预期收益：移动端可以实际跑完 3-5 个短回合，而不是只做视觉 smoke。

### 4.3 重构修

目标：把 HUD 从“桌面面板响应式压缩”改为“移动优先的战斗输入系统”。

建议改动：

1. 将 HUD 拆成固定交互区：资源区、行动建议区、战场目标区、手牌输入区、奖励/结算层。
2. 每个区定义移动端预算：顶部不超过 72px，底部手牌不超过 35% 视口，中心战场至少保留 45-50% 高度。
3. 将卡牌定义补充 mobile short label / mobile effect label，避免 CSS 省略号承担内容取舍。
4. 输入模型区分 `tap to play`、`tap target`、`hold / details`，并保证 pointerdown/click 去重有测试覆盖。
5. 把奖励、局外、debug 都改成独立 overlay 状态，打开时暂停或降权战斗 HUD。
6. 增加 DOM layout regression：检查关键按钮最小尺寸、HUD 覆盖比例、手牌与 run-layer 不重叠。

预期收益：后续加新卡、新奖励、新局外层时，不会每次都把移动端 HUD 撑爆。

## 5. 本轮建议先做什么

本轮建议先做“最小修”的前 4 项，不建议先做大重构。

优先顺序：

1. 先把移动端唯一主语做出来：`下一步行动 + 结束回合后果`，减少顶部重复面板。
2. 立刻处理触控不达标：结束回合、前排显影、前排目标槽全部提高到 44px。
3. 移动端隐藏或压缩 `run-layer-panel`，避免动态手牌时与手牌区重叠。
4. 手牌恢复短效果字段，并把“可打但断链”写清楚。
5. 下一轮再做横屏专用布局和奖励 sheet。

一句话结论：当前 HUD 的移动端不是“不能点”，而是“能点但节奏会被多面板、远距离结束回合、小目标按钮和动态手牌增长打断”。先把主操作线和触控尺寸修好，再谈奖励/局外层入口。
