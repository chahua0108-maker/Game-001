## 1. 本轮必须收敛的目标

第 1 轮必须收敛成一个可复核的 Hyper-Turn P0 样片，而不是继续扩展系统。

本轮唯一主目标：

```text
玩家读懂敌意图
-> 按 0 -> 1 -> 2 接上费用链
-> 获得本回合终局授权 +3
-> 用授权支付 3 费 armed payoff
-> 清掉可见压力 / 避免意图伤害
-> 奖励回应构筑，但不伪装成局外成长
```

制作人裁决：

1. 第 1 轮结束标准不是“再多做一套 run/meta”，而是把当前 P0 链路封成可验收证据。
2. 主线程应把本轮结论写成一份执行清单：合同、已实现代码、自动测试、仍需浏览器人工验收、明确停做项。
3. 如果要继续动代码，只允许修影响上述路径可读性或验收可信度的阻塞问题。
4. 当前已经通过自动测试和构建，本轮下一步更像 QA / 证据整理，不像新功能开发。

## 2. 哪些事情先不要做

本轮先不要做：

- 不做永久 `Max MP +1`、属性面板、账号成长、局外存档或永久解锁。
- 不做完整地图、商店、事件池、boss、随机路线或长 run。
- 不新增大量卡牌、敌人、状态、诅咒、遗物或职业系统。
- 不恢复 realtime heartbeat、自动攻击、no-input damage、60 秒 burst 旧方向。
- 不用测试夹具暗中把 MP 改到 6 来证明 3 费 payoff 合法。
- 不把 `tempAuthorizationMP`、`energyGain`、奖励选卡写成“永久成长”。
- 不做大重构，例如重切 runtime / ECA / renderer 架构；除非当前验收已被证明确实无法通过。
- 不把本轮变成“继续派 10 个专家讨论下一层系统”。第 1 轮已经有足够材料，主线程需要裁决和封口。

## 3. 10 轮节奏如何安排才不跑偏

10 轮要按“每轮只解决一个可验收瓶颈”的节奏推进。每轮开始前先读上一轮 synthesis，结束时必须写一个可执行收敛文档；没有通过上一轮验收，不进入下一轮扩系统。

建议节奏：

| 轮次 | 收敛主题 | 进入下一轮条件 |
| --- | --- | --- |
| 01 | P0 Hyper-Turn 样片封板：0 -> 1 -> 2 -> 授权 -> armed payoff -> 奖励边界 | 自动测试、构建通过；桌面和 390x844 至少人工验一次核心路径 |
| 02 | 卡牌机制契约：cardType / chainRole / payoff / repair / rewardBranch 命名固定 | 现有 16 张卡不再靠描述猜规则 |
| 03 | 发牌与坏手修补：小手牌、抽牌、Wild、当前 MP 返还形成压力 | 玩家能看到一次断链和一次修补成功 |
| 04 | 敌人意图真相源：攻击者、可打前排、补位敌人不混淆 | payoff 避免伤害的数字可被 HUD 和测试共同解释 |
| 05 | 单局奖励与 run 内承接：奖励进入本次 run 后续战斗 | 只做 2-3 遭遇固定脚本，不做地图系统 |
| 06 | 局外成长边界：只定义术语和不可误读文案 | 证明 restart 不保留 run 奖励，不进入存档实现 |
| 07 | HUD / 移动端信息架构 | 390x844 真 tap 不遮挡、不误点、能完成核心链 |
| 08 | 数值与平衡表 | 用参数表调 MP、伤害、意图、奖励阈值，不顺手扩规则 |
| 09 | 自动化与浏览器验收 | 固定 replay / sim / desktop / mobile 形成稳定证据包 |
| 10 | Demo 交付收敛 | 删除噪音、整理说明、只保留玩家能理解的玩法面 |

防跑偏规则：

- 每轮最多一个主目标、一个代码批次、一个 QA 摘要。
- 每轮必须有明确“本轮不做”。
- worker 可以并行，但主线程每轮只接收能支持本轮主目标的产物。
- 文档、代码、测试三者冲突时，先改合同和测试口径，再改实现；不要让 worker 各自解释同一规则。

## 4. 本轮代码改动建议上限

当前自动测试和构建已通过，因此第 1 轮额外代码改动上限应设得很低。

建议上限：

- 默认：不再加功能代码。
- 若必须修：最多 1 个窄补丁，最多 3 个文件。
- 优先允许：HUD 文案 / 移动端遮挡 / 测试命名 / QA 可见证据。
- 谨慎允许：只修会导致 P0 合同失败的 runtime bug。
- 禁止：新增系统字段后继续扩规则、扩卡、扩敌人、扩 run/meta、扩 VFX。

本轮主线程更应该做：

1. 跑并记录 `npm test -- --run` 与 `npm run build`。
2. 做一次桌面和一次 `390x844` 浏览器 smoke。
3. 记录核心路径是否可被用户看懂：MP、终局授权、敌意图、payoff、奖励边界。
4. 如果浏览器验收发现 UI 阻塞，再开一个小修；否则直接进入本轮总结。

## 5. 面向用户验收的简短清单

用户验收只看这些：

- 首屏 3 秒内能看到手牌、当前 MP、敌人意图和可操作按钮。
- 玩家能打出 `0 -> 1 -> 2`，并看到“本回合终局授权 +3”，不是最大 MP 永久增加。
- 3 费 payoff 只能在正确授权下成为 armed payoff；乱序或未授权收益明显弱。
- payoff 后能看到清前排 / 减少结束回合伤害的结果，不需要看 debug 才理解。
- Wild / draw / mana 的作用是修补本回合，不是局外成长。
- 奖励三选一能看出“修补 / payoff / 路线补件”差异。
- Restart 后回到基础 deck、`maxEnergy = 3`、无奖励残留、无局外成长误保留。
- 桌面和 `390x844` 移动端都不出现关键按钮遮挡、文字超框或 tap 路径不可达。
