# Game-001 第 1 轮专家 03：卡组内容架构审查

日期：2026-05-18  
范围：只审查当前 `prototype-web/src/data/cards.ts` 的卡牌内容结构，并提出不侵权的机制等价卡牌包方案。本文不要求修改源码。

## 1. 当前卡池问题

### 1.1 数量与开放面

当前 `cards.ts` 里共有 16 张牌：

| 维度 | 当前情况 | 内容判断 |
| --- | --- | --- |
| 总量 | 16 张 | 足够验证 `0 -> 1 -> 2 -> payoff` 核心链，但不像完整卡牌游戏。 |
| 起手牌 | 4 张：0 费、1 费、1 费、2 费 | 教学链清楚，但第一手没有 3 费 payoff，玩家会得到授权却不一定能消费。 |
| 奖励池 | 11 张 | 已有 reward 概念，但三选一更像按顺序开放，不像有构筑分支的牌池。 |
| 未开放测试牌 | `blood_tithe`、`pulse_draw` 等 | 有过牌骨架，但没有进入主要奖励循环，导致当前体验偏攻击链。 |

### 1.2 类型结构偏窄

按当前 `cardType` 统计：

| 类型 | 数量 | 问题 |
| --- | ---: | --- |
| `attack` | 8 | 攻击密度足，能支撑基础链路。 |
| `draw` | 4 | 过牌有雏形，但多为 reserve / 支援牌，奖励分支感不足。 |
| `repair` | 2 | Wild 修补方向正确，但只有两张，缺少“稳定 vs 贪上限”的选择。 |
| `payoff` | 2 | 都是 3 费全场伤害，缺少单体处决、返费、取消意图等差异。 |
| `skill` | 0 | 没有纯技能位，导致防御、控制、净化、过滤都没有空间。 |
| `resource` | 0 | `wild_mana_stitch` 有返费，但类型仍是 repair，资源路线不清楚。 |
| `status` | 0 | 没有状态牌和负面牌，牌库循环缺少风险压力。 |

### 1.3 费用分布能跑通链路，但缺少替代路线

当前费用分布是：

| 费用 | 数量 | 内容判断 |
| --- | ---: | --- |
| 0 MP | 5 | 起手密度足，但主要是低伤害开链和抽牌。 |
| 1 MP | 5 | 承接段足，但缺少防御、控制、风险资源。 |
| 2 MP | 4 | 前排段和支援段存在，但都是为了进入 3 费 payoff。 |
| 3 MP | 2 | 只有全场清算，缺少 boss / 精英战所需的单体 payoff。 |

现在的牌序答案过于集中：尽量打出 `0 -> 1 -> 2`，然后找全场 payoff。更完整的卡牌游戏至少还需要三类“非唯一正解”：

- 打不死敌人时，用防御或意图处理活下来。
- 手里缺环时，用过牌、过滤、Wild 或资源修补。
- 想打高上限时，接受状态牌、负面牌或透支代价。

### 1.4 效果与目标过于基础

当前效果主要是四类：

- 伤害：单体、前排、全场。
- 抽牌：`drawCards`。
- 当前 MP：`energyGain`。
- 标签：`wild`、`draw`、`reorder`。

问题是：

- `reorder` 目前更像文本标签，还没有真正形成“调牌序”的玩家选择。
- `mark`、`spark`、`reclaim` 等 `comboNode` 多数是路线风味，没有独立战斗效果。
- 没有 block、护盾、减意图、延迟、净化、丢弃、保留、消耗、击杀触发。
- `targets` 只有 `front-enemy`、`front-row`、`all-enemies`、`self`，足够 P0，但不足以表达“目标意图”“目标状态”“牌库顶”等完整卡牌语法。

### 1.5 描述文本过长

当前描述经常把设计目的、资源边界、教学说明都写进一张卡里，例如解释“不是永久 MP 成长”“用于打开授权链”。这对设计文档有用，但放到 UI 卡面会显得像规则注释。

卡面应该只写本张牌的即时规则；链路教育、MP 边界、payoff 限制应放在 HUD、关键词 tooltip 或教学面板。

## 2. 30 张以内原创机制卡包矩阵

建议做成 28 张以内的机制等价卡包：保留当前 16 张作为底盘，新增 12 张补齐防御、资源、状态、风险、奖励和升级分支。这里的名称都是 Redline 自有包装，不使用竞品原名；机制使用卡牌游戏通用语法，但绑定到 Redline 的“清算链 / 临时授权 / 敌意图”核心循环。

### 2.1 设计约束

- 总牌数控制在 28 张，方便 demo 仍然可读。
- 起手牌仍保持 4-6 张，不直接塞完整牌池。
- 奖励池按功能桶生成：攻击 / 修补或过牌 / payoff 或风险奖励。
- P0 优先使用当前 schema 能表达的牌；防御、状态、击杀触发、升级分支作为 P1/P2 规则扩展。
- 机制等价，不复制竞品卡名、图标、职业、数值曲线或专属关键词。

## 3. 每类卡在核心循环里的作用

### 基础牌

基础牌不负责创造复杂 build，而是教玩家读懂 Redline 的语法：0 费开链、1 费承接、2 费扩展、3 费消费授权。基础牌数值可以保守，但目标和费用必须清楚。

### 攻击牌

攻击牌负责把正确牌序翻译成战场变化：敌人 HP 下降、前排意图减少、清场窗口出现。当前攻击牌已经能支撑 demo，但应补一张单体高压解法和一张击杀奖励解法，避免所有终点都是全场爆发。

### 防御牌

防御牌解决“这一回合杀不掉怎么办”。它不应取代攻击，而应成为打不出 payoff 时的备选答案。Redline 更适合用“降低敌意图 / 护栏 / 延迟攻击”表达防御，而不是大堆数值护甲。

### 过牌牌

过牌牌负责让玩家从坏手牌里找下一环。P0 可以先做抽 1、抽 2 后弃 1；P1 再做查看牌库顶、检索 repair、净化状态。过牌越强，状态牌和负面牌越有价值，因为玩家会更快洗回污染牌。

### 资源牌

资源牌只服务本回合，不应叫“成长”。它的职责是延长一手牌：返还当前 MP、产生修补储备、提供 payoff-only 授权。资源牌必须有限制，否则会把 `0 -> 1 -> 2` 的判断变成无脑堆费用。

### 状态牌

状态牌是战斗内的临时污染。它让敌人不只是扣血，也能破坏玩家的下回合手牌质量。P0 可先做“占手牌、可打出消耗”的轻状态，P1 再做回合末惩罚和净化互动。

### 风险牌

风险牌给玩家短期爆发，但把负面牌塞进未来循环。例如 0 费拿 2 当前 MP，但加入一张 Burnout Notice。它能让 demo 出现“这一回合爽，但下回合要还债”的卡牌游戏感。

### 奖励牌

奖励牌把高水平操作反馈出来：击杀有收益、清空意图有收益、armed payoff 有收益。奖励不一定是新系统，也可以是抽 1、返 1、移除 1 张状态牌这类轻反馈。

### 负面牌

负面牌提供长期压力和代价。和状态牌的区别是：状态牌通常来自战斗临时效果，负面牌可以来自风险牌、事件或高收益奖励。P0 不建议做跨 run 诅咒，先做单次战斗或本次 run 内污染即可。

### 升级分支

升级分支让一张牌有两种命运：

- 稳定分支：降费、抽牌、修补、净化、降低手牌事故。
- 上限分支：更高伤害、击杀收益、armed payoff 加成、风险收益更大。

这样比单纯“伤害+3”更像完整卡牌游戏，也能避免直接照搬竞品升级模板。

## 4. P0 / P1 / P2 增补顺序

### P0：先把当前卡池变成可玩的完整闭环

目标：不引入大系统，只补齐当前链路的阅读性和奖励选择。

建议顺序：

1. 把 `blood_tithe`、`pulse_draw` 从 reserve-test 变成玩家可见的过牌选择，验证坏手牌修补。
2. 奖励三选一按功能桶组织：一张攻击或 payoff、一张过牌或修补、一张资源或风险。
3. 增加 1 张单体 3 费 payoff，解决 boss / 精英压力。
4. 增加 1 张击杀奖励攻击牌，让“精确斩杀”有即时收益。
5. 缩短所有卡面描述，把设计解释移到 tooltip 或文档。

P0 可以暂时不做真正 block、状态牌、升级界面；如果要模拟防御，可先用“低伤害 + 减少意图”的攻击/技能牌作为占位。

### P1：补防御、状态、风险与净化

目标：让玩家不是只在伤害牌之间选择，而是在“活下来、贪资源、保持牌库干净”之间选择。

建议顺序：

1. 增加 `guard / preventIntent / reduceIntent` 之类的防御字段或等价事件。
2. 增加 2 张状态牌：一张纯占手牌，一张带回合末惩罚。
3. 增加风险资源牌：短期给 MP 或授权，代价是加入状态或负面牌。
4. 增加净化互动：移除状态、把状态转成抽牌、清空意图后移除状态。
5. 让奖励池至少有三条路线：稳定过牌、修补资源、payoff 上限。

### P2：形成 run 内构筑和升级分支

目标：让 demo 从“单场战斗样片”变成“有牌组成长的短 run”。

建议顺序：

1. 做卡牌升级界面，每张主动牌只给两个分支：稳定 / 上限。
2. 做负面牌的 run 内保留与移除，不急着做永久诅咒。
3. 做路线标签收益：Spark、Mark、Reclaim 不只是 `comboNode`，而是能改变奖励池和升级方向。
4. 做 3 场遭遇的小 run：普通战、精英战、终局战，验证单体 payoff 与全场 payoff 的差异。
5. 做角色或部门身份，但不要让角色牌进入普通战斗手牌循环。

## 5. 如何避免 UI 文本过长

### 5.1 卡面文案分层

卡面只放即时规则，不放设计解释。

| 层级 | 放什么 | 不放什么 |
| --- | --- | --- |
| 卡面 | 费用、目标、效果、关键词 | “用于打开授权链”“不是永久成长”这类设计说明 |
| 关键词 tooltip | 修补、授权、意图、状态、消耗 | 每张牌重复解释同一规则 |
| HUD 教学 | `0 -> 1 -> 2`、临时授权、payoff-only | 单张卡的具体数值 |
| 图鉴 / 详情页 | 完整规则、升级分支、设计风味 | 战斗中必须快速读的内容 |

### 5.2 推荐卡面模板

把当前长句改成短模板：

- 攻击：`造成 9。续链：倍率+1。`
- 前排：`对前排造成 5。完成 0-1-2：授权+3。`
- 过牌：`抽 1。若续链，抽牌受倍率影响。`
- 资源：`获得 1 当前 MP。仅本回合。`
- 修补：`修补：视为当前缺口费用。`
- payoff：`全场 16。armed：结算意图收益。`
- 状态：`无效果。打出后消耗。`
- 风险：`获得 2 当前 MP。加入 1 张过载。`

### 5.3 长度规则

- 中文卡面主效果控制在 18-28 个汉字。
- 最多两行：第一行主效果，第二行条件效果。
- 关键词用 2-4 个汉字：修补、授权、意图、过载、净化、护栏。
- 数字用阿拉伯数字，避免“造成九点伤害”这种占宽写法。
- 升级牌只显示变化后的效果，不显示“原效果 + 额外解释”。
- 英文卡名可以保留，但战斗内效果文字尽量只用中文，避免中英混排挤压空间。

### 5.4 当前卡牌描述的整理方向

当前描述适合放在设计文档，不适合直接放战斗 UI。建议将所有卡拆成：

```text
name: Redline Cut
rulesText: 造成 9。
keywords: 攻击 / 续链
flavorText: 可选，不在默认战斗卡面显示。
designerNote: 只进图鉴或 debug 面板。
```

这样既能保留设计解释，也不会让 demo 的卡牌阅读速度被长文本拖慢。
