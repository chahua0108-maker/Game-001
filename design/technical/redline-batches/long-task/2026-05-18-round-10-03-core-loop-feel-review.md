## 0. 总裁决

当前核心体验已经重新站回“卡牌游戏”的框架：玩家能看到敌意图、手牌费用、接链倍率、授权终结、奖励入下一手、移动端短 token 和自动 QA 证据。它不再只是早期实时 demo 或单纯展示 HUD。

但体验压迫还没有完全稳住。最高风险是：**系统越来越会自动把正确答案摆出来，玩家的主观体验可能从“我在有限手牌里解局”滑成“我按脚本看机制演示”。** 这个风险主要来自固定起手、强提示、奖励入下一手、`paper_shatter` 顶终结和授权 payoff 同时存在。它们单独合理，叠在一起时会削弱失败压力和可复玩性。

第 10 轮不应该再扩新机制。更应该用 3-5 回合体验切片证明：

```text
发牌读压力 -> 选择路线 -> 因费用或意图承压 -> 接链争取授权 -> 终结或失败 -> 奖励修正下一手
```

如果这条链路中的“选择”和“失败后果”不够尖锐，就会变成展示型 demo。

## 1. 证据基线

- 第 1 轮指出核心问题是“脚本化清前排”，缺少牌面级后果预览和奖励闭环。
- 第 3 轮把首奖压到 12 XP，并让奖励先入牌再发下一手。
- 第 5 轮开放 `blood_tithe / pulse_draw`，并把抽牌反馈改为 `抽N仍-X` 方向。
- 第 7 轮压缩 HUD token：`授权+3`、`缺MP1`、`授权付`、`整备`、`回合损N`。
- 第 8 轮只落地 `paper_shatter` 极窄顶终结，不做通用找牌。
- 第 9 轮 `qa:ui` 三档视口通过，`整备：顶终结`、End Turn 可用、无文字超框。

这些证明“可读、可测、可回归”已经成立；本审查只判断“是否还像玩家在打牌”。

## 3. 是否仍然是卡牌游戏

是，但只在“短切片核心循环”层面成立。

它像卡牌游戏的部分：

- 玩家每回合围绕有限手牌、费用、敌意图做选择。
- 非伤害牌不会替玩家消除压力，只提供找解机会。
- `0 -> 1 -> 2 -> 授权终结` 有明确资源节奏和爆点。
- 奖励进入下一手，开始形成单局构筑反馈。
- 移动端和桌面都能读到关键后果。

它仍像展示型 demo 的部分：

- 第一手和关键 QA 场景过于确定性。
- 成功链路证据多，失败链路证据少。
- 顶终结、授权支付、奖励入下一手叠加后，系统有“发答案”的倾向。
- 可复玩性还主要来自奖励池排列，不来自敌人、牌组污染、删改牌或路线事件。

## 4. P0 体验验收清单

第 10 轮最终验收不应只看 `npm run check` 和 `qa:ui`。还需要人工或脚本化记录以下 5 个镜头：

1. 开局 3 秒内能读出：当前 MP、结束伤害、下一张费用、最高威胁。
2. 打错一张牌后仍可继续，但 End Turn 伤害明显更高。
3. 打出 `pulse_draw` 或 `blood_tithe` 后，意图不下降，玩家知道它只是找解。
4. 完成 `0 -> 1 -> 2` 后，授权出现；若手里没有 payoff，爆点不会凭空发生。
5. 选择奖励后，下一手看到该牌，并且该牌回应上一手的真实问题。

缺任一项，都不应说“核心体验已经稳定像卡牌游戏”。

## 5. 最高风险

最高风险：**成功样片被自动 QA 钉住了，但失败样片和选择分叉没有同等可见证据。**

这会导致团队误判为“核心循环已经有压迫”，实际玩家看到的可能是：

```text
按提示出牌 -> 系统给授权 -> 系统顶终结 -> 选奖励 -> 下一手继续看演示
```

第 10 轮收敛时要优先补失败压力镜头，而不是继续增加新牌、新敌人或新 UI。
