## 2. build plan 的扣分规则

第17轮的核心问题不是“界面有没有显示 build plan 四个字”，而是 build plan 是否真的改变玩家决策和后续验证。

| 情况 | 评分处理 | 建议分数上限 |
| --- | ---: | ---: |
| 只显示解释文案，不改变奖励、路线、升级选择，也无法在下一战复测 | 从第17轮目标 `86` 中扣 `4` 分；最多只算信息表达改善 | `82 / 100` |
| 文案能解释当前候选，但选择仍是静态候选池，不能影响后续路线、奖励或升级 | 从第17轮目标 `86` 中扣 `2-3` 分 | `83-84 / 100` |
| build plan 能影响奖励候选、路线 modifier 或升级候选中的一项，并在下一战可见 | 可达到第17轮建议目标 | `86 / 100` |
| build plan 同时影响奖励、路线、升级三者，并能用 3-5 节点连续 run 复测选择后果 | 可超过本轮基准，但仍不能停止 | `87-88 / 100` |

因此，最关键的裁决是：

```text
如果 build plan 只是显示文案：建议 82 / 100，扣 4 分。
如果 build plan 能影响奖励/路线/升级并可复测：建议 86 / 100，最高 88 / 100。
```

## 3. 为什么完成假设下给 86

第16轮的 `80 / 100` 已经证明主流程不断线：

```text
战斗结束 -> 奖励选择 -> 路线选择 -> 下一战继承
```

第17轮如果完成 build plan，真正新增的是“玩家为什么这样选”：

```text
我上一战缺 bridge / payoff / 清污染 / 资源
-> 奖励给出回应
-> 路线强化或惩罚这个计划
-> 升级把关键牌推到下一战可用
-> 下一战验证这个计划是否成立
```

这会显著提高奖励构筑回应、牌成长理解和短 run 连续性，所以可以从 `80` 提到 `86`。但它还不能直接补齐失败压力、内容密度和长期复玩组合，因此不能接近 `95`。

即使按较强实现给到 `88 / 100`，第17轮也不能停止。

| 剩余缺口 | 影响分 | 为什么仍扣分 |

| 失败压力和路线风险仍浅 | 2-3 | build plan 让选择更清楚，但如果失败、精英、高压节点没有硬后果，风险管理仍不够。 |
| 内容组合密度不足 | 3-4 | 计划链路可理解不等于有足够多可复玩的牌组、路线和事件组合。 |
