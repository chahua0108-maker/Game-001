```text
第15轮目标分：72 / 100
本轮预计增量：+8
是否达到停止阈值：否
```

## 1. 现状判断

从 `2026-05-19-100-point-similarity-control.zh.md` 和 `round-14-synthesis.zh.md` 看，当前官方分数为 `64 / 100`。第14轮已经让以下内容成立：

- `0 -> 1 -> 2`、Wild MP3、payoff 续燃等战斗爆点仍成立。
- 消耗、保留、洗回、状态/污染已经进入 sim 与 HUD 反馈。
- `cardUpgrades` 有局内强化雏形，但还没有成为自然奖励选择。
- `run.currentNode/maxNodes/status/rewardHistory` 已经存在，当前默认 `1 / 3`。
- 奖励选择后会 `AddCardToDeck`，并自动进入下一手/下一战。

但玩家体验仍然像“战斗样片 + 奖励样片”，不像一段 card roguelike run：

- 没有明确节点类型，玩家不知道下一步是在打普通战、精英战、补给还是终点。
- 没有路线选择，`currentNode` 更像内部计数，不像冒险推进。
- 奖励对上一战问题的回应很弱，仍偏固定三分支。
- 下一战虽然继承新牌，但玩家不容易读到“上一战选择改变了这一战”。
- 胜利/失败边界存在于状态机，但不是玩家可见的 run 目标。

所以第15轮的加分应该主要来自“短 run / 地牢推进”维度，而不是继续给 QA、cleanup 或测试覆盖加分。

## 3. 第15轮必须交付的玩家闭环

### 3.1 最小短 run 结构

第15轮应把当前 `maxNodes=3` 明确包装成一个短 run：

```text
节点1：普通战斗
节点2：分支选择后的战斗或压力节点
节点3：终点战斗 / 结算节点
```

最低要求：

- HUD 明确显示 `节点 1/3`、`节点 2/3`、`节点 3/3`，不是只在 debug 或测试里可见。
- 每次奖励选择后，玩家能看到“进入下一节点”或等价短反馈。
- 到达最终节点并选择/结算后进入 `Settlement`，显示胜利状态。
- 玩家死亡进入 `Settlement`，显示失败状态。

这里不要求完整地图、不要求商店、不要求事件房、不要求遗物。第15轮只需要把 3 个节点做成可读的短 run。

### 3.2 路线/节点选择的最小形态

第15轮可以采用“二选一下一节点”的最小路线，不需要完整地图 UI。

建议节点选择只提供两类：

| 节点选项 | 玩家含义 | 实现边界 |
| --- | --- | --- |
| 稳定路线 | 下一战压力较低，奖励偏补链/资源 | 只影响下一战敌人强度或奖励 reason，不开新系统。 |
| 高压路线 | 下一战意图更高或污染风险更高，奖励偏 payoff/强化 | 只影响下一战压力和奖励倾向，不做精英/遗物系统。 |

如果工程成本紧，本轮可以先不做真正地图分叉，但必须至少让玩家在奖励后做一次“下一节点选择”，并让该选择影响下一战的一个可见参数，例如：

- 下一战初始意图总伤害。
- 下一战候选奖励标签。
- 下一战是否更容易出现污染/压力。

没有任何玩家选择，只是自动 `currentNode += 1`，本轮“短 run / 地牢推进”最多只能从 `3` 加到 `6`，无法支撑总分 `72`。

### 3.3 奖励进入下一战的可见因果

本轮不要求完整智能奖励，但必须把“我拿了什么，它如何影响下一战”变成玩家看得见：

- 奖励选择后短反馈：`已加入下一战牌组` / `下一战起手可抽到` / `补承接`。
- 下一战 HUD 能看到最近奖励，例如 `最近奖励 Wild Mana Stitch`，但不要只停在历史数字。
- 至少一个验收路径中，新选奖励会进入下一战手牌、抽堆或弃牌循环，并改变一次可见选择。

奖励回应上一战问题的最低标准：

| 上一战问题 | 第15轮奖励回应 |
| --- | --- |
| 断在 0 后缺 1 | 至少一张奖励标记为“补承接”。 |
| 完成授权但缺 3费终结 | 至少一张奖励标记为“补终结”。 |
| 被污染挤手牌 | 至少一张奖励标记为“补抽牌/修补”。 |
| 未清意图承压 | 至少一张奖励标记为“降压/清前排”。 |

本轮只需要覆盖其中 1-2 类，不要试图一次性做完整 reward AI。

## 4. 本轮不做什么

为保证 `64 -> 72` 的增量真实、可控，第15轮明确不做：

- 不开完整遗物、商店、删牌、营火、事件房。
- 不做永久局外成长；`局外档案未开放` 可以继续保留。
- 不大规模扩卡池；最多补支撑短 run 的 1-2 个节点/奖励标签。
- 不把 `cardUpgrades` 做成完整宝石系统；如果接入，也只能作为奖励候选的一个轻量选项，不能抢本轮主线。
- 不重写战斗核心链路，避免破坏第12-14轮已成立的费用链、生命周期和污染合同。
- 不用 QA 通过数、测试数量、cleanup 证据给核心体验加分。

## 5. 验收门槛

### 5.1 核心体验验收

第15轮要达到 `72 / 100`，至少满足：

1. 玩家从节点1完成战斗或奖励后，进入节点2，并能在 HUD 上读到节点变化。
2. 玩家至少经历一次奖励选择，奖励进入后续战斗牌组，而不是只写入历史。
3. 下一战能看到上一奖励的影响，至少表现为手牌/抽堆/可出牌选择变化。
4. 最短 run 有明确终点：3节点胜利进入结算。
5. 失败边界存在且玩家可读：HP 归零进入失败结算。
6. 至少一个节点推进事件和一个奖励选择事件不使用 raw debug enum 直接展示给玩家。

### 5.2 QA 门禁

QA 不计入分数，但第15轮不能在 QA 失败时验收。门禁仍按官方表执行：

- `npm run check` 通过。
- `npm run qa:similarity` 通过。
- `npm run qa:ui` 通过。
- `npm run qa:lifecycle` 通过。
- 浏览器 QA 后 `page/context/browser/server` 全部关闭，端口无残留。
- 无玩家可见文字超框、横向页面溢出、raw debug token 泄漏。

注意：第14轮已暴露默认端口并行抢占风险。第15轮如果并行运行浏览器 QA，必须分配独立端口或串行运行，否则假失败不能作为机制扣分，但会阻止本轮验收。
