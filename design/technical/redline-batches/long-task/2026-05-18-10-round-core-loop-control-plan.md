# 2026-05-18 10 轮核心循环长任务控制计划

## 执行边界

- 主线程职责：流程监督、冲突控制、轮次汇总、最终取舍、验收。
- Worker 职责：每轮 10 个全新专家视角，独立审查或独立实现，不复用上一轮身份。
- 机制复刻边界：复刻卡牌游戏的机制结构、反馈节奏、资源压力、卡组成长和战斗决策；不直接复制第三方受版权保护的卡名、原文描述、美术或素材。
- UI 硬约束：桌面与移动端均不允许文字超框、按钮挤压、浮层遮挡核心操作区。
- Git 边界：从第 16 轮起，每轮主线程验收通过后本地提交一次；提交前排除 `.codex/`、浏览器 QA 输出等本地/生成物。

## 每轮固定流程

1. 派出 10 个新专家，限定互不重叠的视角和输出文件。
2. 专家只写自己的轮次文档或自己负责的代码范围，不改别人的文件。
3. 主线程汇总为本轮执行清单，选择小而可验收的核心循环改动。
4. 实现后运行自动化测试、构建和浏览器 UI 检查。
5. 浏览器验收结束后关闭测试页面，避免残留。
6. 把本轮结论写回 `design/technical/redline-batches/long-task/`。
7. 本轮验收通过后本地 git 提交一次。

## 本轮硬验收

- 第 1 轮必须新增 10 份独立专家文档。
- 每份文档必须给出 3 个以上可执行改动建议，并标注优先级。
- 主线程必须把本轮结论汇总成一个可执行清单。
- 如进入代码修改，必须至少通过 `npm test -- --run` 和 `npm run build`。
- 如打开浏览器验收，结束后必须关闭页面。

## 第 10 轮后的滚动规则

- 10 轮不是终点；只要核心体验评分未达到 `95 / 100`，就继续追加第 11-30 轮。
- 每个追加轮次仍维持 10 个全新专家视角，主线程只做取舍、整合、验收和汇总。
- 每轮只允许选择一个可验收的核心相似度切片，避免把“更像竞品”误执行成无限堆系统。
- 继续遵守版权边界：复刻机制结构和体验节奏，不复制卡名、原文、美术、素材或 UI 构图。
- 浏览器验收后必须关闭 page/context/browser/server，并检查端口无残留。
