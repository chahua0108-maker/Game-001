边界：不修改源码，不提交 git；本文只设计 HUD 表达、短文案和验收。  
目标：基于当前 HUD 的意图预览、费用链、移动端短文案和奖励卡表现，设计“坏手 / 修补 / 奖励”如何在 HUD 上表达，让玩家知道抽牌、修补、奖励牌的价值，并且不超框。

## 0. 一句话结论

P0 不需要再往卡牌按钮里塞长说明。HUD 应把“坏手 / 修补 / 奖励”的价值拆成三层短提示：

```text
坏手提示：你现在缺哪一段，结束回合会吃多少。
修补提示：这张牌能不能把缺口接回链，并找下一张牌。
奖励提示：这张奖励回应的是稳定性、终结力，还是路线密度。
```

玩家要读到的不是“这张牌有 drawCards=1 / utilities=wild”，而是：

```text
缺 MP1 段 -> 修补可接回 -> 抽牌找 MP2 / 终结 -> 奖励补修补或终结
```

移动端只显示结论短句；桌面端可以显示原因和下一步建议。所有提示必须围绕当前 HUD 已有真相源：`chain.nextExpectedCost`、`card-intent-preview`、`enemyIntentSummary.totalDamage`、`rulesText/mobileEffect/keywords/detail`、`reward.choices`。

## 1. 当前可用 HUD 基线

当前界面已经有以下可复用位置：

| HUD 区域 | 当前职责 | 本文建议新增/调整的表达 |
| --- | --- | --- |
| `combat-director` 本回合链路 | 显示 `MP0 -> MP1`、下一张期望费用、敌人意图、终结预览 | 增加“缺口判断”：`缺 MP2`、`可修补`、`抽牌找终结` |

| 卡牌按钮 `.card-intent-preview` | `意图 12->7`、`抽牌找解`、`返MP找解` | 对修补牌优先显示 `修补找链` 或 `修补+抽`；对奖励新牌显示它下一手解决什么 |

## 2. 表达原则

### 2.1 坏手不是错误，而是一个缺口

HUD 不要写“坏手”“糟糕”“无解”。玩家真正需要知道的是：

```text
我现在缺哪一段费用？
我手里有没有办法接回？
如果不接回，结束回合会吃多少？
```

推荐把“坏手”翻译成三种公开状态：

| 状态 | 触发口径 | 桌面文案 | 移动短文案 |
| --- | --- | --- | --- |
| 缺段 | 下一张期望费用不在可支付手牌中 | `缺 MP2 段` | `缺MP2` |
| 有修补 | 有 `chainRole=repair` 或 `utilities=wild/draw/mana` 的可打牌 | `可修补缺口` | `可修补` |
| 需止损 | 无可接链、无可修补，但敌意图仍高 | `先降低意图，或结束承伤` | `先降意图` |

### 2.2 修补提示必须说清“修哪里”

修补牌的价值不是“它有修补标签”，而是它在当前回合能不能代替 `nextExpectedCost`。因此提示应从静态文案变成当前态文案：

```text
期望 MP2，Wild Mana Stitch 可按 MP2 计入链
=> 卡面显示：修补 MP2
```

当修补牌同时抽牌或返 MP，移动端只保留最有价值的一个后续动作：

| 当前局面 | 修补牌短文案优先级 |
| --- | --- |
| 缺当前 expected cost | `修补 MPx` 优先 |
| 已能接链但缺 payoff | `抽1找终结` 优先 |
| MP 不足但有返 MP | `返MP接链` 优先 |
| 敌意图极高且修补本身不降意图 | `修补找解`，不要显示虚假的 `意图 N->M` |

### 2.3 奖励提示必须回应刚才的问题

奖励三选一不要只像卡牌商店。它应该让玩家明白：

```text
我刚才的问题是缺修补 / 缺终结 / 缺路线段。
这张奖励能解决哪个问题？
它什么时候进入下一次决策？
```

奖励卡面建议固定三类 reason：

| 奖励分支 | 玩家读法 | 奖励短文案 | 示例卡 |
| --- | --- | --- | --- |
| `repair-resource` | 我断链或缺资源，下次要更稳 | `补缺口` / `修补资源` | `Wild Mana Stitch`、`Wild Gap Key` |
| `payoff` | 我能接链，需要消费授权 | `拿终结` / `消费授权` | `Severance Burst`、`Red Ledger Burst` |
| `route-bridge` | 我缺 0/1/2 段密度 | `补路线` / `补 MP1` | `Spark Tap`、`Verdict Mark`、`Clearance Order` |

## 3. 移动端策略

移动端 390x844 的目标是“不超框、不卡决策”。每张卡只允许两行真正决策文本：

```text
第 1 行：费用 + 卡名 + 主角色
第 2 行：当前链路结果
第 3 行：意图/修补/抽牌结果，三者择一
```

### 3.1 移动端显示优先级

| 优先级 | 元素 | 示例 | 是否可隐藏 |
| ---: | --- | --- | --- |
| 1 | 支付状态 | `缺MP1`、`授权付` | 不可隐藏 |
| 2 | 链路结果 | `接x2`、`修补MP2`、`断链` | 不可隐藏 |
| 3 | 意图变化 / 找解 | `意图17>12`、`抽1找解`、`修补找链` | 不可隐藏 |
| 4 | 主关键词 | `修补`、`终结` | 空间不足可隐藏 |
| 5 | `card-effect` 全文 | `MP0 · 修补 抽1` | 移动端可隐藏 |
| 6 | payoff 长说明 | `终局授权：终结 x3 · 全场48` | 移动端改短 |

移动端禁止同时显示三条长句：

```text
起链 x1 + 意图仍 17 + MP 0 · 修补 抽1
```

应压成：

```text
修补MP2
抽1找解
```

### 3.2 移动端短文案预算

| 场景 | 短文案 | 字符预算 | 说明 |
| --- | --- | ---: | --- |
| 缺段总览 | `缺MP2` | 5 | 放 director 或卡面状态 |
| 可修补 | `可修补` | 3 | 放 director |
| 修补牌命中缺口 | `修补MP2` | 6 | 替代 `接链 x3` 的增强版 |
| 修补后抽牌 | `抽1找解` | 5 | 替代 `抽牌找解` |
| 修补后返 MP | `返MP接链` | 6 | 说明返的是当前回合 |
| 普通抽牌 | `抽1找MP2` | 7 | 已知缺段时显示目标 |
| 找 payoff | `抽1找终结` | 6 | 已有授权但无 payoff |
| 断链风险 | `会断链` | 3 | 比 `断链 x1` 更直白 |
| payoff 未武装 | `未武装` | 3 | 不写长解释 |
| payoff 可打 | `授权终结` | 4 | 比 `终局授权支付` 短 |
| 奖励修补 | `补缺口` | 3 | 奖励卡 reason |
| 奖励终结 | `拿终结` | 3 | 奖励卡 reason |
| 奖励路线 | `补路线` | 3 | 奖励卡 reason |

### 3.3 移动端不同状态的具体呈现

#### A. 正常顺链

```text
卡面链路：接x2
卡面结果：BRU意图17>12
Director：下一张 MP2
```

#### B. 坏手缺段

```text
Director 主提示：缺MP2
Director 次提示：可修补 / 抽牌找MP2 / 先降意图
结束按钮：结束 -17
```

#### C. 修补牌可用

```text
Wild Mana Stitch
修补MP2
抽1找解
```

如果该牌会让链回到有效 `0 -> 1 -> 2`，卡面不要只写 `抽牌找解`；`修补MP2` 优先级更高。

#### D. 有授权但没有终结

```text
Director：授权可用
卡牌 self/draw：抽1找终结
结束按钮：结束 -N
```

不要写 `终局授权可用 · 等待3费终结牌` 这种长句到移动卡面；它属于 director 或桌面说明。

#### E. 奖励选择

移动奖励卡每张只保留：

```text
补缺口
Wild Mana Stitch
修补。抽1，当前MP+1
```

不要在 reward card 里同时显示 `roleLabel · targetLabel`、`MPx · effect`、`rulesText`、`detail` 四层。移动端 reward panel 只需要 `reason + name + rulesText`。

## 4. 桌面端策略

桌面端可以多一层解释，但仍不能把 HUD 变成调试后台。

### 4.1 Combat Director

桌面端 `combat-director` 建议承担“决策句”，每个 cell 不超过一行主句 + 一行解释：

| Cell | 主句 | 解释 |
| --- | --- | --- |
| 本回合链路 | `MP0 -> MP1` | `下一段 MP2` |
| 下一张期望费用 | `缺 MP2` / `可接 MP2` | `Wild 可修补` / `抽牌找 MP2` |
| 敌人意图 | `结束 -17 HP` | `击杀 BRU 可降 5` |
| 终结预览 | `授权可用` / `终结未武装` | `抽牌找 3 费终结` |

桌面端可以显示“为什么”：

```text
缺 MP2 · Wild 可按 MP2 计入链
授权可用 · 只付 3 费终结牌
```

但这类解释不要进入移动卡片。

### 4.2 卡牌按钮

桌面卡牌可以保留当前多层信息，但应按语义排序：

1. 角色与目标：`修补/抽牌 · 自身`
2. 链路结果：`修补 MP2 x3` / `接链 x2` / `会断链`
3. 战斗结果：`BRU 意图 17->12` / `抽1找MP2` / `返MP接链`
4. 支付状态：`MP0` / `授权付` / `缺MP1`
5. 基础效果：`修补 抽1`

当前 `.card-intent-preview` 可以继续作为第 3 行，但它不应只服务敌意图。对 self / repair / draw 牌，它就是“找解预览”：

```text
攻击牌：BRU 意图 17->12
抽牌牌：抽1找MP2
修补牌：修补MP2 · 抽1
返MP牌：返MP接链
```

### 4.3 Reward Panel

桌面奖励卡可以显示四层：

```text
补缺口 · 自身
Wild Mana Stitch
MP0 · 修补 抽1
修补缺口。抽1，当前MP+1。
```

但必须限制高度，不让三张奖励卡因为 `detail` 撑开。`detail` 只进 title / 详情 / debug，不在 reward card 常驻可见区。

## 5. 场景文案表

### 5.1 坏手提示

| 场景 | 桌面文案 | 移动文案 | 显示位置 |
| --- | --- | --- | --- |
| 当前没有 0 费起手 | `缺 MP0 起链` | `缺MP0` | director |
| 已打 0，手里没有 1 | `缺 MP1 承接` | `缺MP1` | director |
| 已打 0->1，手里没有 2 | `缺 MP2 展开` | `缺MP2` | director |
| 没有修补但有攻击可降意图 | `先降低意图` | `先降意图` | director / card preview |

| payoff 在手但未授权 | `终结未武装` | `未武装` | payoff preview / card |

### 5.2 抽牌提示

| 场景 | 桌面文案 | 移动文案 | 说明 |
| --- | --- | --- | --- |
| 缺 expected cost | `抽 1，找 MP2` | `抽1找MP2` | 比 `抽牌找解` 更具体 |
| 授权已开但无 payoff | `抽 1，找终结牌` | `抽1找终结` | 把抽牌价值连接到授权 |
| 敌意图高但无直接伤害 | `抽牌找解，不降当前意图` | `抽1找解` | 不承诺降意图 |
| 抽牌倍率提高 | `接链抽 2` | `抽2` | 只在实际倍率生效时显示 |
| 牌库空将洗弃牌 | `洗弃后抽 1` | `洗弃抽1` | 可放桌面或详情，移动只在发生时提示 |

### 5.3 修补提示

| 场景 | 桌面文案 | 移动文案 | 说明 |
| --- | --- | --- | --- |
| Wild 代替 expected cost | `修补 MP2，链不断` | `修补MP2` | 核心提示 |
| 修补后可触发授权 | `修补后开授权` | `修补开授权` | 只在链长将达 3 时显示 |
| 修补牌同时抽牌 | `修补 MP2，抽 1` | `修补 抽1` | 移动可拆两行 |
| 修补牌返当前 MP | `返当前 MP，继续接链` | `返MP接链` | 不写最大 MP |
| 修补牌无法救已断链 | `链已断，仅按牌面` | `链已断` | 防止玩家误解 |
| 修补牌当前不是缺口 | `可保留作后续修补` | `后续修补` | 若没有 retain，实现前谨慎使用 |

### 5.4 奖励提示

| 分支 | 桌面文案 | 移动文案 | 玩家价值 |
| --- | --- | --- | --- |
| 修补资源 | `补缺口，下回合更稳` | `补缺口` | 解决坏手 |
| 终结牌 | `拿终结，消费授权` | `拿终结` | 把 0->1->2 转成清场 |
| 路线桥接 | `补路线段，提高接链率` | `补路线` | 降低缺段概率 |
| 0 费起手 | `补 MP0 起链` | `补MP0` | 起手更稳 |
| 1 费承接 | `补 MP1 承接` | `补MP1` | 中段更稳 |
| 2 费展开 | `补 MP2 展开` | `补MP2` | 更容易开授权 |

## 6. 三个完整 HUD 例子

### 6.1 例子一：缺 MP2，但有 Wild 修补

桌面：

```text
Director 下一张：缺 MP2
Director 解释：Wild 可修补
Wild Mana Stitch 卡面：
  修补/抽牌 · 自身
  修补 MP2 x3
```

移动：

```text
缺MP2
Wild Mana Stitch

抽1找终结
```

玩家理解：这张牌不是普通抽牌，它能把断口接回 `0 -> 1 -> 2`，并继续找 payoff。

### 6.2 例子二：授权已开，但没有 payoff

桌面：

```text
Director 终结预览：授权可用
Director 解释：抽牌找 3 费终结
Pulse Draw / Paper Route：
  抽1找终结
  不降当前意图
```

移动：

```text
授权可用
抽1找终结
结束-12
```

玩家理解：现在抽牌的价值不是“多一张牌”，而是寻找能消费授权的终结牌。

### 6.3 例子三：升级奖励三选一

桌面：

```text
补缺口 · 自身
Wild Mana Stitch

修补缺口。抽1，当前MP+1。

拿终结 · 全场
Severance Burst
```

```text
补缺口
Wild Mana Stitch
修补缺口。抽1，当前MP+1。

拿终结
Severance Burst
全场16。可用授权支付。

补路线
```

## 7. 不超框约束

### 7.1 移动端硬预算

| 区域 | 高度/行数预算 | 允许内容 |
| --- | --- | --- |
| 单张手牌 | 98-108px | 卡名、角色、链路、预览、支付状态 |
| 单张手牌决策文本 | 最多 2 行 | `修补MP2`、`抽1找终结` |
| Director cell | 2 行 | 主句 + 解释 |
| Reward card | 3 行正文 | reason、卡名、rulesText |
| 结束回合按钮 | 1 行 | `结束 -17` |

超出时隐藏顺序：

```text
detail -> card-effect 长句 -> 第 2 个关键词 -> payoff 长说明 -> roleLabel 副词
```

不可隐藏：

```text
支付状态、链路结果、意图/找解结果、奖励 reason
```

### 7.2 桌面端硬预算

| 区域 | 约束 |
| --- | --- |
| 卡牌按钮 | 仍保持固定高度，不因文本撑开 |
| `card-intent-preview` | 单行，优先短格式 `17>12` |
| `chain-preview` | 单行，优先 `修补MP2` 而不是长解释 |
| reward card | `rulesText` 最多 2 行，`detail` 不常驻 |
| combat director | 每 cell 主句不超过 10 个中文等宽 |

### 7.3 禁止文案

以下写法容易超框或误导，P0 禁用：

| 禁止写法 | 改成 |
| --- | --- |
| `Wild/修补。0 MP self 抽 1 张并返还 1 MP，只修补本回合出牌节奏；不是 Max MP +1。` | `修补。抽1，当前MP+1。` |
| `终局授权：终结 x3 · 全场 48` | `授权终结` / `全场48` |
| `需要 MP 或本回合终局授权：还差 1` | `缺MP1` / `需授权` |
| `升级奖励会进入后续抽牌循环` | `选牌入本局` |
| `抽牌找解` 在已知缺 MP2 时 | `抽1找MP2` |
| `返MP找解` 在修补缺口时 | `返MP接链` |

## 8. 验收

### 8.1 玩家可读性验收

1. 移动端 390x844 下，玩家在 2 秒内能读出当前是 `可接链`、`缺段`、`可修补` 还是 `需止损`。
2. 当手里有 Wild 修补牌且当前缺 expected cost，卡牌可见文本必须出现 `修补MPx` 或等价短句，不能只显示 `抽牌找解`。
3. 当有授权但没有 payoff，抽牌牌必须显示 `抽1找终结` 或等价短句。
4. 当奖励面板出现时，每张奖励必须有一个 reason：`补缺口`、`拿终结`、`补路线` 或具体 `补MPx`。
5. 玩家不打开 debug、不读 title tooltip，也能知道奖励牌的价值方向。

### 8.2 不超框验收

1. 390x844 与 360x640 下，`.card-button`、`.reward-card`、`.combat-director` 不出现横向滚动或文本溢出到邻近 UI。
2. 移动端卡牌高度保持稳定，出牌、抽牌、奖励进入手牌时 rail 不纵向跳动。
3. 所有移动短文案按本文预算：手牌决策短句不超过 7 个中文等宽，reward reason 不超过 4 个中文等宽。
4. `rulesText` 在 reward card 可最多 2 行；`detail` 不进入常驻卡面。
5. 不依赖 hover/title 解释核心规则；移动端核心提示必须是可见文本。

### 8.3 机制一致性验收

1. `card-intent-preview` 的攻击牌预览仍使用同一份敌意图真相源，不因新增修补文案而伪造降意图。
2. self / draw / repair 牌不显示虚假的 `意图 N->M`；它们显示 `抽1找MPx`、`修补MPx`、`返MP接链` 等找解价值。
3. 修补文案只在当前链未断且 wild 可按 expected cost 计入时显示 `修补MPx`；链已断时必须显示 `链已断` 或回到普通效果。
4. 奖励 reason 与 `rewardBranchesForCard` 或等价分支保持一致：修补资源、payoff、路线桥接不能互相乱标。
5. 如果奖励牌承诺“下一手可用”，实际入牌顺序必须让玩家在下一次决策中看见；否则文案只能写 `入本局`，不能写 `下手可用`。
