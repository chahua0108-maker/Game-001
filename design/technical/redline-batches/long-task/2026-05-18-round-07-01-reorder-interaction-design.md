# 2026-05-18 Round 07-01：真实整备 / Reorder 最小交互设计

身份：第 7 轮专家 01，真实整备 / 重排交互设计师  
工作目录：`/Users/roc/Game-001`  
范围：只审查当前 `paper_shatter / lantern_captain` 的 `reorder` 标签、HUD 信息架构和最小玩家交互方案。  
边界：本文只新增 Markdown 文档，不改源码，不提交 git。

## 0. 结论

当前 `reorder` 还不是玩家可操作的真实重排：`paper_shatter` 与 `lantern_captain` 只在卡牌 metadata、短文案和 HUD 标签里表达为 `整备/找牌`，运行时仍然只是按倍率抽牌。第 7 轮如果要把它推进到“真实整备”，不建议做完整牌库浏览器，也不建议只做自动顶牌无交互；推荐最小切片是：

```text
出牌 -> 进入短暂整备 pending -> 看 K 取 N -> 确认 -> 抽入手牌
```

其中 `N = drawCards * effectMultiplier`，`K = min(N + 1, 4, 可见牌数)`。UI 使用一个贴近手牌区的短整备条，不做全屏大弹窗，不拖拽，不显示完整规则长文，只给 `整备 / 看4取3 / 已选3/3 / 确认` 这类短 token。

## 1. 当前 UI / HUD 事实

### 1.1 卡牌数据事实

| 卡牌 | 当前定义 | 当前承诺 | 当前风险 |
| --- | --- | --- | --- |
| `paper_shatter` / `Paper Route` | 2 MP，`targets: self`，`cardType: draw`，`drawCards: 1`，`utilities: ['draw', 'reorder']`。 | `rulesText: 抽1。接链抽3找终结。`，`mobileEffect: 抽1/3`，关键词为 `抽牌 / 整备`。 | 文案已经暗示“找终结”，但运行时尚无玩家选择和真实重排。 |
| `lantern_captain` / `Lantern Captain` | 2 MP，`targets: self`，`cardType: draw`，`drawCards: 1`，`utilities: ['draw', 'reorder']`。 | `rulesText: 抽1。接链抽3找路线。`，`mobileEffect: 抽1/3`，关键词为 `抽牌 / 整备`。 | 和 `Paper Route` 同构；若 UI 不区分目标，会让两张牌读感重复。 |

现有测试还明确锁定：这两张牌当前是 self draw support，不产生 `reorder` 命令 / 事件；HUD 也只显示 `整备/找牌`，不承诺“重排牌库”。

### 1.2 HUD 信息层事实

- 手牌按钮固定高度，内容是单行省略体系：费用、牌名、`card-meta`、链路、`card-intent-preview`、支付状态、桌面用 `card-effect`。
- `hudCardRoleLabel()` 看到 `utilities.has('reorder')` 时返回 `整备/找牌`。
- `cardEffectLabel()` 在抽牌倍率大于 1 时，`reorder` 牌会显示类似 `抽3 整备`。
- `hudCardIntentPreview()` 对 0 伤害抽牌牌显示 `抽N仍-X`，这已经是第 6 轮落地的核心短反馈。
- 移动端 `<=640px` 会隐藏 `.card-effect` 与 `.card-payoff`，所以真实整备的核心反馈不能只放在桌面说明行。
- 桌面 combat feed 目前会写 `出牌 ${card.name} · xN · 抽N`，新增整备结果必须继续短，不要写成长句。
- 奖励面板可以滚动，但奖励卡内部 `small / em` 仍有省略和行数限制；整备说明应该先在手牌 / pending 交互里说清，不依赖奖励详情。

### 1.3 当前 UI 容量判断

可用空间不在中心战场，而在手牌 rail 附近。真实整备必须满足：

- 不新增常驻 HUD chip。
- 不在状态条写完整规则。
- 不把 `看4取3并重排牌库寻找终结牌` 塞进手牌按钮。
- 不让移动端从横向手牌 rail 退回多行 grid。
- pending 阶段可短暂接管手牌区，因为玩家此刻不能继续出牌，必须先完成整备。

## 2. 三个交互方案

### 方案 A：偏好选择 chip

流程：

```text
出牌 -> 显示 2-3 个偏好 chip -> 玩家点 找终结 / 找路线 / 自动 -> 系统顶牌或抽牌
```

UI：

- 桌面：在手牌上方显示一行 `整备：找终结 | 找路线 | 自动`。
- 移动：同一行放进手牌 rail 区域，按钮保持 44px 高。
- combat feed 只写 `整备 找终结 · 抽3`。

优点：

- UI 最轻，几乎不需要卡牌预览。
- 对 `paper_shatter` / `lantern_captain` 的差异可以直接表达为默认偏好不同。
- 移动端风险最低。

问题：

- 玩家没有看到真实牌，体验更像“选择搜索策略”，不是直观的 reorder。
- 如果结果没命中，玩家容易觉得按钮没有作用。
- 仍需要运行时搜索规则，交互价值偏弱。

裁决：适合 P0.5 过渡，不适合作为第 7 轮“真实整备”的推荐终态。

### 方案 B：看 K 取 N 整备条

流程：

```text
出牌 -> 暂停抽牌 -> 看 K 张候选 -> 选择 N 张 -> 确认抽入手牌
```

规则：

- `N = card.drawCards * effectMultiplier`。
- `K = min(N + 1, 4, drawPile + 可回填 discard 的可见数量)`。
- 如果 `K <= N`，没有真实选择空间，跳过 pending，按原顺序抽牌。
- 默认选中原牌顶前 N 张；玩家可以点候选牌切换选择。
- 已选牌按显示顺序抽入手牌；未选牌按原相对顺序放回牌堆顶，不弃置、不置底、不消耗。

UI：

```text
整备 看4取3 · 找终结
[候选1] [候选2] [候选3] [候选4]
已选3/3        [按原序] [确认抽3]
```

`paper_shatter` 默认文案：`整备 看4取3 · 找终结`。  
`lantern_captain` 默认文案：`整备 看4取3 · 找路线`。

优点：

- 玩家看到了真实候选牌，选择有可感知价值。
- `抽1` 时是 `看2取1`，`接链抽3` 时是 `看4取3`，复杂度随收益自然增长。
- 最多 4 张候选，不需要完整牌库浏览器。
- 不需要拖拽；点击选择 + 确认即可。
- 可以复用现有卡牌短样式，只显示费用、短名、角色标签。

问题：

- 需要一个 pending choice 状态。
- 需要定义候选从 discard 回填时的可见顺序。
- 多选比单按钮重一点，但仍可控。

裁决：推荐方案。它是“真实整备”的最小玩家交互切片。

### 方案 C：完整整备面板

流程：

```text
出牌 -> 打开整备面板 -> 预览牌库顶 N 张 / 弃牌候选 -> 拖拽排序 / 弃置 / 置底 -> 确认
```

UI：

- 中央或右侧弹层，展示更完整的卡牌信息。
- 支持拖拽排序、上移 / 下移、置底、确认、取消。

优点：

- 最像完整 deckbuilder 的 scry / reorder。
- 后续可以扩展为弃置、保留、检索、净化等复杂牌区操作。

问题：

- UI 重，移动端难做，容易遮挡战场。
- 需要更多输入锁、焦点管理和拖拽触控处理。
- 会把当前 demo 的爽点从“抽牌找终结”转成牌库管理。
- 会增加大量边界：取消后回滚、拖拽顺序、空牌堆、重复牌、同名牌实例。

裁决：不适合当前最小切片，最多作为 P2 方向。

## 3. 推荐最小方案：看 K 取 N 整备条

### 3.1 信息架构

常驻 HUD 不新增内容。信息分三层：

| 层级 | 显示位置 | 显示内容 | 不显示内容 |
| --- | --- | --- | --- |
| 手牌按钮 | 原 `.card-button` | `整备/找牌`、`抽1/3`、`抽N仍-X`、`抽N 整备`。 | 不写“打开牌库”“重排牌库顶四张”。 |
| 整备 pending 条 | 手牌 rail 上方或临时替换手牌 rail | `整备 看K取N`、候选短卡、`已选N/N`、`确认抽N`。 | 不写完整规则说明，不显示全牌库。 |
| 详情 / tooltip / debug | title、debug trace、combat feed | `Paper Route 整备：看4取3，未选牌回到牌堆顶。` | 不作为移动端唯一信息来源。 |

### 3.2 交互细节

1. 玩家打出带 `reorder` 且有 `drawCards` 的牌。
2. 费用、链路倍率、出牌事件先正常结算。
3. 在真实 `DrawCards` 前进入 `PreparePending`。
4. 系统展示最多 4 张候选。候选卡只显示：
   - `MP` 费用。
   - 短牌名或正式牌名一行省略。
   - `终结 / 路线 / 修补 / 抽牌` 这类 2-4 字标签。
   - 命中偏好时显示短 badge：`终结` 或 `路线`。
5. 默认选中原本会抽到的前 N 张。
6. 玩家点击候选卡切换选择：
   - 未满 N：点未选卡加入选择。
   - 已满 N：点未选卡时替换最早选中的一张，避免玩家先取消再选择。
   - 点已选卡时，如果已选数量大于 1，可取消；如果会低于 1，则保持选中。
7. `确认抽N` 在已选数量等于 N 时可用。
8. `按原序` 直接使用默认前 N 张，减少老玩家操作成本。
9. 确认后抽入手牌，未选候选回到 `drawPile` 顶部，保持原相对顺序。
10. combat feed 写短句：`整备 Paper Route · 看4取3`，下一条仍可写 `发牌 3 张`。

### 3.3 桌面布局

建议 DOM 结构是一个临时 surface，不是大 modal：

```text
[status strip]                         [combat feed]

                 combat playfield

         [整备 看4取3 · 找终结] [已选3/3]
         [mini card] [mini card] [mini card] [mini card] [确认抽3]

                       [原 hand rail 被暂停或置灰]
```

约束：

- 宽度：`min(640px, calc(100vw - 32px))`。
- 高度：桌面不超过 144px。
- 候选卡最多 4 张，不能因为 5 张以上而换行。
- 只在 pending 阶段接收 pointer events。
- pending 期间普通手牌按钮不可点，避免出牌和整备同时发生。

### 3.4 移动布局

移动端不要新增中心弹窗。推荐直接临时替换底部手牌 rail：

```text
整备 看4取3     已选3/3
[候选1] [候选2] [候选3] [候选4]
[按原序] [确认抽3]
```

约束：

- 整备条整体高度不超过 156px。
- 候选卡横向滚动，但首版最多 4 张，360px 下应基本可通过一次滑动触达。
- 操作按钮高度不低于 44px。
- 不依赖 hover / title。
- 不显示 `.card-effect` 级长说明。

### 3.5 文案红线

推荐短 token：

| 场景 | 推荐 | 避免 |
| --- | --- | --- |
| 手牌角色 | `整备/找牌` | `重排牌库寻找终结牌` |
| 整备标题 | `整备 看4取3` | `查看牌库顶部四张并选择三张抽入手牌` |
| Paper 偏好 | `找终结` | `优先寻找3费全场payoff终结牌` |
| Lantern 偏好 | `找路线` | `优先寻找费用链路线修正牌` |
| 选择计数 | `已选2/3` | `你还需要再选择一张牌才能确认` |
| 主按钮 | `确认抽3` | `确认选择并抽取这些牌` |
| 默认按钮 | `按原序` | `取消整备并按照原始牌库顺序抽牌` |
| 日志 | `整备 Paper · 看4取3` | `Paper Route 触发整备，玩家查看四张牌后选择三张抽入手牌` |

## 4. 桌面 / 移动验收

### 4.1 功能验收

- [ ] `paper_shatter` 在非接链时进入 `看2取1`；在 0 -> 1 后作为 2 MP 段打出时进入 `看4取3`。
- [ ] `lantern_captain` 使用同一交互，但标题偏好显示 `找路线`。
- [ ] 如果候选数量不足以产生选择空间，直接按原顺序抽牌，不出现空 pending。
- [ ] 确认后已选牌进入手牌，未选牌仍在牌堆循环中，不进入弃牌堆或消耗区。
- [ ] pending 期间不能继续打手牌、不能结束回合、不能打开奖励选择。
- [ ] `按原序` 与旧抽牌结果一致，方便回归测试。
- [ ] 同名牌情况下只按牌区顺序移动对应的一张定义牌；首版不引入实例 ID。
- [ ] 不改变当前 MP、`maxEnergy`、临时授权、Wild 修补合同。

### 4.2 桌面验收：`1366x768` / `1280x720`

- [ ] 页面无横向滚动：`document.documentElement.scrollWidth <= window.innerWidth + 1`。
- [ ] 整备条不遮挡 `.status-strip`、`.combat-feed`、`.run-layer-panel`、`.deal-panel`。
- [ ] 整备条高度不超过 144px；候选 4 张时不换成两行。
- [ ] `.card-row` 在 pending 时可置灰或被替换，但布局高度不跳变超过 32px。
- [ ] `Paper Route`、`Lantern Captain`、`Severance Burst`、`Wild Mana Stitch` 同时出现在候选或手牌时，关键短词不被省略掉。
- [ ] combat feed 中整备日志单条不超过两行，核心动作在前半句。
- [ ] 键盘可用：`1-4` 切换候选，`Enter` 确认，`Esc` 或 `按原序` 恢复默认。

### 4.3 移动验收：`390x844` / `360x780` / `360x640`

- [ ] 页面无横向滚动：`max(documentElement.scrollWidth, body.scrollWidth) <= viewportWidth + 1`。
- [ ] pending 时底部整备条不超过 156px；不把中心战场变成说明面板。
- [ ] 候选卡、`按原序`、`确认抽N` 都是可触达控件，触控高度不低于 44px。
- [ ] 不依赖 hover；`看4取3`、`已选N/N`、`确认抽N` 在屏幕上直接可见。
- [ ] 手机仍不显示 `.card-effect` / `.card-payoff` 作为核心说明；核心动作必须在 pending 标题和按钮上。
- [ ] 候选横向 rail 可以触达第 4 张；不能因为卡名过长导致确认按钮掉出 viewport。
- [ ] `360x640` 压力尺寸下允许候选 rail 横滑，不允许页面本体纵向滚动救场。
- [ ] 选择、取消、确认后，手牌 rail 回到原位置，不覆盖 End Turn / deal panel。

## 5. 后续落地顺序建议

1. 先把运行时 pending 状态和 `看K取N` 数据模型写窄：只服务 `paper_shatter / lantern_captain`。
2. 再做 HUD pending 条，不碰奖励面板和常驻状态条。
3. 最后更新测试：把“当前不产生 reorder”改成“进入 PreparePending 并按确认结果抽牌”，同时保留 `按原序` 与旧抽牌一致的回归用例。

不建议本轮顺手做完整牌库浏览器、拖拽排序、弃置 / 置底、实例 ID、随机洗牌或通用效果解释器。当前 demo 需要的是玩家能在一次出牌后看到“我确实整理了下一抽”，而不是打开一个新的牌库管理游戏。
