# Redline Experience Boundary Review 04

日期：2026-05-18
角色：体验边界 / PM 专家
范围：只定义玩家体验和制作管理边界；不改 runtime、HUD、卡牌、测试或数值。

## 0. 结论

Redline 下一轮必须把四个层级分开说：

1. **局外成长属性**：跨 run 或跨冒险保留的长期能力。
2. **单次冒险**：从进入走廊到失败 / 击败 Boss / 完成一段路线的本局构筑。
3. **单局战斗**：一次遭遇内的短回合链路、敌人意图、奖励触发。
4. **单次循环发牌**：一个回合 / 一手牌内的出牌、断链、修补、payoff。

当前 P0 的主战场是 **单次循环发牌** 和 **单局战斗**，不是局外成长。`Max MP +1` 只能作为 P1 的单点奖励证据，不能拿来解释 P0 为什么能在 3 MP 底盘下打出 3 费 payoff。P0 应使用“本回合临时授权 / 返费 / 修补储备”解释这一点。

一句 PM 裁决：

```text
P0 验收：玩家用一手牌建立 0 -> 1 -> 2 链，临时获得终局授权，打出 armed payoff，降低敌意图或清前排。
P0 不验收：玩家通过永久成长、局外天赋、等级面板或默认 max MP 膨胀解决战斗。
```

## 1. 四个层级的严格边界

| 层级 | 推荐中文名 | 英文工作名 | 生命周期 | 玩家问题 | 允许影响 | 不允许混入 |
| --- | --- | --- | --- | --- | --- | --- |
| 局外成长属性 | 局外授权 | Meta Progression | 跨 run 保留 | “我下次开局有什么长期变化？” | 初始卡池、初始 max MP、角色授权、永久被动、解锁池 | 本回合返费、临时授权、当前手牌修补 |
| 单次冒险 | 本次清算 | Run / Expedition | 本 run 内保留 | “这次走廊路线正在成为什么 build？” | 本 run 卡组、遗物、临时上限、Boss 规则、奖励路线 | 跨 run 永久属性、单回合 MP 返还 |
| 单局战斗 | 当前遭遇 | Encounter / Battle | 一场战斗内 | “这场战斗我如何活下来并拿奖励？” | 敌人意图、前排压力、战斗奖励、战斗内状态 | 永久成长结算、下一 run 开局属性 |
| 单次循环发牌 | 本回合 / 本手牌 | Turn / Deal Cycle | 一次发牌到结束回合 | “这一手先打哪张，断链后怎么补，什么时候 payoff？” | 当前 MP、临时授权、chain、repair、payoff armed、End Turn 后果 | 等级成长、永久 max MP、run 内大地图推进 |

### 1.1 局外成长属性

定义：玩家失败或完成冒险后仍保留的能力、解锁、角色授权、起始牌组或长期数值。

Redline 例子：

- `初始最大 MP +1`
- `开局加入一张 Redline Cut`
- `清算长授权：每次冒险首个奖励必含 repair`
- `解锁新 payoff 卡进入奖励池`

PM 边界：

- 只能回答“下一次开局更强 / 更多选择”。
- 不应该出现在 P0 固定 5 回合验收里作为过关条件。
- 文案必须带“永久、局外、下次冒险、开局”这类词。
- 如果一个效果只持续本回合，绝对不要叫“成长属性”。

### 1.2 单次冒险

定义：一次 run 内从开局到失败 / 通关的构筑过程。它会在多个战斗之间保留，但 run 结束后不一定保留。

Redline 例子：

- 本次清算的卡组变化。
- 本次清算获得的遗物 / 印章。
- 本次清算 max MP 从 3 变 4，但失败后不保留。
- Boss 修改本 run 的规则。

PM 边界：

- 回答“这次 build 朝什么方向发展”。
- 适合承接奖励三选一：repair / extension / payoff。
- 不要让它承担单手牌解释。例如 Turn 4 为什么突然能打 3 费 payoff，应由“临时授权”解释，而不是说“冒险成长了”。

### 1.3 单局战斗

定义：一次遭遇内的敌人、回合、意图、奖励和战斗状态。它通常持续 3-5 个短回合或更短。

Redline 例子：

- 当前前排 5 个敌人和总意图。
- 本场战斗奖励触发。
- 本场战斗内的 debuff、临时规则、敌方污染。
- 击杀给 XP，触发一次 Reward。

PM 边界：

- 回答“这场是否能活下来、能不能清前排、奖励何时出现”。
- 可以有战斗内成长，但必须写清“本场有效”。
- 不能把单次发牌的当前 MP 变化叫“战斗成长”。

### 1.4 单次循环发牌

定义：从发牌完成到玩家结束回合之间的最小决策循环。Redline Hyper-Turn 的核心爽感就在这里。

Redline 例子：

- `0 -> 1 -> 2` 费用链。
- 当前 MP：`3 -> 0`。
- Wild / draw / mana 修补。
- 临时授权 `+3`，只支付本回合 3 费 payoff。
- payoff `未武装 / 已武装 / 过载清算`。
- End Turn 预览：结束会受多少伤害。

PM 边界：

- 回答“我现在点哪张牌，点完会发生什么”。
- P0 的主要体验指标必须落在这里。
- 这里的资源变化默认回合结束清空，不能写成“成长”。

## 2. 混用会造成的设计弯路

### 2.1 把“临时授权”叫成“最大 MP 成长”

弯路：

- 数值会被迫设计 4/5/6 MP 曲线。
- HUD 会被迫做属性面板，而不是解释当前下一张牌。
- QA 会用“测试把 MP 改到 6 后能清场”当通过证据。
- 玩家会以为答案是堆 MP，而不是打顺 chain。

制作损失：

- P0 从 5 回合 demo 变成长期资源系统立项。
- 敌人血量、意图、奖励池都要跟着重平衡。
- worker 会分裂：有人做永久成长，有人做回合返费，有人做奖励升级，最后没有一条能验收的玩家路径。

### 2.2 把“单次冒险奖励”当成“局外成长”

弯路：

- 奖励文案会承诺永久变强，但实际只在当前 run 有效。
- 玩家失败后发现效果没保留，会觉得系统吞奖励。
- 存档、重开、解锁池会提前进入 scope。

制作损失：

- 需要补存档、局外 UI、解锁记录、回退逻辑。
- 原本只需要证明 reward 回应构筑，最后变成做 meta progression。

### 2.3 把“战斗内奖励”当成“本回合修补”

弯路：

- 玩家当前手牌断链，但系统给一个下一场才生效的奖励，救不了眼前压力。
- HUD 解释不清“为什么我现在还是打不出 payoff”。
- QA 记录会出现“奖励已获得，但本回合体验没有变化”。

制作损失：

- 奖励反馈延迟，5 回合 demo 看不到闭环。
- 修补牌、抽牌、返费这些真正 P0 需要的工具被低估。

### 2.4 把“单次循环发牌”叫成“单局战斗”

弯路：

- “这一手牌断了”会被误判成“整场战斗失败”。
- 一次 End Turn 后果会被设计成全战斗永久惩罚。
- 玩家不敢乱序出牌，因为文案像不可逆失败。

制作损失：

- 断链不是禁牌的合同会被破坏。
- 回合内高速试错被做成慢速保守决策。

### 2.5 把“局外成长”塞进 HUD 主屏

弯路：

- 首屏信息从“先打 MP0”变成“等级、天赋、永久授权、卡池成长”。
- 移动端空间被长期系统占掉，手牌和 End Turn 后果变弱。

制作损失：

- Hyper-Turn 的 3 秒可操作验收失败。
- UI 团队会优化错误的信息层级。

## 3. 命名建议

### 3.1 概念命名

| 不建议 | 原因 | 建议 |
| --- | --- | --- |
| 成长 | 太泛，无法判断持续时间 | 局外授权 / 本次授权 / 本场增益 / 本回合授权 |
| MP 成长 | 容易误读为永久 max MP | 最大 MP +1、当前 MP +1、临时授权 +3 分开写 |
| 升级 | 默认会被理解为长期或等级 | 奖励、授权、修补、终局授权 |
| 回合 | 中文里可能混淆 battle round 与 card turn | 本回合 / 本手牌 / 第 N 轮发牌 |
| 冒险 | 容易和单场战斗混用 | 本次清算 |
| 战斗 | 可保留，但不要指代单手牌 | 当前遭遇 |

### 3.2 推荐术语表

| 玩家可见中文 | 开发字段建议 | 含义 |
| --- | --- | --- |
| 局外授权 | `metaAuthorization` | 跨 run 保留的长期能力 |
| 本次清算 | `run` / `expedition` | 当前 run |
| 当前遭遇 | `encounter` | 当前战斗 |
| 本回合 / 本手牌 | `dealCycle` / `turn` | 一次发牌到结束回合 |
| 当前 MP | `currentMp` | 本回合普通可用资源 |
| 最大 MP | `maxMp` | 本回合刷新上限 |
| 临时授权 | `tempAuthorizationMp` | 本回合额外资源，结束清空 |
| 修补储备 | `repairReserve` | 只用于补 expected cost 的资源 |
| 清算链 | `chain` | `0 -> 1 -> 2` 等费用链 |
| 终局授权 | `payoffAuthorization` | 完成链后给 payoff 的临时支付许可 |
| 已武装 | `payoffArmed` | payoff 获得高收益条件 |
| 过载清算 | `overloadedPayoff` | 使用临时授权支付的强 payoff |

## 4. HUD 文案建议

HUD 的主语必须服务“下一次点击”，不要在玩家层同时展示四个生命周期。

### 4.1 顶部状态

推荐：

- `HP 43`
- `MP 2 / 3`
- `清算链：0 -> 1，下一张 MP2`
- `结束会受 12`

不推荐：

- `成长：MP +3`
- `本局 MP 提升`
- `FSM PlayerTurn`
- `CHAIN MP0?`

### 4.2 临时授权出现时

推荐：

- `终局授权 +3：本回合可支付清算牌`
- `MP 0 / 3 + 授权 3`
- `授权将在结束回合清空`
- payoff 卡面：`已武装：可用授权支付`

不推荐：

- `最大 MP +3`
- `升级成功`
- `MP 成长`
- `本局资源提升`

### 4.3 断链时

推荐：

- `可打，断链 x1`
- `非推荐：不会获得终局授权`
- `打出后仍可结束回合`

不推荐：

- `非法`
- `失败`
- `禁用`
- `链路死亡`

### 4.4 End Turn 区域

推荐：

- `结束回合：承受 12`
- `未使用授权会清空`
- `无可打牌，建议结束`

不推荐：

- 只写 `End Turn`
- 把伤害后果只放进 tooltip / title / aria-label
- 在移动端隐藏结束后果

## 5. 奖励文案建议

奖励必须写出生命周期，否则玩家和 worker 都会误解。

| 奖励类型 | 推荐标题 | 推荐副文案 | 生命周期 |
| --- | --- | --- | --- |
| repair | `证据续链` | `本回合第一次断口可用 Wild 补上` | 本回合 / 本场，按设计写清 |
| draw | `追索线索` | `抽 1 张，优先寻找下一段费用` | 本回合 |
| temp authorization | `终局授权` | `完成 0 -> 1 -> 2 后，本回合获得授权 +3，只支付清算牌` | 本回合 |
| run extension | `信用额度` | `本次清算最大 MP +1，上限 4` | 本次清算 |
| meta growth | `清算长备案` | `下次冒险开局加入 1 张修补牌` | 局外 |
| payoff | `终局印章` | `已武装 payoff 的前排伤害提高` | 本次清算或本场，必须写清 |

禁用文案：

- `永久变强`，除非真的跨 run 保留。
- `升级 MP`，除非明确是最大 MP，且写清生效范围。
- `成长奖励`，除非旁边标明 `局外 / 本次清算 / 本场 / 本回合`。
- `解锁大招`，除非卡池和奖励池真的发生持久解锁。

## 6. 制作管理口径

### 6.1 P0 只允许的范围

P0 允许：

- 当前 MP。
- 最大 MP 固定 3。
- 清算链 `0 -> 1 -> 2`。
- 本回合临时授权。
- Wild / draw / mana 修补。
- payoff armed / unarmed。
- 当前遭遇内一次奖励触发。
- HUD 显示下一步、结束回合后果、授权清空。

P0 不允许：

- 完整局外成长。
- 属性面板。
- 永久 max MP 曲线。
- 多来源最大 MP 堆叠。
- 存档 / 解锁池。
- 用测试夹具直接给 6 MP。
- 把临时授权文案写成 MP 成长。

### 6.2 P1 允许的扩展

P1 可以验证：

- 一次性 `本次清算最大 MP +1`。
- 奖励三选一中同时出现 repair / extension / payoff。
- 奖励在下一手牌或下一次 draw 中快速可见。
- 单个局外预告，但不做完整局外系统。

P1 限制：

- `Max MP +1` 只出现一次。
- demo 上限固定 4。
- 不影响 Turn 1-4 核心验收。
- 不作为打出 Turn 4 payoff 的必要条件。

### 6.3 P2 以后再讨论

P2 才讨论：

- 局外天赋树。
- 永久属性成长。
- 多角色授权。
- 遗物池。
- 商店升级。
- 失败后保留货币。
- 多 run 解锁。

## 7. 验收口径

### 7.1 文档 / 合同验收

必须能在合同中逐条回答：

- 这个效果属于 `局外 / 本次清算 / 当前遭遇 / 本回合` 哪一层？
- 它什么时候生效？
- 它什么时候失效？
- 它是否跨 run 保留？
- 它是否改变最大 MP，还是只改变当前 MP / 临时授权？
- HUD 用什么文案告诉玩家？
- trace 用什么字段告诉 QA？

如果回答不出来，不允许进入实现。

### 7.2 HUD 验收

390x844 和 1440x1000 首屏必须满足：

- 玩家 3 秒内能说出第一张推荐牌。
- 玩家能看到当前 MP 和最大 MP。
- 玩家能看到结束回合会受多少伤害。
- 临时授权出现时，和最大 MP 分开显示。
- 局外成长不占据战斗主 HUD。

失败口径：

- 首屏出现 `成长 +3 MP` 但实际只是本回合临时授权。
- End Turn 后果只在 tooltip。
- `Max MP +1` 和 `当前 MP +1` 用同一文案。

### 7.3 Sim / QA 验收字段

每个固定 seed 至少记录：

```text
scope: meta | run | encounter | dealCycle
round
currentMpBefore
maxMp
tempAuthorizationMpGained
tempAuthorizationMpSpent
cardsPlayed
costSequence
chainLength
chainBreakReason
repairMethod
payoffArmedValue
intentBeforeEndTurn
resolvedDamage
rewardScope
rewardResponseType
```

通过证据必须能说清：

```text
本回合完成 0 -> 1 -> 2
  -> 获得本回合终局授权 +3
  -> 使用授权支付 3 费 payoff
  -> payoff 因已武装降低敌意图或清前排
  -> 奖励回应 repair / extension / payoff 中的一类问题
```

不接受：

- `maxMp = 6` 但没有来源。
- 奖励标题写成长，但 trace 看不出生效范围。
- 只记录击杀数，不记录敌意图减少。
- 只在 debug 中可见，HUD 玩家层不可见。

## 8. Worker 分工建议

### Contract / QA Worker

- 在所有验收描述里增加 `scope` 字段。
- 禁止使用“成长”作为裸词。
- 检查测试是否通过隐藏 max MP 解决 payoff。

### Runtime Worker

- 把 `currentMp`、`maxMp`、`tempAuthorizationMp` 分开。
- 所有资源变化必须带 `source` 和 `expiresAt`。
- payoff 记录 `armedValue`、击杀数、意图减少。

### HUD Worker

- 只把本回合可行动信息放到主 HUD。
- 临时授权和最大 MP 必须视觉分层。
- End Turn 后果必须可见文本化。

### Reward / Progression Worker

- 每个奖励必须声明生命周期。
- P0 奖励只服务 3-5 回合闭环。
- `Max MP +1` 只作为 P1 extension 选项，不进入 P0 通过条件。

## 9. 最终裁决

Redline 当前最重要的不是“有没有成长”，而是“玩家能不能理解这一次点击为什么成立”。因此：

- **局外成长属性**：先冻结，只保留命名和未来预告。
- **单次冒险**：允许奖励形成 build 想象，但不解决 P0 payoff 合法性。
- **单局战斗**：负责敌意图、前排压力、奖励触发。
- **单次循环发牌**：负责 P0 核心体验，所有 MP、chain、repair、payoff 的首要验收都在这里。

下一轮所有设计、HUD、测试和实现描述，只要出现 `成长 / 升级 / 授权 / MP +1 / 奖励`，必须同时标注生命周期。没有生命周期标注的条目，视为未通过 PM 边界评审。
